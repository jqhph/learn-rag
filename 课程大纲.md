## 模块一：RAG 核心引擎（L1-L7）✅ 已完成

> **目标**：理解 RAG 核心概念，从零搭建可运行的 RAG 检索系统，掌握检索评估方法论。
> **默认技术栈**：Python + Chroma + OpenAI（text-embedding-3-small + GPT-4o-mini），这些是学习期的临时选择，模块二 L8 会系统对比替代方案。

| 编号 | 课程名称 | 核心内容 | 依赖 | 学习产出 |
|------|----------|----------|------|----------|
| **L1** | RAG 最小闭环 | RAG 核心概念：Chunk → Embed → Store → Retrieve → Prompt → Answer → Evaluate；理解为什么 LLM 需要外部知识 | 无 | 能画出 RAG 完整流程图，解释每一步的作用 |
| **L2** | Python RAG Demo 实战 | 约 80 行代码搭建 Chroma + OpenAI RAG 系统；环境配置、API 调用、向量存储与检索 | L1 | 拥有可运行的 RAG 代码，能对文档库提问并获得带引用来源的回答 |
| **L3** | 文档解析与分块策略 | PDF/HTML/Markdown 解析（pdfplumber/BeautifulSoup）；语义分块（段落→句子→token 合并）；tiktoken 精确计数；chunk_size 与 overlap 调参 | L2 | 能将真实文档（PDF/网页）自动解析分块入库，替代手工 DOCS 列表 |
| **L4** | 检索评估与重排序 | Recall@K、MRR 指标；golden test set 构建；Cross-Encoder 重排序；两阶段检索（bi-encoder → cross-encoder） | L3 | 能用量化指标评估检索质量，通过重排序提升答案准确率 |
| **L5** | 混合检索与查询转换 | BM25 + 向量 + RRF 融合；Query Rewrite、Multi-Query、HyDE 三种查询变换策略；jieba 中文分词 | L4 | 能处理精确匹配（订单号/SKU）与语义理解混合查询场景 |
| **L6** | 生产级多源检索架构 | 查询路由（Query Routing）；Function Calling / Tool Use；NL2SQL（只读副本+参数化查询）；联邦检索；数据分水岭原则 | L5 | 理解事务型数据不应入向量库的原则，掌握多数据源架构设计 |
| **L7** | 答案质量评估与 RAGAS | Faithfulness、Answer Relevancy、Context Precision、Context Recall 四大指标；LLM-as-a-Judge 实践；四种 judge 偏见及缓解策略 | L6 | 能自动评估 RAG 回答质量，定位幻觉问题，建立质量基线 |

> ⚠️ **模块一完成后的状态**：学习者用 Chroma + OpenAI 完整跑通了 RAG 核心链路。但接下来面临一系列选择——向量库换不换？框架用哪个？模型用哪家？模块二的 L8 就是为解决这些困惑而设。

## 模块二：RAG 质量地基（L8-L12）🆕 基础补全（L9-L12 ✅）

> **目标**：建立 RAG 技术栈全景认知 + 补全模块一遗漏的四块地基（拒绝策略、对话管理、Prompt Engineering、Embedding 选型）。
> **核心理念**：先看清全局有哪些选择，再深入每项决策——拒绝策略是底线、多轮对话是刚需、Prompt 是容易被忽视的质量杠杆、Embedding 是检索质量的根。

| 编号 | 课程名称 | 核心内容 | 依赖 | 学习产出 |
|------|----------|----------|------|----------|
| **L8** | RAG 技术栈全景图 🆕 | **编程语言**：Python（生态最全）/ JavaScript（前端全栈）/ Go（高性能服务）的 RAG 生态对比；**LLM 调用框架**：LangChain（最全但重）vs LlamaIndex（数据驱动）vs 原生 SDK（轻量可控）vs Semantic Kernel（.NET/Java 企业场景）；**向量数据库**：Chroma（轻量开发）vs Milvus（高性能分布式）vs Qdrant（Rust 高性能+过滤强）vs Weaviate（Hybrid 原生支持）vs Pinecone（全托管无运维）vs PGVector（复用 PostgreSQL）— 按规模/时延/过滤能力/运维成本/中文支持五维度对比；**Embedding 模型类型**：闭源 API（OpenAI/Cohere/通义）vs 开源模型（BGE/m3e/stella/GTE）vs 本地部署（Ollama/Infinity）；**生成模型类型**：闭源（OpenAI/通义/文心/DeepSeek）vs 开源本地（Qwen/Llama/DeepSeek via Ollama/vLLM）；**搜索引擎**：Elasticsearch vs Meilisearch vs BM25 内存库 vs RediSearch；**完整技术选型决策框架**：根据场景（客服 Q&A/企业 Wiki/合同审查/医疗合规）匹配技术栈的决策树与权衡矩阵 | L7（学完全部核心链路后，需要全景认知来指导后续选型） | 能画出 RAG 技术选型全景图，能根据业务需求（成本预算/时延要求/精度要求/私有化需求/中文支持/团队技能）做出理性的技术栈选择，理解每种选择背后的 trade-off |
| **L9** | RAG 拒绝策略与兜底机制 | 检索空结果处理；置信度评估（检索分数阈值、生成概率）；"我不知道"回答模板设计（透明告知+部分信息+引导用户）；多层兜底链（本地检索→联网搜索→人工转接）；知识库覆盖度检测；拒答 vs 幻觉的权衡 | L7（需能够评估答案质量才能判断何时该拒答） | 能让 RAG 系统在知识库无覆盖时诚实拒答，而非编造幻觉——这是客服场景的底线能力 |
| **L10** | 对话式 RAG 与多轮对话管理 | Chat history 管理策略（滑动窗口/摘要压缩/关键轮次提取）；追问与指代消解（"它多少钱？"→ 关联上文产品）；会话状态管理与上下文注入；多模型对话 API 统一适配层（OpenAI/通义/DeepSeek）；history 长度与延迟的权衡 | L9（拒答+多轮兜底天然关联） | 能实现带上下文记忆的多轮 RAG 对话，正确处理指代消解和追问，支持多模型切换 |
| **L11** | Prompt Engineering 在 RAG 中的实践 | System prompt 设计模式（角色设定+知识范围+拒答指令+输出格式）；Grounding prompt 模板（引用来源、逐条对应）；Few-shot 示例在 RAG prompt 中的应用；Chain-of-Thought 推理提示词；生成参数调优（temperature/top_p/max_tokens 的 RAG 场景最优值）；多模型 prompt 适配差异 | L7（需理解 answer 生成与评估） | 能针对不同业务场景设计高质量的 RAG prompt，降低幻觉率 30-50%，提升回答格式一致性 |
| **L12** | Embedding 模型全景与选型 | OpenAI/BGE/Cohere/Jina 四类模型横向对比；MTEB 榜单解读；中文 embedding 专项评测（BGE-zh/m3e/text2vec/stella/GTE）；模型切换适配层设计（统一接口、配置化切换）；本地 Ollama/Infinity 部署私有化 embedding；embedding 维度对检索速度/质量的影响；基于 L4 评估体系做 embedding 模型的 A/B 对比实验 | L5（需已有检索系统做对比实验）、L8（需技术栈全景认知指导选型） | 能根据场景（中文/英文/垂直领域/成本敏感/私有化）选择最优 embedding 模型，支持模型热切换 |

> ⚠️ **模块二完成后的状态**：学习者不仅知道"有哪些选择"，还知道"怎么选"——技术栈选型有依据、拒答策略完善、多轮对话可用、prompt 精良、embedding 选型有数据支撑。接下来要解决"如何让系统稳定跑在生产环境"。

## L8 详细内容规划：RAG 技术栈全景图

> 这节课是模块二的开篇，意义重大——让学习者在做任何技术选型之前，先看清楚全局有哪些选择，以及每种选择背后的 trade-off。

### 课程结构

**第一部分：技术栈全景地图（建立全局认知）**

以 RAG 核心链路（Chunk → Embed → Store → Retrieve → Rerank → Generate → Evaluate）为横轴，以可选技术方案为纵轴，画出一张完整的「RAG 技术选型地图」。

**第二部分：编程语言选择**

| 语言 | RAG 生态成熟度 | 优势 | 劣势 | 适用场景 |
|------|:---:|------|------|----------|
| Python | ⭐⭐⭐⭐⭐ | 生态最全（LangChain/LlamaIndex/HuggingFace）；所有模型官方 SDK 优先支持；社区最大 | GIL 性能瓶颈；部署镜像大 | 90% 的 RAG 场景首选 |
| JavaScript/TS | ⭐⭐⭐⭐ | 前端全栈；Edge Runtime 友好；LangChain.js/LlamaIndex.TS 可用 | 数据处理库弱于 Python；部分模型无官方 SDK | 前端集成 RAG、Edge 部署 |
| Go | ⭐⭐⭐ | 高性能；低内存；部署简单（单二进制） | RAG 框架几乎空白；需手写大量胶水代码 | 高性能检索服务、API 网关层 |
| Java/Kotlin | ⭐⭐⭐ | Spring AI + Semantic Kernel；企业已有技术栈 | 生态碎片化；响应慢 | 大型企业 Java 技术栈接入 |

**第三部分：LLM 调用框架对比**

| 框架 | 定位 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|----------|
| **LangChain** | 通用 LLM 应用框架 | 生态最大；组件最全（500+ integrations）；社区最活跃；LangSmith 可观测 | 抽象层太多；版本迭代快、Breaking Change 多；调试困难；适合 Demo 但上生产需额外工作 | 快速原型、复杂 Chain/Agent |
| **LlamaIndex** | 数据驱动的 RAG 框架 | RAG 专项最强；数据连接器丰富（160+）；索引结构灵活（Tree/Keyword/Graph）；文档解析强 | 非 RAG 场景弱；学习曲线陡 | RAG 为核心的应用 |
| **原生 SDK** | 直接调用 API | 最轻量；完全可控；无框架锁定；调试直观 | 需手写所有编排逻辑；重复造轮子 | 简单 RAG 管线、对框架有顾虑的团队 |
| **Haystack** | 企业级 NLP 框架 | Pipeline 抽象清晰；生产部署友好；OpenSearch/ES 集成强 | 社区较小；中文生态弱 | 欧洲/OpenSearch 生态企业 |
| **Dify/FastGPT** | 低代码 RAG 平台 | 可视化编排；开箱即用；非技术人员可维护 | 灵活性受限；定制开发困难 | 内部工具、非技术团队维护 |

**第四部分：向量数据库对比**

| 向量库 | 类型 | 优势 | 劣势 | 最佳场景 |
|--------|------|------|------|----------|
| **Chroma** | 嵌入式 | 5 行代码启动；Python 原生；开发体验最好 | 不支持分布式；百万级性能下降；生产需迁移 | 本地开发、Demo、PoC |
| **Milvus** | 分布式 | 十亿级性能；GPU 索引；混合查询强；云原生 | 部署运维复杂（需 etcd/Pulsar/MinIO）；资源消耗大 | 大规模生产、高性能需求 |
| **Qdrant** | 分布式 | Rust 编写性能极高；过滤能力强（payload index）；单机也能跑 | 生态比 Milvus 小；中文社区小 | 需要高性能过滤的场景 |
| **Weaviate** | 分布式 | Hybrid 搜索原生支持；GraphQL 接口；多模态原生 | 资源消耗高（JVM）；中文文档少 | Hybrid 搜索优先的场景 |
| **Pinecone** | 云托管 | 零运维；自动扩缩容；Serverless 可用 | 贵；数据出境风险；无法私有化 | 不想管基础设施的团队 |
| **PGVector** | 插件 | 复用 PostgreSQL；ACID 事务；元数据+向量同一张表 | 十亿级性能不如 Milvus；索引构建慢 | 已有 PostgreSQL、中小规模 |
| **Elasticsearch** | 搜索引擎+向量 | 不需要多一套存储；BM25+向量天然一家；运维体系成熟 | 向量性能不如专用库；内存消耗大 | 已有 ES 基础设施的团队 |

**第五部分：Embedding 模型分类与对比**

| 类别 | 代表模型 | 优势 | 劣势 | 适用场景 |
|------|----------|------|------|----------|
| 闭源 API | OpenAI text-embedding-3-small/large、Cohere Embed v3 | 效果最好；零运维；多语言强 | 成本随量增长；数据出境；无法微调 | 对精度要求最高、预算充足 |
| 中文开源 | BGE-zh/m3e/stella/GTE/text2vec | 中文效果接近 OpenAI；可私有化部署 | 英文弱；需要 GPU | 国内场景、数据不出境 |
| 多语言开源 | BGE-M3/Jina-CLIP/Multilingual-E5 | 中英文兼顾；社区活跃 | 模型大（0.5-2B） | 多语言知识库 |
| 本地部署 | Ollama/Infinity/Xinference | 完全离线；无 API 费用；数据安全 | 需要 GPU；效果不如 API | 军工/金融等高合规场景 |

**第六部分：生成模型分类与对比**

| 类别 | 代表模型 | 优势 | 劣势 | 适用场景 |
|------|----------|------|------|----------|
| 海外闭源 | GPT-4o/Claude 4/Gemini | 综合能力最强；推理强 | 贵；数据出境；延迟较高 | 对答案质量要求极高 |
| 国内闭源 | 通义千问/文心一言/DeepSeek/Kimi | 中文理解强；价格低；数据合规 | 复杂推理弱于 GPT-4o | 国内场景、成本敏感 |
| 开源本地 | Qwen/Llama/DeepSeek via Ollama/vLLM | 完全私有化；零 API 费用；可微调 | 需要大量 GPU；效果不如闭源 | 高合规场景、大规模部署摊薄硬件成本 |

**第七部分：技术选型决策框架**

以典型业务场景为输入，给出推荐技术栈组合：

| 场景 | 推荐技术栈 | 选型理由 |
|------|-----------|----------|
| **个人学习/原型** | Python + Chroma + OpenAI | 最简开发体验 |
| **中小企业客服 Q&A** | Python + LlamaIndex + Qdrant + BGE-zh + 通义千问 | 成本可控+中文优化+无数据出境 |
| **大型企业知识库** | Python + LangChain + Milvus + ES + BGE-M3 + DeepSeek | 高性能+混合检索+可私有化 |
| **高合规（军工/金融）** | Python + 原生 SDK + PGVector + Ollama(BGE-zh + Qwen) | 全离线+全可控 |
| **海外多语言** | Python + LlamaIndex + Pinecone + Cohere + GPT-4o | 零运维+多语言强 |
