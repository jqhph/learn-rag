一、置信度评估三信号

信号获取方式优点局限
检索分数result.score（余弦相似度 / BM25 分数）最容易获取；零额外成本不同模型分数分布不同；高分数不代表高相关性
Cross-Encoder 相关性sentence-transformers Cross-Encoder 或 LLM 判断比检索分数更准；能识别"高分噪音"需要额外模型推理；增加时延 ~50-200ms
生成 logprobsresponse.logprobs（部分 API 支持）反映 LLM 自身的自信度不是所有模型支持；解释困难

二、置信度决策逻辑

def decide(query, results):
    if not results:
        return ("refuse", "no_knowledge")
    top_score = max(r.score for r in results)
    if top_score < SCORE_THRESHOLD:
        if top_score < SCORE_THRESHOLD / 2:
            return ("refuse", "low_confidence")
        else:
            return ("refuse", "low_confidence")  # 可改写重试
    relevance = cross_encoder_verify(query, results)
    if relevance >= 0.5:
        return ("answer", None)
    elif relevance >= 0.3:
        return ("partial", "partial_knowledge")
    else:
        return ("refuse", "low_confidence")

三、三种拒答模板

类型触发条件模板核心关键点
🟡 透明拒答检索为空 / 完全不相关"我的知识库中没有关于 XX 的信息"归因于系统局限而非用户问题
🟢 部分信息部分相关性，但不完整"关于 XX 我可以告诉您 YY，但不完整"明确标出信息边界
🔵 引导拒答问题模糊 / 需要澄清"为了更准确地帮助您，能否提供更多关于 ZZ 的信息？"给用户具体可操作的澄清方向

所有模板都必须包含：透明原因 + 后续行动建议（换种问法 / 联系人工 / 帮助中心）。

四、多层兜底链

层级操作时延成本触发条件
1️⃣本地向量检索 + 置信度评估~200ms≈0所有查询
2️⃣改写查询重试（Query Rewrite）+~500ms少量 LLM token第一级低置信度时
3️⃣联网搜索+~1-3s搜索 API 费本地检索失败 + 需要实时信息
4️⃣人工转接 + 工单生成+~5s客服工资复杂 / 敏感 / 高价值问题

原则：按成本递增——从便宜的本地操作开始，逐步升到昂贵的操作。

五、覆盖度检测指标

指标计算目标值说明
回答率回答数 / 总查询数≥ 70%（稳定）不是越高越好——要同时保证答案质量
拒答率拒答数 / 总查询数≤ 20%突然升高 = 检索退化信号
转人工率转接数 / 总查询数≤ 15%过高意味着知识库缺口大
分类覆盖度按知识库分类统计回答率各分类 ≥ 60%发现具体哪个分类缺知识

六、阈值校准

不同 embedding 模型的常用阈值经验值：

  • OpenAI text-embedding-3-small → min_score = 0.65
  • BGE-zh 系列 → min_score = 0.50
  • BGE-M3 → min_score = 0.45
  • 本地小模型 (Ollama) → min_score = 0.30

以上是经验起点——必须用你的数据和场景进行 A/B 校准。校准方法：采样 200 条查询 → 人工标注"可用/不可用" → 遍历阈值找最大 F1。

七、检查清单

  • ☐ 检索为空时不硬答
  • ☐ 检索分数 + Cross-Encoder 双重验证
  • ☐ 拒答回复包含透明原因 + 后续建议
  • ☐ 多层兜底链（改写 → 联网 → 人工）
  • ☐ 所有拒答被记录到日志
  • ☐ 每周生成覆盖度报告
  • ☐ 拒答率被监控，异常升高有告警
  • ☐ 阈值已针对你的 embedding 模型校准
