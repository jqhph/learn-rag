一、System Prompt "四块积木"

积木核心问题一句话模板常见错误
① 角色设定 你是谁 你是一个{公司}客服助手，专业、准确、友好。 不设角色 → 鼓励幻觉
② 知识范围 你知道什么 知识来源仅限于下方"检索结果"，不要使用训练数据中的知识。 知识边界模糊 → LLM 混用训练数据
③ 拒答指令 什么不能做 检索结果不包含答案时，直接说"暂无相关信息"，不要编造。 没有拒答 → LLM 宁愿编也不说不知道
④ 输出格式 怎么回答 每条信息标注[来源N]，使用分点结构。 不指定格式 → 风格不稳定
关键词： A/B 测试每块积木——固定其他三块，只换一块，用 RAGAS 指标量化改进。

模式Faithfulness幻觉率场景
strict0.92-0.98<5%推荐 金融/医疗/法律

三、Few-shot 示例策略

场景推荐做法示例数
格式敏感（表格/对比）必须加1-2 个
多步推理推荐加1-2 个
情感/语气控制推荐加1 个
标准 QA一般不需要—
动态选择 > 固定给： 用 embedding 或关键词匹配，选最相关的 1-2 个示例。比固定给 3 个不相关的示例更省 token 且效果更好。

四、Chain-of-Thought 模式

模式做法Token 开销场景
direct"请一步步思考"+50-100多跳推理、对比
structured已知信息→需要推理→结论+100-300可审计的严谨回答
verify先检视每个声称是否有来源+150-400高准确度场景
注意： CoT 会增加延迟（约 1.5x-2.5x）。只在需要多步推理时使用。简单的 FAQ 问答不需要 CoT。

六、多模型 Prompt 适配差异

模型最适配的 prompt 风格CoT 触发词注意事项
GPT-4o / 4o-mini指令式（最省事）"请一步步思考"指令遵循最强，不需要过度强调

七、快速排查清单

回答质量差，从哪开始查？

  1. 模型在用自己的知识而非检索结果？ → 切到 strict grounding 模式 + 加"不要使用训练数据"指令
  2. 引用格式不稳定？ → 检查 temperature（应 < 0.3）+ 加 few-shot 示例
  3. 拒答不及时（编造答案）？ → 确认 system prompt 中有明确的拒答指令积木
  4. 回答太啰嗦？ → 切换输出格式为"简洁回答"或降低 max_tokens
  5. 换了模型效果大降？ → 调用 adapt_system_prompt() 做模型适配
  6. 多轮对话历史越长质量越差？ → 降低 window_size + 确认滑动窗口裁剪生效
