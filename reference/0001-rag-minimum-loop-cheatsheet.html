RAG 最小闭环速查表

用于客服问答 Demo：把可维护的知识放在外部文档里，让模型先检索、再基于证据回答。

七步闭环

1. Source

准备 FAQ、产品说明、售后政策、工单模板等原始资料。

2. Chunk

把长文档切成能独立表达意思的小片段，并保留来源元数据。

3. Embed

用 embedding 模型把每个片段转成向量，用于语义相似度搜索。

4. Store

把向量、原文、标题、URL、版本、业务标签存入向量库集合。

5. Retrieve

用户提问时，先把问题向量化，再取最相似的 top-k 片段。

6. Generate

把问题和检索片段一起放入提示词，让 LLM 只基于证据回答。

7. Evaluate

检查答案是否正确、是否引用来源、是否拒答未知问题。

最容易犯错的点

  1. 只做生成，不做检索：模型会凭记忆回答，难以更新和追溯。
  2. 切块太大：检索结果含噪声，模型难以定位关键条款。
  3. 切块太小：上下文丢失，答案缺少完整依据。
  4. 没有元数据：答案无法显示来源，也难以按产品线或权限过滤。
  5. 没有拒答规则：资料里没有答案时，系统仍然编造。

Demo 阶段默认参数

从这些默认值开始，不要一上来调复杂架构：chunk_size 约 300-800 中文字，overlap 约 50-100 字，top_k 取 3-5，先用本地 Chroma，先用 Python。
