二、生产级 Dockerfile（多阶段构建）

# 阶段一：Builder — 安装依赖
FROM python:3.12.4-slim AS builder
WORKDIR /build
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# 阶段二：运行镜像
FROM python:3.12.4-slim
RUN groupadd -r rag && useradd -r -g rag rag
WORKDIR /app
COPY --from=builder /install /usr/local
COPY app/ ./app/
USER rag
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
黄金法则：slim 镜像 + 多阶段 + 非 root + HEALTHCHECK + 分离 COPY 层（依赖→代码）

关键配置片段

services:
  rag-api:
    build: .
    restart: unless-stopped
    environment:
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      - LLM_PROVIDER=${LLM_PROVIDER:-openai}
      - CHROMA_HOST=chroma
      - REDIS_HOST=redis
    depends_on:
      chroma:  { condition: service_healthy }
      redis:   { condition: service_healthy }
    networks: [rag-network]

  chroma:
    image: chromadb/chroma:0.5.0
    restart: unless-stopped
    volumes: [./data/chroma:/chroma/chroma]
    healthcheck: # ...

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    command: redis-server --appendonly yes --requirepass ${REDIS_PASSWORD:-}
    volumes: [./data/redis:/data]

  nginx:
    image: nginx:alpine
    ports: ["80:80"]
    volumes: [./nginx.conf:/etc/nginx/conf.d/default.conf:ro]

networks:
  rag-network: { driver: bridge }

五、重启策略与健康检查

重启策略行为推荐场景
restart: unless-stopped非手动停止时崩溃自动重启✅ 生产服务首选
restart: always总是重启（含手动 stop）少用
restart: on-failure仅退出码非 0 时重启一次性任务
restart: no从不重启调试

依赖启动顺序

services:
  rag-api:
    depends_on:
      chroma:  { condition: service_healthy }
      redis:   { condition: service_healthy }

RAG API 会在 Chroma 和 Redis 的健康检查通过后才启动。

六、JSON 结构化日志

# 日志格式
{"time":"2026-06-20T10:30:45","level":"INFO","logger":"RAGPipeline",
 "message":"检索完成","query":"RAG是什么","results":5,"latency_ms":320}

# 查看日志
docker compose logs -f rag-api

# 过滤错误
docker compose logs rag-api | grep '"level":"ERROR"'

# 分析延迟
docker compose logs rag-api | grep '检索完成' | \
  python -c "import sys,json; lines=[json.loads(l) for l in sys.stdin]; \
  ls=[l['latency_ms'] for l in lines if 'latency_ms' in l]; \
  print(f'平均: {sum(ls)/len(ls):.0f}ms' if ls else '无数据')"

七、常用命令速查

命令用途
docker compose up -d后台启动全部服务
docker compose down停止并移除容器
docker compose ps查看服务状态
docker compose logs -f实时查看全部日志
docker compose logs -f rag-api查看指定服务日志
docker compose up -d --build rag-api重新构建并启动单个服务
docker compose exec rag-api python ...在运行中的容器内执行命令
docker compose restart rag-api重启单个服务
docker compose down -v停止并删除数据卷（⚠️ 丢数据）

八、Nginx 反向代理关键配置

upstream rag_api { server rag-api:8000; }
server {
    listen 80;
    location /api/ {
        rewrite ^/api(/.*)$ $1 break;
        proxy_pass http://rag_api;
        proxy_read_timeout 120s;
    }
    location /health { proxy_pass http://rag_api/health; }
}

九、故障排查 checklist

  • 🔄 容器不断重启？→ docker compose logs 查看原因
  • 🔑 API Key 无效？→ 检查 .env 文件是否存在且格式正确
  • 🔌 连不上 Chroma？→ 确认 condition: service_healthy 和网络名匹配
  • 🚪 端口被占？→ 用 lsof -i :6379 或 netstat -tlnp 查找
  • 📦 502 Bad Gateway？→ 先直接调用 RAG API 排除 Nginx 问题
  • 🐢 LLM 响应慢超时？→ 增大 proxy_read_timeout
