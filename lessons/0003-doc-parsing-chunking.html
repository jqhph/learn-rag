Lesson 0003

文档解析与文本切块：从真实文件到可检索片段

本课目标：告别手写 DOCS 列表，学会从真实的 PDF、Markdown 文件中提取文本，并用正确的切块策略把它们变成高质量的可检索片段。

你今天要获得的能力

学完本课，你将拥有一个 loader.py 模块，它能：

  • 自动识别文件类型，从 PDF、Markdown、HTML 中提取纯文本
  • 用 token 感知的切块策略把长文档切成语义完整的片段
  • 控制 chunk_size 和 overlap，理解它们如何影响检索质量
  • 替换掉第 2 课的硬编码 DOCS 列表，让你的 RAG 系统真正"吃"进真实文件

这是从"玩具 Demo"到"企业可用原型"的关键一步。真实企业的知识库不会是一个 Python 列表——它们是成百上千个 PDF、网页和文档。

主推荐阅读：LangChain Text Splitters 概念、tiktoken 仓库、pdfplumber 文档。速查表：文档解析与切块速查表。

为什么这一步如此重要？

回顾第 2 课，你的知识库长这样：

DOCS = [
    {"id":"doc-001", "text":"7 天无理由退货仅适用于未拆封商品...", "source":"return-policy.md", ...},
    {"id":"doc-002", "text":"普通发票在订单完成后 24 小时内...", "source":"invoice-policy.md", ...},
    # ... 每条都是一句话，完美切好
]

但真实场景下，你的 return-policy.md 可能是 2000 字的长文，product-manual.pdf 可能是 50 页。如果你把整篇文档当成一个 chunk 扔进向量库，会发生什么？

长文档不切块的三个后果

  1. 检索精度低：用户问"退货要几天"，整篇 2000 字的政策文档向量里混杂了退货、换货、退款、发票等所有主题，语义被稀释，相似度算不准。
  2. 上下文浪费：LLM 的上下文窗口是有限的。你把 2000 字的整篇文档塞进提示词，但其中只有 200 字和用户问题相关——剩下 1800 字白白消耗 token 预算。
  3. 来源引用粗糙：你只能说"来自 return-policy.md"，无法精确到"第三章第 2 条"。审计追溯要求你定位到具体段落。

切块的目标很明确：把长文档切成语义自包含的小片段，让每个片段聚焦一个主题，提高检索精度，降低上下文噪音，同时保留来源追溯能力。

第一步：从文件中提取文本

不同格式需要不同的解析器。下面是三种最常见企业文档格式的解析方法。

Markdown —— 最简单

Markdown 本质是纯文本，直接读即可。如果你想保留结构信息（标题层级），可以用 mistune 或 markdown-it 解析 AST，但初版直接按行读就够了。

def load_markdown(filepath: str) -> str:
    with open(filepath, "r", encoding="utf-8") as f:
        return f.read()

PDF —— 最常用也最容易踩坑

PDF 是二进制格式，文本提取面临几个挑战：文字可能嵌入在图片里（需要 OCR）、双栏布局可能打乱阅读顺序、表格数据可能变成碎片文本。对中文客服文档来说，最常用的方案是 pdfplumber：它能较好地保留布局信息，提取准确率比 PyPDF2 高，API 也直观。

# 先安装：pip install pdfplumber
import pdfplumber

def load_pdf(filepath: str) -> str:
    text_parts = []
    with pdfplumber.open(filepath) as pdf:
        for page in pdf.pages:
            page_text = page.extract_text()
            if page_text:
                text_parts.append(page_text)
    return "\n\n".join(text_parts)

如果 PDF 是扫描件（图片型 PDF），你需要 OCR。推荐 pytesseract + pdf2image，但那是另一个话题；企业知识库通常以电子文档为主，先假定可提取文本。

HTML —— 网页和导出文档

用 BeautifulSoup 提取可见文本，去掉 script/style 标签噪音。

# 先安装：pip install beautifulsoup4
from bs4 import BeautifulSoup

def load_html(filepath: str) -> str:
    with open(filepath, "r", encoding="utf-8") as f:
        soup = BeautifulSoup(f.read(), "html.parser")
    # 去掉不可见元素
    for tag in soup(["script", "style", "nav", "footer", "header"]):
        tag.decompose()
    return soup.get_text(separator="\n", strip=True)

格式检测策略

生产环境用文件扩展名 + MIME 类型双重检测。本课 Demo 用扩展名就够了：

LOADERS = {
    ".md":   load_markdown,
    ".txt":  load_markdown,   # 纯文本复用同一函数
    ".pdf":  load_pdf,
    ".html": load_html,
    ".htm":  load_html,
}

第二步：切块策略 —— 本课的核心

切块不是简单地把文档按字符数等分。好的切块策略要同时满足三个目标：

  1. 语义完整：一个 chunk 应该是一个自包含的信息单元——读者不需要看其他 chunk 就能理解它的意思。
  2. 大小可控：每个 chunk 的 token 数应该在 embedding 模型的最佳范围内（text-embedding-3-small 单次最多 8191 tokens，但 300-800 tokens 的片段检索效果最好）。
  3. 边界自然：切在段落、句子边界上，而不是把一个词拦腰截断。

策略 A：固定大小切块（Fixed-size Chunking）

最直观的方法：设定 chunk_size（字符数），从头到尾滑动窗口，相邻窗口之间保留 overlap。

无 Overlap

认后可全额退款。退款到账 1-3 天。"

Overlap 的作用：当用户的答案恰好跨在两个 chunk 的边界上时，overlap 给检索提供了"冗余"，让关键信息至少完整地出现在一个 chunk 里。代价是存储和 embedding 计算量增加，但通常 10-20% 的 overlap 是值得的。

策略 B：Token 感知切块（Token-aware Chunking）

字符数不等于 token 数。中文尤其如此：一个中文字 ≈ 1.5-2 个 token（在 cl100k_base 编码下），一个英文单词 ≈ 1-2 个 token。用字符数估算 token 数会有 50% 以上的误差。用 tiktoken 精确计算：

# pip install tiktoken
import tiktoken

encoder = tiktoken.get_encoding("cl100k_base")  # text-embedding-3-small 使用的编码

def chunk_by_tokens(text: str, chunk_size: int = 500, overlap: int = 100) -> list[dict]:
    """按 token 数切块，返回带位置信息的片段列表。"""
    tokens = encoder.encode(text)
    chunks = []
    i = 0
    chunk_idx = 0
    while i < len(tokens):
        chunk_tokens = tokens[i:i + chunk_size]
        chunk_text = encoder.decode(chunk_tokens)
        chunks.append({
            "text": chunk_text,
            "chunk_index": chunk_idx,
            "token_count": len(chunk_tokens),
            "char_start": len(encoder.decode(tokens[:i])) if i > 0 else 0,
        })
        i += (chunk_size - overlap)
        chunk_idx += 1
    return chunks

cl100k_base 是 text-embedding-3-small/3-large 和 GPT-4 系列使用的编码。如果你换用其他 embedding 模型（如 Cohere、BGE），需要用对应的 tokenizer。tiktoken 文档 列出了所有可用编码。

策略 C：语义边界切块（Semantic Chunking）

最理想但也最复杂的策略：在段落、句子、甚至语义转折点处切分。基本思路是：

  1. 先按段落（\n\n）切分
  2. 如果段落太长，再按句子（。！？）切分
  3. 合并短片段直到接近 chunk_size
  4. 在片段之间保留 overlap
import re

def chunk_semantic(text: str, max_tokens: int = 500, overlap_sentences: int = 1) -> list[str]:
    """按段落→句子层级切分，合并短片段至 max_tokens 附近。"""
    # 1. 按段落切分
    paragraphs = [p.strip() for p in text.split("\n\n") if p.strip()]

    # 2. 段落内部按句子切分（中英文句号）
    sentences = []
    for para in paragraphs:
        sents = re.split(r'(?<=[。！？.!?])\s*', para)
        sentences.extend([s.strip() for s in sents if s.strip()])

    # 3. 合并句子到目标 token 数
    chunks = []
    current = []
    current_tokens = 0
    for sent in sentences:
        sent_tokens = len(encoder.encode(sent))
        if current_tokens + sent_tokens > max_tokens and current:
            chunks.append(" ".join(current))
            # 保留 overlap_sentences 个句子作为下一个 chunk 的开头
            current = current[-overlap_sentences:] if len(current) > overlap_sentences else []
            current_tokens = sum(len(encoder.encode(s)) for s in current)
        current.append(sent)
        current_tokens += sent_tokens
    if current:
        chunks.append(" ".join(current))
    return chunks

三种策略的选择指南

策略适用场景实现复杂度检索质量
固定字符快速原型、英文文档⭐⭐⭐
Token 感知精准控制上下文预算⭐⭐⭐⭐⭐
语义边界中文长文、政策/法规文档⭐⭐⭐⭐⭐⭐⭐

生产环境通常结合使用：语义边界做第一层切分，token 感知做第二层大小控制。LangChain 的 RecursiveCharacterTextSplitter 和 LlamaIndex 的 SentenceSplitter 都是这个思路的工业级实现。

第三步：完整代码 —— loader.py

现在把所有东西整合到一个模块里。创建 loader.py，它会：遍历 docs/ 目录 → 识别文件类型 → 提取文本 → 切块 → 返回结构化数据。

"""
loader.py — 文档加载与切块模块
支持格式：.md / .txt / .pdf / .html
切块策略：语义边界 + token 感知（中文友好）
"""
import os, re
from typing import Optional

# ── 可选依赖：按需导入 ──────────────────────
try:
    import pdfplumber
    HAS_PDF = True
except ImportError:
    HAS_PDF = False

try:
    from bs4 import BeautifulSoup
    HAS_HTML = True
except ImportError:
    HAS_HTML = False

try:
    import tiktoken
    HAS_TIKTOKEN = True
    ENCODER = tiktoken.get_encoding("cl100k_base")
except ImportError:
    HAS_TIKTOKEN = False
    ENCODER = None


# ── 1. 格式解析器 ──────────────────────────
def load_text(filepath: str) -> str:
    with open(filepath, "r", encoding="utf-8") as f:
        return f.read()

def load_pdf(filepath: str) -> str:
    if not HAS_PDF:
        raise ImportError("需要安装 pdfplumber: pip install pdfplumber")
    parts = []
    with pdfplumber.open(filepath) as pdf:
        for page in pdf.pages:
            t = page.extract_text()
            if t:
                parts.append(t)
    return "\n\n".join(parts)

def load_html(filepath: str) -> str:
    if not HAS_HTML:
        raise ImportError("需要安装 beautifulsoup4: pip install beautifulsoup4")
    with open(filepath, "r", encoding="utf-8") as f:
        soup = BeautifulSoup(f.read(), "html.parser")
    for tag in soup(["script", "style", "nav", "footer"]):
        tag.decompose()
    return soup.get_text(separator="\n", strip=True)

LOADERS = {
    ".md": load_text, ".txt": load_text,
    ".pdf": load_pdf, ".html": load_html, ".htm": load_html,
}


# ── 2. 切块引擎 ────────────────────────────
def count_tokens(text: str) -> int:
    """用 tiktoken 精确计数；未安装时用字符数近似（中文约 0.6 token/字）。"""
    if HAS_TIKTOKEN:
        return len(ENCODER.encode(text))
    return int(len(text) * 0.6)  # 粗略估算


def split_sentences(text: str) -> list[str]:
    """按中英文句号、问号、感叹号切分句子。"""
    raw = re.split(r'(?<=[。！？.!?])\s*', text)
    return [s.strip() for s in raw if s.strip()]


def chunk_semantic(
    text: str,
    chunk_tokens: int = 500,
    overlap_sentences: int = 1,
) -> list[dict]:
    """
    语义边界切块：
    段落 → 句子 → 合并到目标 token 数 → 保留句子级 overlap
    返回 [{"text":..., "chunk_index":..., "token_count":..., "char_start":...}]
    """
    paragraphs = [p.strip() for p in text.split("\n\n") if p.strip()]
    all_sentences = []
    for para in paragraphs:
        all_sentences.extend(split_sentences(para))

    chunks = []
    buf, buf_tokens = [], 0
    char_cursor = 0
    chunk_idx = 0

    # 记录每个句子的字符位置
    sent_positions = []
    pos = 0
    for s in all_sentences:
        # 在原文中定位（近似）
        idx = text.find(s, pos)
        sent_positions.append(idx if idx >= 0 else pos)
        pos = (idx if idx >= 0 else pos) + len(s)

    for si, sent in enumerate(all_sentences):
        st = count_tokens(sent)
        if buf_tokens + st > chunk_tokens and buf:
            chunk_text = "".join(buf)
            chunks.append({
                "text": chunk_text,
                "chunk_index": chunk_idx,
                "token_count": buf_tokens,
                "char_start": sent_positions[max(0, si - len(buf))],
            })
            # Overlap: 保留最后 N 句
            keep = min(overlap_sentences, len(buf))
            buf = buf[-keep:] if keep > 0 else []
            buf_tokens = sum(count_tokens(s) for s in buf)
            chunk_idx += 1
        buf.append(sent)
        buf_tokens += st

    # 最后一个 chunk
    if buf:
        chunk_text = "".join(buf)
        chunks.append({
            "text": chunk_text,
            "chunk_index": chunk_idx,
            "token_count": buf_tokens,
            "char_start": sent_positions[max(0, len(all_sentences) - len(buf))],
        })

    return chunks


# ── 3. 统一入口 ────────────────────────────
def load_docs(
    docs_dir: str = "./docs",
    chunk_tokens: int = 500,
    overlap_sentences: int = 1,
) -> list[dict]:
    """
    主函数：遍历 docs_dir，加载所有支持的文件，切块后返回。
    返回格式与第 2 课的 DOCS 兼容：
    [{"id": "return-policy.md#0", "text": "...", "source": "return-policy.md",
      "chunk_index": 0, "token_count": 480, "format": "md"}, ...]
    """
    results = []
    for fname in sorted(os.listdir(docs_dir)):
        _, ext = os.path.splitext(fname)
        loader = LOADERS.get(ext.lower())
        if not loader:
            continue

        fpath = os.path.join(docs_dir, fname)
        print(f"📄 加载 {fname} ...")
        try:
            raw_text = loader(fpath)
        except Exception as e:
            print(f"  ⚠️ 跳过 {fname}: {e}")
            continue

        chunks = chunk_semantic(raw_text, chunk_tokens, overlap_sentences)
        for ch in chunks:
            results.append({
                "id": f"{fname}#{ch['chunk_index']}",
                "text": ch["text"],
                "source": fname,
                "chunk_index": ch["chunk_index"],
                "token_count": ch["token_count"],
                "char_start": ch["char_start"],
                "format": ext.lower(),
            })

        print(f"  ✓ 切出 {len(chunks)} 个片段（共 {count_tokens(raw_text)} tokens）")

    return results


# ── CLI 预览（方便调试）────────────────────
if __name__ == "__main__":
    docs = load_docs()
    for d in docs:
        print(f"\n{'─'*60}")
        print(f"📌 {d['id']}  |  {d['token_count']} tokens  |  {d['source']}")
        print(f"{d['text'][:200]}...")

为什么要保持接口兼容？

因为 ingest_docs() 只关心三样东西：text（要向量化的文本）、id（去重用）、source（追溯用）。loader.load_docs() 返回的字典格式和原来手写的 DOCS 完全对齐，只是多了 chunk_index、token_count、format 等额外字段。这是好的模块设计——关注点分离：loader 负责"从哪来、怎么切"，管道负责"怎么存、怎么查"。

练习一：切块参数敏感度

同一个文档，chunk_tokens=200 vs chunk_tokens=800，切出来的片段数量和质量有什么不同？

练习二：动手实验

对比实验：overlap=0 vs overlap=1

在 loader.py 里，先用 overlap_sentences=0 切分你的文档，挑一个跨 chunk 边界的问题去检索，看答案是否完整。再改成 overlap_sentences=1，重新入库和检索，对比答案质量。

关键观察：如果一条退货政策被切在了"质量问题"和"15 天内"之间，没有 overlap 时模型可能只看到半句话，回答不完整。有 overlap 后完整的条款至少出现在一个 chunk 中。

中文切块的特殊注意事项

五个中文特有的坑

  1. 没有空格分词：英文可以按空格和单词边界切，中文不行。按字符切会导致词被拆散（"退货政策"→"退货"+"政策"），但实践中 embedding 模型对子词级别的中文语义有足够的鲁棒性，字符级切块不影响检索质量。真正该担心的是句子边界。
  2. 句号不统一：中文句号是 。（全角），英文是 .（半角）。你的切句正则要同时匹配：[。！？.!?]。
  3. 引号和括号：中文使用全角引号" "和括号（），它们内部的句号不应视为句子边界。但初版可以接受这个瑕疵——生产环境用专业分句器（如 jieba 的 cut_sentence 或 spaCy 的 zh_core_web_sm）。
  4. tiktoken 对中文的效率：cl100k_base 编码下，1 个中文字 ≈ 1.5-2 个 token，1 个英文单词 ≈ 1-2 个 token。所以 500 tokens 的 chunk 大约能装 250-350 个中文字。这意味着中文文档比英文文档信息密度更高——同样 token 预算下，中文能塞更多语义。
  5. 数字和英文字母混排：企业文档常见"型号 ABC-123"、"2024 年第 3 季度"等。tiktoken 对混合文本的 token 计数比纯中文更费 token（因为 ASCII 字符每个 1 token，中文每个 1.5-2 token）。

你现在应该记住的五句话

  1. 不切块的 RAG 是低精度的 RAG。整篇文档向量化 = 语义被稀释，检索命中全靠运气。
  2. 切块的第一原则是语义完整。宁可在句子边界处多留几个 token，也不要把一个完整的意思拦腰截断。
  3. Overlap 是检索质量的保险。10-20% 的 overlap 能极大降低"答案被切在边界上"的风险。
  4. 用 tiktoken 而不是 len() 来估算 token 数。中文字符数和 token 数的误差可达 50%+。
  5. loader 和管道是正交的。切块策略改变时，embedding、存储、检索、生成的代码不需要动——这是好的模块边界。

接下来学什么

现在你的 RAG 系统能"吃"真实文件了。但你会立刻发现新问题：

  • 检索质量不稳定：有时候前 3 个结果里没有正确答案。怎么办？
  • 只有一个 embedding 模型：text-embedding-3-small 对中文好吗？有没有更好的选择？
  • 没有量化指标：你只能靠眼睛判断"检索结果好不好"。企业环境需要 Recall@K、MRR 等自动化指标。
