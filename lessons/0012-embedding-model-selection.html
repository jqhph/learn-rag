第一步：Embedding 模型在 RAG 中的位置

文档 Chunk → Embedding 模型 → 向量 [0.023, -0.15, …, 0.87] → 向量数据库

Embedding 模型的核心作用就是把文本转换成固定长度的向量，使得语义相似的文本在向量空间中距离相近。它在 RAG 中出现在两个环节：

  1. 索引阶段：将每一篇文档 chunk 通过 embedding 模型变成向量，存入向量数据库
  2. 检索阶段：将用户查询通过 同一个 embedding 模型变成向量，在向量库中做相似性搜索

⚠️ 关键约束：索引和检索必须使用 同一个 embedding 模型。如果用 A 模型索引、B 模型检索，向量空间的分布不同，检索质量会断崖式下降。

所以，embedding 模型选型有多重要？

一句话：embedding 模型是检索质量的天花板。

  • 如果你的 embedding 模型无法区分"苹果是水果"和"苹果是公司"，那么后续的所有优化（重排序、query 转换、prompt 优化）都是在弥补这个缺陷
  • 换了更好的 embedding 模型，Recall@K 可能直接提升 10-20%，而且这个改进与后续 pipeline 无关（乘法效应）
  • embedding 模型的维度决定了向量库的存储量和检索速度，影响整个系统的成本

一个真实的案例

某团队在中文客服知识库上用 OpenAI text-embedding-ada-002（1536 维）换到 bge-large-zh（1024 维），Recall@10 从 0.72 提升到 0.85，同时维度缩小了 1/3，检索速度反而更快了。换模型花了 2 天，但效果持续到今天。

第二步：MTEB 榜单——什么值得看、什么不能信

MTEB（Massive Text Embedding Benchmark） 是当前最权威的 embedding 模型排行榜，覆盖 100+ 数据集、8 大任务。但很多人在读榜单时犯了错误。

常见误区：只看总分不看 Retrieval Avg

MTEB 的标准排序是所有 8 项任务的平均分。但很多模型在 Classification 上得分高、在 Retrieval 上表现平平——而你做 RAG 只需要 Retrieval。比如有的模型总分 64+ 但 Retrieval Avg 只有 50 出头，这在 RAG 场景下就是低质量模型。

正确做法：看 MTEB 榜单时，找到 Retrieval Average 列，按这个排序。这是对 RAG 最直接的参考指标。

中文场景怎么看？

MTEB 的 retrieval 任务主要是英文数据集（NFCorpus、SciFact 等）。对中文场景，有两个调整：

  1. C-MTEB（Chinese MTEB）：包含专门的中文检索数据集（T2Retrieval、MMarcoRetrieval 等），更能反映模型的中文能力
  2. 自己建测试集才是最终答案：MTEB 再怎么好，也不如你在自己的业务数据上跑一次 Recall@K 对比。本课最后的动手实践就是做这个。

快速判定指南

当你要选一个 embedding 模型时，按这个顺序查看：

  1. Retrieval Avg 在不在前 20？（MTEB 上目前前排模型 Retrieval Avg 在 55-65 之间）
  2. 中文场景？→ 看 C-MTEB 的 Retrieval 子项分数
  3. 维度 vs 效果权衡？→ 如果两个模型 Retrieval Avg 差不到 1 分，选维度小的那个（速度更快、存储更省）
  4. 成本敏感？→ 开源模型（BGE/m3e）通常免费但需要 GPU；闭源 API 按 token 收费但零运维

模型详解

OpenAI：行业标杆，但封闭

  • text-embedding-3-small：性价比之王。512 维就能达到接近前代 ada-002 的效果，建议作为 RAG 项目的默认选择
  • text-embedding-3-large：支持降维（256/512/1024/3072），3072 维时效果最佳，但成本高 6.5 倍
  • 核心优势：零部署、效果稳定、多语言（含中文）表现优秀
  • 核心劣势：数据出境、无法私有化、成本随用量线性增长

BGE (BAAI)：中文场景第一选择

  • BGE-M3：支持多语言、多粒度（词/句/文档）、多种检索方式（Dense/Sparse/Multi-Vector）的单一模型，目前中文+英文最佳开源选择
  • bge-large-zh-v1.5：纯中文场景经典选择，1024 维，效果对标 ada-002
  • bge-multilingual-gemma2：基于 Gemma 2 的 2.7B 参数大模型，效果接近商业 API
  • 核心优势：中文效果接近 OpenAI、完全开源、支持私有化部署
  • 核心劣势：需要 GPU 推理（小模型可 CPU，但慢）；部署运维成本

Cohere：英文检索之王

  • embed-english-v3.0：英文 Retrieval 效果经常排名前三
  • embed-multilingual-v3.0：支持 100+ 语言，但中文效果不及 BGE
  • 核心优势：英文检索质量极佳；支持压缩（压缩率 32 倍）减少存储
  • 核心劣势：中文效果一般；纯 API 无私有化；最大上下文仅 512 token

Jina AI：Long Context 专精

  • jina-embeddings-v3：支持 8192 token 上下文，适合长文档场景
  • Jina-CLIP v2：支持图文多模态 embedding
  • 核心优势：长上下文；开源可自部署；支持多模态
  • 核心劣势：中文效果不如 BGE；社区规模较小

一个容易犯的错误：instruction 格式

BGE 和 GTE 这类模型对输入格式有特别要求：

  • 查询（query）需要加特定前缀，如 BGE 用 "为这段文字生成表示以用于检索："
  • 文档（document）不需要前缀，或者用不同的前缀

如果忽略这一点，检索效果可能下降 10-20%。不同版本的 BGE 要求不同，请务必查看模型的 HuggingFace 文档。

# BGE-zh-v1.5 的正确用法
query_emb = model.encode("为这段文字生成表示以用于检索：" + query)
doc_emb  = model.encode(doc_text)  # 文档不需要前缀

第五步：构建 EmbeddingAdapter——可热切换的模型适配层

前面的课程里，我们用 OpenAI 的 embedding 接口是直接硬编码的：

# 旧代码：硬编码，换模型要改多处
response = openai_client.embeddings.create(
    model="text-embedding-3-small",
    input=text,
)
emb = response.data[0].embedding

如果要切换到 BGE 或者其他模型，就得改多处代码，而且不同的模型有不同的参数、不同的调用方式。这显然不是生产级的做法。

下面我们设计一个统一的 EmbeddingAdapter 抽象层：

应用代码 → EmbeddingAdapter.encode() → OpenAI
(API) 或 HuggingFace
(本地模型) 或 Ollama
(本地服务) 
"""embedding_adapter.py — 统一 Embedding 模型适配层"""
import numpy as np
from abc import ABC, abstractmethod
from typing import list


class EmbeddingAdapter(ABC):
    """抽象基类：所有 embedding 模型的统一接口。"""

    @abstractmethod
    def encode(self, texts: list[str]) -> np.ndarray:
        """
        将文本列表编码为向量矩阵。
        返回: shape (len(texts), dim) 的 numpy 数组
        """
        ...

    @abstractmethod
    def dim(self) -> int:
        """返回向量的维度"""
        ...

    @property
    @abstractmethod
    def name(self) -> str:
        """模型名称标识"""
        ...


# ─── OpenAI 实现 ───────────────────────────────────────────
class OpenAIEmbedding(EmbeddingAdapter):
    def __init__(self, model: str = "text-embedding-3-small", dimensions: int = 512):
        from openai import OpenAI
        self.client = OpenAI()
        self.model = model
        self._dim = dimensions

    def encode(self, texts: list[str]) -> np.ndarray:
        # OpenAI 支持批量，但为了简化按单条处理
        # 生产环境建议用 batch 减少 API 调用次数
        response = self.client.embeddings.create(
            model=self.model,
            input=texts,
            dimensions=self._dim,
        )
        # 按输入顺序排列
        sorted_data = sorted(response.data, key=lambda x: x.index)
        vectors = [item.embedding for item in sorted_data]
        return np.array(vectors, dtype=np.float32)

    def dim(self) -> int:
        return self._dim

    @property
    def name(self) -> str:
        return f"openai:{self.model}"


# ─── HuggingFace 本地模型实现 ──────────────────────────────
class HuggingFaceEmbedding(EmbeddingAdapter):
    def __init__(
        self,
        model_name: str = "BAAI/bge-large-zh-v1.5",
        device: str | None = None,
        query_instruction: str = "为这段文字生成表示以用于检索：",
    ):
        from sentence_transformers import SentenceTransformer
        self.model = SentenceTransformer(model_name, device=device)
        self._dim = self.model.get_sentence_embedding_dimension()
        self.query_instruction = query_instruction

    def encode(self, texts: list[str]) -> np.ndarray:
        embeddings = self.model.encode(
            texts,
            normalize_embeddings=True,
            show_progress_bar=False,
        )
        return np.array(embeddings, dtype=np.float32)

    def encode_queries(self, texts: list[str]) -> np.ndarray:
        """给查询加指令前缀后再编码（BGE 系列必用）。"""
        prefixed = [self.query_instruction + t for t in texts]
        return self.encode(prefixed)

    def dim(self) -> int:
        return self._dim

    @property
    def name(self) -> str:
        return f"hf:{self.model_name}"


# ─── Ollama 本地服务实现 ────────────────────────────────────
class OllamaEmbedding(EmbeddingAdapter):
    def __init__(self, model: str = "bge-m3", base_url: str = "http://localhost:11434"):
        import requests
        self.model = model
        self.base_url = base_url.rstrip("/")
        self._session = requests.Session()
        # 获取维度（Ollama 支持的模型在 pulling 后会返回）
        self._dim = 1024  # BGE-M3 默认 1024，生产环境可动态获取

    def encode(self, texts: list[str]) -> np.ndarray:
        vectors = []
        for text in texts:
            resp = self._session.post(
                f"{self.base_url}/api/embeddings",
                json={"model": self.model, "prompt": text},
            )
            resp.raise_for_status()
            vectors.append(resp.json()["embedding"])
        return np.array(vectors, dtype=np.float32)

    def dim(self) -> int:
        return self._dim

    @property
    def name(self) -> str:
        return f"ollama:{self.model}"


# ─── 配置化工厂 ────────────────────────────────────────────
def create_embedding(config: dict) -> EmbeddingAdapter:
    """
    从配置字典创建 embedding 适配器。

    配置示例:
    config = {
        "provider": "openai",
        "model": "text-embedding-3-small",
        "dimensions": 512,
    }
    config = {
        "provider": "huggingface",
        "model_name": "BAAI/bge-large-zh-v1.5",
        "device": "cpu",
    }
    config = {
        "provider": "ollama",
        "model": "bge-m3",
        "base_url": "http://localhost:11434",
    }
    """
    provider = config["provider"]
    if provider == "openai":
        return OpenAIEmbedding(
            model=config.get("model", "text-embedding-3-small"),
            dimensions=config.get("dimensions", 512),
        )
    elif provider == "huggingface":
        return HuggingFaceEmbedding(
            model_name=config.get("model_name", "BAAI/bge-large-zh-v1.5"),
            device=config.get("device"),
            query_instruction=config.get("query_instruction",
                                          "为这段文字生成表示以用于检索："),
        )
    elif provider == "ollama":
        return OllamaEmbedding(
            model=config.get("model", "bge-m3"),
            base_url=config.get("base_url", "http://localhost:11434"),
        )
    else:
        raise ValueError(f"不支持的 provider: {provider}")


# ─── 快速测试 ──────────────────────────────────────────────
if __name__ == "__main__":
    # 测试切换不同的 embedding 模型
    configs = [
        {"provider": "openai", "model": "text-embedding-3-small", "dimensions": 512},
        # 如果有本地模型，取消注释以下配置：
        # {"provider": "huggingface", "model_name": "BAAI/bge-large-zh-v1.5", "device": "cpu"},
        # {"provider": "ollama", "model": "bge-m3", "base_url": "http://localhost:11434"},
    ]

    test_texts = ["退货政策是什么", "订单发货时间", "如何申请发票"]

    for cfg in configs:
        adapter = create_embedding(cfg)
        emb = adapter.encode(test_texts)
        print(f"[{adapter.name}] dim={adapter.dim()}, shape={emb.shape}")
        # 检查余弦相似度（语义相近的应该分高）
        cos = np.dot(emb[0], emb[1]) / (
            np.linalg.norm(emb[0]) * np.linalg.norm(emb[1])
        )
        print(f"  退货政策 vs 订单发货 相似度: {cos:.4f}")

适配层带来的好处

  • 一行配置切换模型：改 config 字典即可，不需要改业务代码
  • 可以自由扩展新模型：写一个新类继承 EmbeddingAdapter，在 create_embedding 加一个 case 就行
  • 便于 A/B 测试：在同一个评估流程里，只需替换 adapter 实例即可对比不同模型
  • 向量标准化统一处理：HuggingFace 实现里 normalize_embeddings=True，确保余弦相似度计算正确

第六步：本地部署私有化 Embedding——Ollama 和 Infinity

对于数据敏感的场景（金融、医疗、军工），或者希望长期节省 API 费用，本地部署是唯一选择。有两种主流方式：

方案一：Ollama（最简单，推荐入门）

Ollama 是一个本地大模型运行工具，支持一键下载和运行 embedding 模型。

curl -fsSL https://ollama.com/install.sh | sh

# 2. 拉取 embedding 模型（以 bge-m3 为例）
ollama pull bge-m3

# 3. 测试 embedding
curl http://localhost:11434/api/embeddings \
  -d '{"model": "bge-m3", "prompt": "退货政策是什么"}'

Ollama 还能做什么？

第七步：Embedding 维度对检索速度与质量的影响

很多人以为"维度越大，效果越好"——这不一定对。维度、质量和速度之间存在复杂的权衡。

维度越大 = 检索越慢（线性关系）

向量相似性搜索的时间复杂度是 O(N × D)，其中 N 是向量数量，D 是维度。维度翻倍，检索时间也翻倍。更重要的是，HNSW/IVF 这类索引的构建时间和内存消耗也随 D 线性增长。

维度越大 ≠ 效果越好（非线性关系）

实验数据表明：

  • 从 256 维 → 512 维 → 1024 维：效果显著提升（信息容量增加）
  • 从 1024 维 → 3072 维：提升微乎其微（边际效益递减）
  • 更极端的：在某些数据集上，1024 维和 3072 维的 Retrieval 效果几乎一样，但速度慢了 3 倍

实用建议

向量维度适用场景
256-512移动端、边缘设备、极端低延迟场景；使用 text-embedding-3-small (512d)
768-1024通用场景的最佳平衡点；大部分开源模型的默认维度，推荐选择
1536-3072对精度要求极高且不介意成本和速度；OpenAI large 的 3072 或 GTE-Qwen2 的 1536

正则化与余弦相似度

绝大多数现代 embedding 模型输出的向量都做了 L2 正则化（长度 = 1）。被正则化后，余弦相似度和点积是等价的：cos(a, b) = a · b（因为 |a| = |b| = 1）。

这意味着你可以在向量数据库里存正则化后的向量，用点积（dot product）代替余弦相似度来计算相似度——点积在某些向量库中比余弦相似度快。

第八步：A/B 对比实验——让数据替你选择

最终的选择不能靠"我感觉"——要用数据说话。我们将基于第 4 课的评估框架，用同一个 golden test set，对比不同 embedding 模型的 Recall@K 和 MRR。

"""embedding_ab_test.py — Embedding 模型 A/B 对比实验"""
import json
import numpy as np
from typing import Callable

# 引入第 4 课的评估器
# 假设你已经把第 4 课的 evaluator.py 保留在项目中
# from evaluator import RetrieverEvaluator, RecallAtK, MRR

from embedding_adapter import create_embedding, EmbeddingAdapter


def build_simple_retriever(
    adapter: EmbeddingAdapter,
    documents: list[dict],
) -> Callable:
    """
    用给定 embedding adapter 构建一个简单的向量检索器。

    返回一个函数 retriever(query, top_k) → list[dict]
    """
    # 为所有文档预计算向量
    texts = [doc["text"] for doc in documents]
    doc_embeddings = adapter.encode(texts)

    def retriever(query: str, top_k: int = 10) -> list[dict]:
        query_emb = adapter.encode([query])
        # 余弦相似度（向量已正则化 → dot product）
        scores = np.dot(doc_embeddings, query_emb.T).squeeze()
        top_indices = np.argsort(scores)[::-1][:top_k]

        results = []
        for idx in top_indices:
            results.append({
                "id": documents[idx]["id"],
                "text": documents[idx]["text"],
                "score": float(scores[idx]),
                "retriever": adapter.name,
            })
        return results

    return retriever


def run_ab_test(
    configs: list[dict],
    documents: list[dict],
    golden_set: list[dict],
) -> dict:
    """
    对比多组 embedding 配置的检索效果。

    configs: [{"provider":"openai", "model":"text-embedding-3-small", "dimensions":512}, ...]
    documents: [{id, text}, ...]  # 知识库文档列表
    golden_set: [{"query":..., "relevant_ids":[...]}, ...]  # golden test set

    返回: {config_label: {"recall@5":..., "recall@10":..., "mrr":...}, ...}
    """
    results = {}

    for cfg in configs:
        print(f"\n{'='*50}")
        print(f"测试配置: {cfg}")
        print("=" * 50)

        adapter = create_embedding(cfg)
        retriever = build_simple_retriever(adapter, documents)

        # 使用第 4 课的评估器（这里直接计算，没有 import 依赖）
        recall_at_5 = 0
        recall_at_10 = 0
        mrr = 0

        for item in golden_set:
            query = item["query"]
            relevant = set(item["relevant_ids"])
            retrieved = retriever(query, top_k=10)
            retrieved_ids = [r["id"] for r in retrieved]

            # Recall@K
            hits_5 = sum(1 for rid in retrieved_ids[:5] if rid in relevant)
            hits_10 = sum(1 for rid in retrieved_ids if rid in relevant)
            recall_at_5 += hits_5 / max(len(relevant), 1)
            recall_at_10 += hits_10 / max(len(relevant), 1)

            # MRR
            for rank, rid in enumerate(retrieved_ids, 1):
                if rid in relevant:
                    mrr += 1.0 / rank
                    break

        n = len(golden_set)
        results[adapter.name] = {
            "recall@5": round(recall_at_5 / n, 4),
            "recall@10": round(recall_at_10 / n, 4),
            "mrr": round(mrr / n, 4),
            "dim": adapter.dim(),
        }

        print(f"  Recall@5:  {results[adapter.name]['recall@5']}")
        print(f"  Recall@10: {results[adapter.name]['recall@10']}")
        print(f"  MRR:       {results[adapter.name]['mrr']}")
        print(f"  维度:      {results[adapter.name]['dim']}")

    return results

怎样做一次严肃的 A/B 测试？

  1. Golden set 要覆盖多种查询类型：包含语义匹配（退货政策）、精确匹配（SKU号）、单文档查询（包邮条件）、多文档查询（投诉渠道+时效）
  2. 不要用你自己的查询：请同事或朋友写测试查询——你设计的查询可能无意中偏向你熟悉的模型
  3. 控制变量：只换 embedding 模型，所有其他参数（chunk_size、top_K、重排序）保持不变
  4. 重复 3 次取平均：某些检索有随机性（HNSW），重复多次避免单次偏差
  5. 记录时间：除了 Recall/MRR，还要记录索引构建时间和检索延迟，因为"够好但快 2 倍"可能是更好的选择

最后提醒：embedding 不是越多越好

很多团队犯的一个错误是：在同一个向量库里混用多个 embedding 模型生成的向量。比如文档用 BGE 索引、查询用 OpenAI 生成向量。向量空间是不同的，这样的检索效果会极差。

正确的做法：整个 RAG 系统在某一时刻只使用一个 embedding 模型。切换模型时，需要重新索引所有文档。

常见错误与排查

场景一：换了 embedding 模型后，检索效果反而变差了

可能的原因：

  1. 没有重新索引：新模型生成的向量和老模型的向量不在同一个向量空间，必须重新索引全部文档
  2. 忘记加 instruction 前缀：BGE 等模型需要给查询加特定前缀，不加的话效果会下降
  3. 向量未正则化：余弦相似度对未正则化的向量计算不正确。检查 normalize_embeddings 是否设为 True

排查步骤：

# 1. 检查向量是否已正则化
emb = adapter.encode(["测试文本"])
norm = np.linalg.norm(emb[0])
print(f"向量长度: {norm:.6f}")  # 应该在 0.9999-1.0001 之间

# 2. 检查原始 embedding 质量（不通过检索，直接看语义相似度）
texts = ["退货政策是什么", "如何退货", "今天天气很好"]
emb = adapter.encode(texts)
# 期待："退货政策是什么" vs "如何退货" 相似度 > 0.7
#       "退货政策是什么" vs "今天天气很好" 相似度 < 0.5
print(np.dot(emb[0], emb[1]))  # 语义相近 → 分高
print(np.dot(emb[0], emb[2]))  # 语义无关 → 分低

场景二：Ollama embedding 服务返回错误

症状：requests.exceptions.ConnectionError: HTTPConnectionPool

可能的原因和解决：

  1. Ollama 服务未启动：运行 ollama serve 启动服务
  2. 模型未拉取：检查 ollama list 是否有你需要的模型；用 ollama pull bge-m3 拉取
  3. 端口不对：Ollama 默认端口是 11434，确认 base_url 是否指向正确的端口

场景三：HuggingFace 模型加载 OOM（内存不足）

症状：CUDA out of memory 或进程被 kill

解决方案：

  1. 换小模型：BGE-large（335M）→ BGE-base（110M）→ BGE-small（24M）
  2. 明确指定 device="cpu"：
    model = SentenceTransformer("BAAI/bge-large-zh-v1.5", device="cpu")
  3. 用 Infinity 的服务化部署代替进程内加载，避免每个进程都加载一次模型
