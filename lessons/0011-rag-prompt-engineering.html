场景导入：同一套检索结果，不同的回答

先看一个对比。检索系统返回了完全相同的内容：

检索结果（3 条片段）：

[来源1] 华为 Mate60 Pro 采用 6.82 英寸 OLED 曲面屏，分辨率为 2720×1260。
[来源2] 华为 Mate60 Pro 搭载 5000 万像素超感知摄像头，支持 F1.4-F4.0 可变光圈。
[来源3] 华为 Mate60 Pro 配备 5000mAh 电池，支持 88W 有线快充和 50W 无线快充。

现在看两种 prompt 的结果差异：

❌ 初级 Prompt："请基于以下内容回答"
你是一个客服助手。请基于以下内容回答问题：
{context}
问题：华为Mate60 Pro有哪些特点？
回答：

👉 生成结果："华为Mate60 Pro是一款高端智能手机，拥有出色的屏幕、强大的拍照功能和快速充电能力。具体参数请参考官网。"

问题分析：没有要求引用来源——模型用自己的知识"脑补"了原话，而且没有引用具体数据。Faithfulness 得分可能只有 0.5-0.6。

✅ 专业 RAG Prompt："逐条对应 + 引用来源"
你是一个企业知识库客服助手。你的回答必须遵守以下规则：

1. 只使用下方"检索结果"中的信息，不要使用自己的知识
2. 每一条信息都要标注来源编号 [来源N]
3. 如果结果中不包含答案，直接说"知识库中暂无相关信息"
4. 用清晰的分点结构回答

检索结果：
{context}

问题：华为Mate60 Pro有哪些特点？
回答：

👉 生成结果："华为Mate60 Pro的主要特点包括： - 📱 屏幕：6.82 英寸 OLED 曲面屏，分辨率 2720×1260 [来源1] - 📷 摄像头：5000 万像素超感知摄像头，支持 F1.4-F4.0 可变光圈 [来源2] - 🔋 充电与续航：5000mAh 电池，支持 88W 有线快充和 50W 无线快充 [来源3]"

优势：每条信息都标注了来源、结构清晰、没有使用外部知识。Faithfulness ≈ 1.0。

同样的检索结果，同样的 LLM，只是 prompt 不同——回答质量天差地别。 这就是 Prompt Engineering 在 RAG 中的价值：零成本、高杠杆、立竿见影。

第一部分：System Prompt 的"四块积木"设计骨架

在 RAG 场景中，system prompt 承担着设定行为基线的作用——它告诉 LLM"你是谁、你能做什么、你不能做什么、怎么做"。一个好的 RAG system prompt 由四块积木组成：

① 角色设定
你是谁
→
② 知识范围
你知道什么
→
③ 拒答指令
什么不能做
→
④ 输出格式
怎么回答

1.1 四块积木详解

积木核心问题示例常见错误
① 角色设定 LLM 该扮演什么角色、用何种语气？ "你是一个企业知识库客服助手，专业、准确、友好。" 不设定角色，或设定不切实际的角色（"你是无所不能的 AI"——这反而鼓励幻觉）
② 知识范围 LLM 应该/只能用哪些知识来源？ "你的知识来源仅限于下方提供的检索结果。不要使用训练数据中的知识。" 不说清楚知识边界，LLM 会混用训练数据和检索结果
③ 拒答指令 知识不足时该怎么做？ "如果检索结果不包含答案，请直接说'知识库中暂无相关信息'，不要编造。" 没有拒答指令——LLM 宁愿编造也不愿说"不知道"
④ 输出格式 回答应该长什么样？ "使用分点结构，每条信息标注[来源N]。如果涉及数字，准确引用。" 不指定格式，LLM 自由发挥导致风格不一致

# ========== Prompt 装配器 ==========

@dataclass
class SystemPromptConfig:
    """System Prompt 四块积木的配置"""
    role_key: str = "客服助手"
    role_variables: dict = field(default_factory=dict)
    knowledge_key: str = "严格检索"
    refusal_key: str = "简洁拒答"
    output_key: str = "引用标注"
    additional_instructions: list[str] = field(default_factory=list)


class PromptDesigner:
    """
    System Prompt 积木式构建器。

    使用方式：
        designer = PromptDesigner()
        prompt = designer.build(SystemPromptConfig(
            role_key="客服助手",
            role_variables={"company_name": "华为"},
            knowledge_key="严格检索",
            refusal_key="透明拒答",
            output_key="引用标注",
        ))
    """

    def __init__(self):
        self.roles = ROLE_TEMPLATES
        self.knowledge_scopes = KNOWLEDGE_SCOPE_TEMPLATES
        self.refusals = REFUSAL_TEMPLATES
        self.outputs = OUTPUT_FORMAT_TEMPLATES

    def build(self, config: SystemPromptConfig) -> str:
        """将四块积木组装为完整的 system prompt"""
        parts = []

        # 1. 角色
        role_info = self.roles.get(config.role_key, self.roles["客服助手"])
        role_text = role_info["template"].format(**config.role_variables)
        parts.append(role_text)

        # 2. 知识范围
        ks_info = self.knowledge_scopes.get(
            config.knowledge_key, self.knowledge_scopes["严格检索"]
        )
        parts.append(ks_info["template"])

        # 3. 拒答指令
        ref_info = self.refusals.get(config.refusal_key, self.refusals["简洁拒答"])
        parts.append(ref_info["template"])

        # 4. 输出格式
        out_info = self.outputs.get(config.output_key, self.outputs["引用标注"])
        parts.append(out_info["template"])

        # 5. 附加指令
        if config.additional_instructions:
            parts.append("\n".join(config.additional_instructions))

        return "\n\n".join(parts)

    def list_presets(self, category: str) -> list[dict]:
        """列举指定类别的所有预设"""
        source = {
            "role": self.roles,
            "knowledge": self.knowledge_scopes,
            "refusal": self.refusals,
            "output": self.outputs,
        }.get(category, {})
        return [
            {"key": k, "desc": v["description"]}
            for k, v in source.items()
        ]

为什么"不设角色"是最常见的大坑？

没有角色设定的 system prompt，相当于对 LLM 说"随便答"——它会默认使用训练数据中的所有知识，而不是你给它的检索结果。设定角色 = 设定行为边界。"客服助手"这个角色天然包含了"不能乱说、不能编造"的行为约束，比没有角色的 prompt 幻觉率低 20-40%。

积木式设计的核心优势

把 system prompt 拆成四块积木，你获得了一个重要的工程能力——A/B 测试每块积木。例如：

  • 固定角色 + 知识范围 + 输出格式，只更换"拒答指令"类型，看哪种拒答方式用户满意度最高
  • 固定其他三块，只切换"输出格式"——对比"引用标注"vs"表格输出"的阅读理解效率

这种模块化的 prompt 管理方式，在 L7 的 RAGAS 评估框架下，可以系统性地找到最优组合。

第二部分：Grounding Prompt——让回答"落地"的引用系统

上一部分解决了"谁来答、怎么答"的骨架问题，现在解决最核心的痛点：怎么让 LLM 的回答每句话都对应到检索结果。

Grounding（接地）是把检索结果"锚定"到 LLM 回答的过程——它是 RAG 中对抗幻觉的最强武器。它的核心思想是：

检索结果
N 条文本片段
→
Grounding
映射
→
结构化回答
逐句标注来源

2.1 Grounding Prompt 的经典模板

"""
grounding_prompts.py — RAG Grounding Prompt 模板库

包含三种不同严格程度的 grounding 模板。
"""

from typing import Optional


def build_grounding_prompt(
    context: str,
    query: str,
    mode: str = "standard",
    language: str = "zh",
) -> list[dict]:
    """
    构建 RAG 生成的 grounding prompt。

    参数：
        context: 检索结果文本（已格式化的来源列表）
        query: 用户问题
        mode: "standard" | "strict" | "lenient"
        language: "zh" | "en"

    返回：
        [system_message, user_message] 格式的 messages 列表
    """

    # ========== System Prompt ==========
    system_prompts = {
        "strict": (
            "你是一个严格遵循检索结果回答问题的知识库助手。\n\n"
            "规则（请严格遵守）：\n"
            "1. 【只使用】下面检索结果中的内容，绝对不要使用你训练数据中的知识\n"
            "2. 【逐句标注】回答中的每一条信息都需要标注对应的[来源N]\n"
            "3. 【不改写数值】不要修改检索结果中的数字、名称、日期——原样引用\n"
            "4. 【不推理】不要做"根据我的理解""推测来说"这类推理——只转述检索结果\n"
            "5. 【诚实拒答】如果检索结果完全没有相关信息，回答："
            ""抱歉，知识库中暂无相关信息。请尝试换个关键词提问或联系人工客服。"\n"
            "6. 【部分回答】如果检索结果只能回答部分问题，先回答能答的部分，"
            "再说明哪些部分无法回答"
        ),
        "standard": (
            "你是一个企业知识库助手。你的回答基于下方提供的检索结果。\n\n"
            "要求：\n"
            "1. 每条信息标注来源 [来源N]\n"
            "2. 如果检索结果不充分，诚实告知\n"
            "3. 可以基于多个来源进行总结，但总结需忠实于原文\n"
            "4. 使用清晰的结构回答"
        ),
        "lenient": (
            "你是一个智能助手。请优先使用下方检索结果中的信息回答问题。\n\n"
            "指南：\n"
            "1. 尽量引用检索结果（标注来源编号）\n"
            "2. 如果信息不足，可以适当补充你的通用知识\n"
            "3. 补充的知识需明确标注（如"据我所知..."）\n"
            "4. 不要让用户混淆"来自知识库"和"来自模型知识"的信息"
        ),
    }

    # ========== 用户 Prompt：Grounding 模板 ==========
    user_templates = {
        "strict": (
            "检索结果：\n"
            "{context}\n\n"
            "---\n\n"
            "请基于以上检索结果，回答以下问题。"
            "记住：每条信息都要标注 [来源N]，不要添加检索结果中没有的内容。\n\n"
            "问题：{query}\n\n"
            "回答："
        ),
        "standard": (
            "以下是检索到的相关信息：\n\n"
            "{context}\n\n"
            "---\n\n"
            "问题：{query}\n\n"
            "请基于上述信息回答，标注信息来源于哪个来源编号。"
        ),
        "lenient": (
            "检索结果：\n"
            "{context}\n\n"
            "---\n\n"
            "问题：{query}\n\n"
            "回答："
        ),
    }

    system_prompt = system_prompts.get(mode, system_prompts["standard"])
    user_prompt = user_templates.get(mode, user_templates["standard"]).format(
        context=context, query=query
    )

    return [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt},
    ]

2.2 为什么 grounding 如此重要？——来自 L7 RAGAS 的测量

在 L7 中，你学到了 Faithfulness（忠实度）指标——衡量模型回答是否与检索结果一致。Grounding prompt 是影响 Faithfulness 的最直接因素：

Grounding 模式平均 Faithfulness引用完整度幻觉率推荐场景
无 grounding（"请回答"）0.55-0.65~20%30-50%— 不推荐任何生产场景
lenient0.70-0.80~50%15-25%创意辅助、头脑风暴
standard0.85-0.93~85%5-15%客服 Q&A、企业知识库（推荐）
strict0.92-0.98~95%<5%金融/医疗/法律等需要极高准确度的场景

实战建议：从 strict 开始，按需放松

很多人的直觉是"先宽松，出问题了再收紧"——但在 RAG 中请反过来：先用 strict 模式上线，收集用户反馈，确认系统稳定后再按需放松。因为：

  • strict 模式下，即使用户不满意，至少答案是准确的（不会产生错误信息带来后续麻烦）
  • 从 strict 切换到 standard 只需要改一个参数（mode），可以快速 A/B 测试
  • 如果先用了 lenient，你很难区分"用户不满"是因为模型胡说还是因为信息不足

第三部分：Few-shot 示例——给 LLM 一个"范本"

在 RAG prompt 中加入示例（few-shot）可以显著提升回答的格式一致性和引用规范度。但给多了浪费 token，给少了没效果——关键是怎么给。

3.1 什么时候该用 Few-shot？

场景Few-shot 效果推荐做法
标准 QA（一问一答）⭐⭐不需要：system prompt 已经够用
格式敏感（表格/对比/排序）⭐⭐⭐⭐⭐必须加：1-2 个完整示例
多步推理（比较/分析/推荐）⭐⭐⭐⭐推荐加：示例展示推理过程
情感/语气控制（投诉/敏感对话）⭐⭐⭐⭐推荐加：示例展示正确的语气
语言风格要求高⭐⭐⭐可以加：1 个示例即可

# ========== 示例选择器 ==========

class FewShotSelector:
    """
    动态 Few-shot 选择器。

    通过关键词匹配选择最相关的示例。
    生产环境可替换为 embedding 相似度匹配（参见 L4 的 Cross-Encoder）。
    """

    def __init__(self, examples: list[Example] | None = None):
        self.examples = examples or EXAMPLE_LIBRARY

    def select_by_keyword(
        self, query: str, top_k: int = 1,
    ) -> list[Example]:
        """
        基于关键词匹配度选择示例。

        简单的关键词打分策略：
        - 查询与示例的 query 共享的关键词越多，分数越高
        """
        query_tokens = set(self._tokenize(query))
        scored = []

        for ex in self.examples:
            ex_tokens = set(self._tokenize(ex.query))
            overlap = len(query_tokens & ex_tokens)
            scored.append((overlap, ex))

        scored.sort(key=lambda x: x[0], reverse=True)
        return [ex for _, ex in scored[:top_k]]

    def select_by_embedding(
        self, query: str, top_k: int = 1,
        embedding_fn: callable = None,
    ) -> list[Example]:
        """
        基于 embedding 语义相似度选择示例。
        需要传入 embedding 函数（如 OpenAI 的 text-embedding-3-small）。
        """
        if not embedding_fn:
            return self.select_by_keyword(query, top_k)

        query_vec = embedding_fn(query)
        scored = []

        for ex in self.examples:
            ex_vec = embedding_fn(ex.query)
            similarity = self._cosine_similarity(query_vec, ex_vec)
            scored.append((similarity, ex))

        scored.sort(key=lambda x: x[0], reverse=True)
        return [ex for _, ex in scored[:top_k]]

    def to_few_shot_prompt(
        self, examples: list[Example],
    ) -> str:
        """将示例列表转换为 prompt 中的 few-shot block"""
        parts = ["以下是几个回答示例，请参考其格式和风格：\n"]
        for i, ex in enumerate(examples, 1):
            parts.append(
                f"示例 {i}：\n"
                f"用户问题：{ex.query}\n"
                f"理想回答：{ex.ideal_response}\n"
            )
        return "\n".join(parts)

    @staticmethod
    def _tokenize(text: str) -> list[str]:
        """简单中文分词（按字和常见词切割）"""
        # 生产环境建议使用 jieba（参考 L5）
        import re
        # 提取中英文单词和数字
        tokens = re.findall(r'[一-鿿]+|[A-Za-z0-9]+', text)
        return [t.lower() for t in tokens]

    @staticmethod
    def _cosine_similarity(a: list[float], b: list[float]) -> float:
        import math
        dot = sum(ai * bi for ai, bi in zip(a, b))
        na = math.sqrt(sum(ai ** 2 for ai in a))
        nb = math.sqrt(sum(bi ** 2 for bi in b))
        return dot / (na * nb) if na and nb else 0.0

Few-shot 的 token 成本与收益平衡

每个示例平均消耗 150-300 token。假设你给 3 个示例：

  • 每天 10 万次对话 → 额外消耗 ~30-90 万 token → 约 $1-3/天（GPT-4o-mini）
  • 如果通过动态选择只给 1 个最相关的示例 → 成本降至 1/3

建议：先用 1 个固定示例测试格式改善效果。如果需要更多，优先用动态选择（按 embedding 相似度捞出最相关的 1-2 个）而不是固定给 3 个。

第四部分：Chain-of-Thought——让 LLM"先想后答"

在 RAG 场景中，CoT（思维链）的价值在于：让 LLM 在生成最终回答前，先展示它的推理过程——这样你（和用户）都能看到它是如何从检索结果推导出答案的，而幻觉也会在推理过程中自我暴露。

4.1 RAG 中三种 CoT 模式

模式做法适用场景Token 开销
CoT 直接推理 在 prompt 中加"请一步步思考" 多跳推理、对比分析 +50~100 token（输出更多）
CoT 结构化推理 要求 LLM 先列出"已知信息→需要推理→结论"的结构 需要可审计的严谨回答 +100~300 token
CoT 链式验证 回答前先检查每条"声称"是否有来源支持 高准确度要求场景 +150~400 token

4.2 结构化 CoT 在 RAG 中的代码实现

"""
rag_cot.py — 在 RAG prompt 中嵌入 Chain-of-Thought 推理

三种 CoT 模式的 prompt 模板，以及一个简单的调用示例。
"""


def build_cot_prompt(
    context: str,
    query: str,
    cot_mode: str = "direct",
) -> list[dict]:
    """
    构建包含 CoT 推理的 RAG prompt。

    参数：
        cot_mode: "direct" | "structured" | "verify"
    """

    system_base = (
        "你是一个企业知识库助手。你的回答必须基于以下检索结果。\n"
    )

    cot_instructions = {
        "direct": (
            "在回答前，请一步步思考：\n"
            "1. 问题中的关键信息是什么\n"
            "2. 检索结果中有哪些相关来源\n"
            "3. 这些来源是否足够回答问题\n"
            "4. 基于以上思考，给出最终答案\n\n"
            "格式：先给出推理过程（标注"思考过程："），再给出最终答案（标注"答案："）。"
        ),
        "structured": (
            "请按以下结构回答：\n\n"
            "【已知信息】列出检索结果中与问题相关的所有事实，每条标注来源\n\n"
            "【需要推理】说明哪些答案可以直接从已知信息得出，哪些需要组合多个来源\n\n"
            "【推理过程】如果有需要推理的地方，说明你的推理步骤\n\n"
            "【最终答案】给出简洁明确的最终回答\n\n"
            "【未回答部分】如果检索结果不足以回答全部问题，说明哪些部分无法回答"
        ),
        "verify": (
            "你的回答必须经过以下验证。请按顺序输出：\n\n"
            "步骤 1 - 事实提取：列出可以从检索结果中直接提取的事实（逐条标注来源）\n"
            "步骤 2 - 事实审查：检查每一条事实是否真的有对应的来源。"
            "如果某条"事实"没有来源支持，删除它\n"
            "步骤 3 - 回答构建：基于审查后的事实，构建最终答案\n\n"
            "注意：如果在步骤 2 中发现没有足够的事实支持回答，"
            "最终答案必须诚实地说明"检索结果不足以回答这个问题"。"
        ),
    }

    system_prompt = system_base + "\n" + cot_instructions.get(
        cot_mode, cot_instructions["direct"]
    )

    user_prompt = (
        "检索结果：\n"
        "{context}\n\n"
        "---\n\n"
        "问题：{query}\n"
    ).format(context=context, query=query)

    return [
        {"role": "system", "content": system_prompt},
        {"role": "user", "content": user_prompt},
    ]

CoT 的代价——延迟增加

CoT 会让模型"想更多"——这意味着输出 token 数增加，TTFT 和总延迟都会上升。实测数据：

  • 无 CoT：输出 ~80 token，延迟 ~1.2s
  • direct CoT：输出 ~250 token（+推理过程），延迟 ~2.0s
  • structured CoT：输出 ~400 token，延迟 ~2.8s
  • verify CoT：输出 ~500 token，延迟 ~3.2s

建议：只在需要多步推理或高准确度的场景使用 CoT。简单的 FAQ 问答用 standard grounding 就够了。你也可以让 CoT 内容不出现在最终回答中（只用内部推理过程），但需要模型支持这种"隐藏思维"能力。

第五部分：生成参数调优——找到 RAG 场景的"甜蜜点"

同样的 prompt，不同的 temperature / top_p / max_tokens，输出质量可以相差 30% 以上。调参是 Prompt Engineering 的最后一公里。

5.1 三个核心参数在 RAG 中的影响

参数作用值范围对 RAG 的影响
temperature 控制输出的随机性/创造性 0.0 - 2.0 越高越"发散"：温度高时 LLM 更可能跳出检索结果的限制，产生幻觉
越低越"忠实"：温度接近 0 时 LLM 倾向于选择最可能的输出，更适合 RAG
top_p 控制 token 采样的候选集大小 0.0 - 1.0 越低越"保守"：只从高概率 token 中选择
越高越"多样"：候选集更大，输出更丰富但更不可控
通常与 temperature 协同使用，但建议固定一个调另一个
max_tokens 限制回答的最大长度 视场景而定 设置太短 → 回答被截断，信息不完整
设置太长 → token 浪费，延迟增加
RAG 场景通常 512-2048 之间

5.2 RAG 场景最佳参数参考表

RAG 场景temperaturetop_pmax_tokens说明
客服 Q&A（标准） 0.0 - 0.1 0.9 - 1.0 512 - 1024 需要忠实、准确，不要创造力
知识库摘要/总结 0.3 - 0.5 0.9 1024 - 2048 适量创造力让总结更流畅
产品对比 0.0 - 0.1 0.9 1024 - 2048 数据必须准确，创造力是敌人
多轮对话（带历史） 0.3 - 0.5 0.9 1024 稍微提高温度让对话更自然
金融/医疗回答 0.0 1.0 512 - 1024 零容忍幻觉，temperature 必须为 0
多跳推理（CoT） 0.1 - 0.3 0.9 1536 - 3072 需要推理路径，适当长度
创意文案生成 0.7 - 1.0 0.95 2048 需要创造力，不与 RAG 结合

5.3 参数调优实验代码

"""
parameter_tuning.py — RAG 生成参数调优实验框架

在 L7 的 RAGAS 评估框架下，跑不同参数组合的对比实验。
"""

from dataclasses import dataclass, field
from typing import Optional, Callable
import json
import time


@dataclass
class GenParams:
    """生成参数配置"""
    temperature: float = 0.1
    top_p: float = 0.9
    max_tokens: int = 1024
    presence_penalty: float = 0.0
    frequency_penalty: float = 0.0

    def label(self) -> str:
        """返回参数的简短标签"""
        return f"T{self.temperature}_P{self.top_p}_M{self.max_tokens}"

    def to_dict(self) -> dict:
        return {
            "temperature": self.temperature,
            "top_p": self.top_p,
            "max_tokens": self.max_tokens,
            "presence_penalty": self.presence_penalty,
            "frequency_penalty": self.frequency_penalty,
        }


# ========== 参数搜索空间 ==========

def grid_search_params() -> list[GenParams]:
    """
    生成参数网格搜索组合。

    从这个默认网格开始，你可以根据场景调整搜索范围。
    注意：每次实验至少 20 个测试样本，结果才有统计意义。
    """
    params = []

    # temperature 网格：关注低值区域（RAG 场景主要范围）
    for temp in [0.0, 0.05, 0.1, 0.3, 0.5, 0.7]:
        params.append(GenParams(temperature=temp, top_p=0.9, max_tokens=1024))

    # top_p 网格：固定 temperature=0.1
    for top_p in [0.8, 0.9, 0.95, 1.0]:
        params.append(GenParams(temperature=0.1, top_p=top_p, max_tokens=1024))

    # max_tokens 网格
    for mt in [256, 512, 1024, 2048]:
        params.append(GenParams(temperature=0.1, top_p=0.9, max_tokens=mt))

    return params

    answer_relevancy: float  # L7 的 Answer Relevancy（0~1）
    avg_latency_ms: float  # 平均延迟
    avg_output_tokens: float  # 平均输出 token 数
    sample_count: int  # 测试样本数
    examples: list[dict] = field(default_factory=list)  # 几个示例输出


class ParamTuningExperiment:
    """
    生成参数调优实验。

    使用方式：
        experiment = ParamTuningExperiment(
            llm_call_fn=my_chat_function,
            eval_fn=my_ragas_evaluator,
        )
        results = experiment.run(
            samples=test_samples,
            param_list=grid_search_params()[:5],  # 先跑 5 个组合
        )

        # 找出最优参数
        best = results[0]  # 按 faithfulness 降序排列
        print(f"最优参数：{best.params}")
    """

    def __init__(
        self,
        llm_call_fn: Callable,  # llm_call_fn(messages, params) -> str
        eval_fn: Callable,      # eval_fn(query, answer, context) -> dict
    ):
        self.llm_call_fn = llm_call_fn
        self.eval_fn = eval_fn

    def run(
        self,
        samples: list[EvalSample],
        param_list: list[GenParams],
    ) -> list[ParamExperimentResult]:
        """运行参数调优实验，结果按 faithfulness 降序排列"""
        results = []

        for params in param_list:
            print(f"\n实验参数：{params.label()}")

            faithfulness_scores = []
            relevancy_scores = []
            latencies = []
            output_lengths = []
            example_outputs = []

            for i, sample in enumerate(samples):
                # 构建 prompt
                messages = [
                    {"role": "system", "content":
                     "你是一个知识库助手。只使用检索结果回答问题，逐条标注来源。"},
                    {"role": "user", "content":
                     f"检索结果：\n{sample.context}\n\n问题：{sample.query}\n回答："},
                ]

                # 调用 LLM
                start = time.time()
                answer = self.llm_call_fn(messages, params)
                latency = (time.time() - start) * 1000

                # 评估（使用 L7 的 RAGAS 指标）
                eval_result = self.eval_fn(
                    query=sample.query,
                    answer=answer,
                    context=sample.context,
                )

                faithfulness_scores.append(eval_result.get("faithfulness", 0))
                relevancy_scores.append(eval_result.get("answer_relevancy", 0))
                latencies.append(latency)
                output_lengths.append(len(answer) // 2)  # token 估算

                # 只记录前 3 个样本的完整输出
                if i < 3:
                    example_outputs.append({
                        "query": sample.query,
                        "answer": answer[:200],  # 截断
                    })

            # 汇总
            avg_faith = sum(faithfulness_scores) / len(faithfulness_scores)
            avg_rel = sum(relevancy_scores) / len(relevancy_scores)

            results.append(ParamExperimentResult(
                params=params,
                faithfulness=avg_faith,
                answer_relevancy=avg_rel,
                avg_latency_ms=sum(latencies) / len(latencies),
                avg_output_tokens=sum(output_lengths) / len(output_lengths),
                sample_count=len(samples),
                examples=example_outputs,
            ))

            print(f"  Faithfulness: {avg_faith:.3f} | "
                  f"Relevancy: {avg_rel:.3f} | "
                  f"Latency: {results[-1].avg_latency_ms:.0f}ms | "
                  f"Output: {results[-1].avg_output_tokens} tokens")

        # 按 faithfulness 降序排列
        results.sort(key=lambda r: r.faithfulness, reverse=True)

        return results

def adapt_system_prompt(
    prompt: str,
    model_provider: str,
    model_name: str | None = None,
) -> str:
    """
    根据目标模型调整 system prompt 风格。

    示例：
        prompt = "请基于检索结果回答问题。每条信息标注[来源N]。"
        adapted = adapt_system_prompt(prompt, "tongyi")
        # → "您好！请您基于检索结果回答问题。每条信息请标注[来源N]，谢谢。"
    """
    rules = MODEL_ADAPTATION_RULES.get(model_provider)

    if not rules:
        return prompt  # 不认识的模型，不处理

    style = rules["system_style"]

    if style == "指令式":
        # OpenAI: 简短直接的指令即可
        return prompt

    elif style == "结构化指令":
        # DeepSeek: 结构化，分点
        lines = prompt.split("\n")
        adapted = []
        for line in lines:
            if line.strip() and not line.strip().startswith(("1.", "2.", "-")):
                # 把非列表行转为结构化指令
                adapted.append(f"- {line.strip()}")
            else:
                adapted.append(line)
        return "\n".join(adapted)

    elif style == "礼貌指令":
        # 通义千问：对礼貌用语敏感
        if not prompt.startswith("您好") and not prompt.startswith("你好"):
            prompt = "您好！" + prompt
        # 在句尾加"谢谢"
        lines = prompt.split("\n")
        if lines and lines[-1].strip():
            lines[-1] = lines[-1].rstrip("。") + "，谢谢。"
        return "\n".join(lines)

    elif style == "明确指令+示例":
        # 本地模型：加强约束
        prompt += (
            "\n\n【重要】请严格遵循以上指令。"
            "不要添加任何不在检索结果中的信息。"
            "如果检索结果中没有相关信息，请回答"暂未找到相关信息"。"
        )
        return prompt

    elif style == "规则清单":
        # Llama：规则清单 + 英文 CoT 触发
        prompt = (
            "Rules you must follow:\n"
            + "\n".join(f"{i+1}. {line.strip()}"
                        for i, line in enumerate(prompt.split("\n"))
                        if line.strip())
        )
        return prompt

    return prompt


def adapt_cot_trigger(
    model_provider: str,
) -> str:
    """返回适合目标模型的 CoT 触发词"""
    rules = MODEL_ADAPTATION_RULES.get(model_provider, {})
    return rules.get("cot_trigger", "请一步步思考")

一个容易被忽略的坑：system prompt 角色格式

不同模型对 system prompt 的角色格式有不同的"期待"：

  • OpenAI / DeepSeek：system 角色就是 system，不需要在内容中重复"你是 XX 助手"——但加上也无妨
  • 通义千问 DashScope：system prompt 中的角色定义要清晰，它会影响整个对话的语气
  • Llama 官方格式：要求 <|begin_of_text|><|start_header_id|>system<|end_header_id|> 等特殊 token——如果用 OpenAI SDK 兼容模式通常不需要手动加

from dataclasses import dataclass, field
from typing import Optional, Callable
from enum import Enum

class GroundingMode(str, Enum):

class CoTMode(str, Enum):
    NONE = "none"
    DIRECT = "direct"
    STRUCTURED = "structured"
    VERIFY = "verify"


@dataclass
class PromptEngineeringConfig:
    """Prompt Engineering 配置"""
    # System Prompt 积木
    role_key: str = "客服助手"
    role_variables: dict = field(default_factory=lambda: {"company_name": "企业"})
    knowledge_key: str = "严格检索"
    refusal_key: str = "简洁拒答"
    output_key: str = "引用标注"

    # 其他
    history_window_size: int = 6
    retrieval_top_k: int = 5
    session_timeout_minutes: int = 30


class PromptEngineeringRAG:
    """
    集成 Prompt Engineering 的 RAG 管线。

    在 L10 ConversationalRAG 的基础上，扩展了以下能力：
    - 积木式 system prompt 构建
    - 多种 grounding 模式
    - 动态 few-shot 示例
    - CoT 推理
    - 参数配置
    - 多模型 prompt 适配

    使用方式：
        rag = PromptEngineeringRAG()
        session = rag.create_session("user_001")
        reply = rag.ask("Mate60 Pro的屏幕多大？", session.session_id)
    """

    def __init__(
        self,
        config: PromptEngineeringConfig | None = None,
        retrieve_fn: Callable | None = None,
        example_library: list[Example] | None = None,
    ):
        self.config = config or PromptEngineeringConfig()

        # 组件初始化
        self.prompt_designer = PromptDesigner()
        self.few_shot_selector = FewShotSelector(examples=example_library)
        self.session_manager = SessionManager(
            session_timeout_minutes=self.config.session_timeout_minutes
        )
        self.adapter = create_chat_adapter(
            self.config.model_provider,
            self.config.model_name,
        )
        self._retrieve_fn = retrieve_fn or self._default_retrieve

        # 统计
        self.stats = {
            "total_queries": 0,
            "total_tokens": 0,
            "grounding_mode_used": {},
            "cot_mode_used": {},
        }

    def set_retrieve_fn(self, fn: Callable):
        self._retrieve_fn = fn

    def _default_retrieve(self, query: str, top_k: int = 5) -> list[dict]:
        return []

    def create_session(
        self,
        user_id: str,
        system_prompt: str | None = None,
    ):
        """创建会话，使用 PromptDesigner 构建 system prompt"""
        if system_prompt is None:
            system_prompt = self._build_system_prompt()
        return self.session_manager.create_session(user_id, system_prompt)

    def _build_system_prompt(self) -> str:
        """使用 PromptDesigner 构建 system prompt"""
        sys_config = SystemPromptConfig(
            role_key=self.config.role_key,
            role_variables=self.config.role_variables,
            knowledge_key=self.config.knowledge_key,
            refusal_key=self.config.refusal_key,
            output_key=self.config.output_key,
        )
        prompt = self.prompt_designer.build(sys_config)

        # 多模型适配
        if self.config.enable_prompt_adaptation:
            prompt = adapt_system_prompt(
                prompt, self.config.model_provider, self.config.model_name
            )

        return prompt

    def ask(self, query: str, session_id: str) -> dict:
        """处理一次对话请求"""
        import time
        start = time.time()

        # 1. 获取会话
        session = self.session_manager.get_session(session_id)
        if not session:
            return {"answer": "会话已过期，请重新开始对话。", "error": "session_expired"}

        self.stats["total_queries"] += 1

        # 2. 检索（使用增强查询，保留 L10 的指代消解能力）
        # 注：实际项目中在此调用 resolve_references()
        results = self._retrieve_fn(query, top_k=self.config.retrieval_top_k)

        # 3. 构建 grounding 上下文
        context_parts = []
        if results:
            for i, r in enumerate(results):
                context_parts.append(f"[来源{i+1}] {r.get('text', '')}")
            context_str = "\n".join(context_parts)
        else:
            context_str = "（未检索到相关信息）"

        # 4. 构建 prompt
        if self.config.cot_mode == CoTMode.NONE:
            # 标准 grounding prompt
            messages = build_grounding_prompt(
                context=context_str,
                query=query,
                mode=self.config.grounding_mode.value,
            )
        else:
            # CoT prompt
            messages = build_cot_prompt(
                context=context_str,
                query=query,
                cot_mode=self.config.cot_mode.value,
            )

        # 5. 注入 system prompt（使用会话中的 system prompt）
        system_prompt = self._build_system_prompt()
        messages[0]["content"] = system_prompt

        # 6. 注入 few-shot（可选）
        if self.config.enable_few_shot:
            selected = self.few_shot_selector.select_by_keyword(
                query, top_k=self.config.few_shot_top_k
            )
            if selected:
                few_shot_block = self.few_shot_selector.to_few_shot_prompt(selected)
                messages[0]["content"] += f"\n\n{few_shot_block}"

        # 7. 注入对话历史（如果有）
        history = session.history.get_messages()
        # 排除 system message（已经在 messages[0] 中）
        history_content = [m for m in history if m.get("role") != "system"]
        if history_content:
            # 在 user message 前插入历史
            user_msg = messages[-1]
            messages[-1] = {
                "role": "user",
                "content": (
                    "历史对话：\n"
                    + "\n".join(
                        f"{'用户' if m['role']=='user' else '助手'}: {m['content']}"
                        for m in history_content[-4:]  # 最近 2 轮
                    )
                    + f"\n\n最新问题：{query}"
                ),
            }

        # 8. 调用模型
        try:
            answer = self.adapter.chat(
                messages,
                temperature=self.config.generation.temperature,
                top_p=self.config.generation.top_p,
                max_tokens=self.config.generation.max_tokens,
            )
        except Exception as e:
            answer = f"抱歉，模型调用出错了：{str(e)}。请稍后重试。"

        # 9. 更新历史
        session.history.add_message("user", query)
        session.history.add_message("assistant", answer)

        # 10. 统计
        mode_key = f"grounding={self.config.grounding_mode.value}_cot={self.config.cot_mode.value}"
        self.stats["grounding_mode_used"][mode_key] = \
            self.stats["grounding_mode_used"].get(mode_key, 0) + 1
        self.stats["total_tokens"] += sum(len(m.get("content", "")) for m in messages) // 2

        latency_ms = int((time.time() - start) * 1000)

        return {
            "answer": answer,
            "model": self.adapter.get_model_info()["model"],
            "grounding_mode": self.config.grounding_mode.value,
            "cot_mode": self.config.cot_mode.value,
            "temperature": self.config.generation.temperature,
            "history_count": len(session.history.messages) // 2,
            "latency_ms": latency_ms,
            "retrieval_results": len(results),
        }

    def switch_config(self, **config_updates):
        """运行时切换配置"""
        for key, value in config_updates.items():
            if hasattr(self.config, key):
                setattr(self.config, key, value)
        # 如果换了模型，重建适配器
        if "model_provider" in config_updates or "model_name" in config_updates:
            self.adapter = create_chat_adapter(
                self.config.model_provider,
                self.config.model_name,
            )

故障诊断：常见问题与排查

❌ 问题一：无论给什么检索内容，模型都用"自己的知识"回答

表现：你给了检索结果，但回答中出现了检索结果中没有的信息，或者没有引用来源。

原因：system prompt 中的"知识范围"积木不够强硬。

解决方案：

  1. 切换到 strict grounding 模式——它在 system prompt 和 user prompt 中都强调了知识边界
  2. 检查你的 role 设定——"客服助手"天然比"知识助手"更容易产生幻觉吗？不一定，但要注意"助手"这个词可能让 LLM 认为应该"帮助"用户
  3. 加入"反幻觉"指令："如果你不确定或者信息不在检索结果中，不要说'据我所知'或'一般来说'——直接说不知道"

❌ 问题二：引用格式混乱——有的地方标[来源1]，有的地方不标

表现：模型"有时候标，有时候不标"，或者标了但位置不对。

原因：format instruction 不够具体，或者 temperature 太高导致格式不稳定。

排查步骤：

  1. 检查 temperature——如果 > 0.3，降到 0.0-0.1 试试
  2. 加 few-shot 示例——展示一个"完美引用"的范例，让模型照做
  3. 把引用格式要求从 system prompt 移到 user prompt 末尾——越靠近模型生成的位置，指令优先级越高
  4. 对于本地模型（如 Qwen2.5、Llama），考虑用结构化输出（JSON schema）强制格式——但这是 L13 之后的内容了

❌ 问题三：加了 CoT 后回答变得啰嗦，用户抱怨"说太多"

表现：CoT prompt 要求"先展示推理过程再回答"，但用户只想要一个简洁的答案。

解决方案：

  • 使用 direct 而非 structured——direct 的推理部分更短
  • 在 prompt 中明确格式："思考过程请用简短的语言在[思考]中展示，最终答案在[答案]中展示"——这样用户可以选择只看答案部分
  • 如果用户反馈"说太多"，说明这个场景根本不需要 CoT——切回 CoTMode.NONE

❌ 问题四：切换模型后，prompt 效果明显下降

表现：在 GPT-4o-mini 上效果很好的 prompt，换到通义千问后格式崩塌或拒答指令失效。

原因：不同模型对指令的"理解方式"不同——GPT 系列的指令遵循能力最强。

排查步骤：

  1. 检查 adapt_system_prompt() 是否被调用——它应该自动处理模型适配
  2. 通义千问对"礼貌指令"敏感——把"你不要..."改成"请你不要..."
  3. DeepSeek 对结构化指令响应最好——把多行文本转成 bullet list
  4. 本地模型（Qwen2.5/Llama）需要更明确的格式约束——增加"【重要】必须遵守"等强调词
