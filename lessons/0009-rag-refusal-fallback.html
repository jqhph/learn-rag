Lesson 0009 — 模块二 · 质量地基

RAG 拒绝策略与兜底机制：不回答的智慧

本课目标：你不必回答所有问题——事实上，知道何时不回答，是客服场景中最关键的底线能力。本课教你如何诚实拒答而非编造幻觉，并建立从本地检索到人工转接的多层兜底链路。

你今天要获得的能力

学完本课，你将拥有：

  • 一套 检索置信度评估机制：用检索分数 + 生成概率判断"该不该答"
  • 三种 "我不知道"回答模板：透明拒答 / 部分信息 / 引导建议
  • 一条 多层兜底链：本地检索 → 联网搜索 → 人工转接，逐级 fallback
  • 一种 知识库覆盖度检测方法：主动发现"知识盲区"并驱动知识库迭代
  • 一份 拒答 vs 幻觉的权衡框架：用 L7 的 Faithfulness 指标量化决策边界

这是你 RAG 系统从"能回答问题"到"可靠地回答问题"的关键一跃——用户问得刁钻，系统更可靠。

场景导入：一次拒答失误的代价

❌ 真实案例：客服 RAG 的"幻觉拒答"

用户："我不小心把手机掉水里了，你们的防水等级是多少？"

知识库中已有："IP68 防水等级，可在 1.5 米水深下浸泡 30 分钟。"（但被错误分类到"产品规格"文件夹，未被检索到）

检索返回：空调产品说明书（完全不相关）、保修条款片段（也不相关）

LLM 生成："我们的手机在正常使用条件下是防溅水的，但建议避免接触液体。具体情况请咨询客服。"

问题：系统检索到了噪音内容，LLM 没有充分依据却仍然给出了模糊回答——既不是准确答案，也不是诚实拒答。用户可能据此做出错误的"尝试"（比如用水冲手机），造成实际损失。

这个案例暴露了两个问题：

  1. 检索质量：真正的答案（IP68）未被召回
  2. 生成策略：检索结果不可靠时，LLM 没有选择拒答，而是"硬着头皮回答"

本课专注解决 第二个问题：当检索结果不可靠时，系统如何做正确的决定——不答、部分答、还是转接？

拒答 vs 幻觉：一个关键权衡

倾向拒答（保守）倾向回答（激进）
优点零幻觉；用户信任度高；合规安全用户满意度高（问题被尝试回答）；转化率高
缺点拒答率过高可能让用户挫败；知识库"明明有却不答"可能产生幻觉/错误信息；流失用户信任（一次幻觉毁十次正确回答）
适用场景医疗建议、法律咨询、金融交易闲聊、产品推荐、非关键场景
评估指标拒答率、准确率（被答的问题里有多少正确）回答率、用户满意度

这不是"哪个更好"的问题——而是 你的场景决定了你的位置。客服 Q&A 应该居于中间偏保守：宁可说"我不确定"再转接人工，也不要给出错误答案。

第一部分：检索置信度评估——判断"该不该答"

要让系统知道"什么时候不该回答"，第一步是量化"我有多确定检索到的内容是可靠的"。

1.1 三种置信度信号

单一信号不足以做出可靠判断——你需要组合使用以下三类信号：

📊 检索分数

检索返回的相似度/分数绝对值

  • 余弦相似度（embedding 距离）
  • BM25 分数
  • RRF 融合后的排名分数

🔬 上下文相关性

检索到的内容是否真正回答了问题

  • Cross-Encoder 打分（L4 学过）
  • LLM-as-a-Judge 快速判断
  • 关键词匹配比例

🎯 生成概率

LLM 生成答案时的"自信度"

  • logprobs 输出 token 概率
  • 回答中是否有模糊词（"可能"、"也许"）
  • 生成内容的内部一致性

实践中，检索分数是最容易获取的信号，但单独使用时可靠性有限（不同 embedding 模型的分数分布不同）。最可靠的策略是"检索分数阈值 + Cross-Encoder 验证"的两步判断。

1.2 空结果处理：第一个"不该答"的场景

最明显的拒答场景：检索结果为空。但"空"并不总是零结果——有时 top-1 的分数只有 0.15，这本质上也是"空"。

"""
confidence.py — 检索置信度评估模块

判断检索结果是否足够可靠，支持"该不该答"的决策。
依赖：pip install sentence-transformers（可选，用于 Cross-Encoder）
"""
import os
import statistics
from typing import Any
from openai import OpenAI

client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))


def assess_retrieval_confidence(
    query: str,
    results: list[dict],
    *,
    min_score: float = 0.4,
    min_results: int = 1,
) -> dict:
    """
    评估检索结果的置信度。

    参数：
        query: 用户查询
        results: 检索结果列表，每项含 {"text": ..., "score": float}
        min_score: 相似度最低阈值（低于此值视为"不可靠"）
        min_results: 最少需要的有效结果数量

    返回：
        {
            "is_confident": bool,
            "confidence_level": "high" | "medium" | "low" | "empty",
            "top_score": float,
            "avg_score": float,
            "effective_count": int,  # 分数 > min_score 的结果数
            "reason": str
        }
    """
    if not results:
        return {
            "is_confident": False,
            "confidence_level": "empty",
            "top_score": 0.0,
            "avg_score": 0.0,
            "effective_count": 0,
            "reason": "检索结果为空，无法回答",
        }

    scores = [r.get("score", 0.0) for r in results]
    top_score = max(scores)
    avg_score = statistics.mean(scores)
    effective = sum(1 for s in scores if s >= min_score)

    # 判断逻辑
    if effective < min_results:
        return {
            "is_confident": False,
            "confidence_level": "low",
            "top_score": round(top_score, 4),
            "avg_score": round(avg_score, 4),
            "effective_count": effective,
            "reason": f"有效结果不足（{effective}/{min_results}），最高分 {top_score:.4f} 低于阈值 {min_score}",
        }
    elif top_score >= min_score * 2:              # 高分：远高于阈值
        level = "high"
    elif top_score >= min_score:                  # 中分：刚过线
        level = "medium"
    else:
        level = "low"

    return {
        "is_confident": level in ("high", "medium"),
        "confidence_level": level,
        "top_score": round(top_score, 4),
        "avg_score": round(avg_score, 4),
        "effective_count": effective,
        "reason": f"置信度 {level}（top={top_score:.4f}, avg={avg_score:.4f}, 有效={effective}）",
    }


def cross_encoder_verify(
    query: str,
    results: list[dict],
    *,
    threshold: float = 0.3,
) -> float:
    """
    用 LLM 快速判断检索结果与查询的相关性（Cross-Encoder 的轻量替代）。
    返回"相关"结果的比例。
    如果安装了 sentence-transformers，可用真实 Cross-Encoder 替代。
    """
    # 更精确的做法是用 Cross-Encoder 模型（如 BAAI/bge-reranker-v2-m3）
    # 这里作为轻量替代，用 LLM 快速打分
    system_prompt = """你是一个检索相关性判断器。
对于每个 (查询, 文档) 对，判断该文档是否与查询相关。
只回答 "相关" 或 "不相关" —— 不输出其他内容。

相关定义：文档包含回答该查询所需的任何有用信息。
不相关定义：文档与查询完全无关，或只涉及相同的宽泛主题但没有有用信息。"""

    relevant_count = 0
    for r in results:
        user_prompt = f"""查询：{query}

文档：{r["text"][:500]}"""
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            temperature=0.0,
            max_tokens=16,
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_prompt},
            ],
        )
        answer = resp.choices[0].message.content.strip()
        if "相关" in answer:
            relevant_count += 1

    ratio = relevant_count / len(results) if results else 0.0
    return ratio


def decide_if_should_answer(
    query: str,
    results: list[dict],
    *,
    score_threshold: float = 0.4,
    cross_encoder_threshold: float = 0.3,
    min_effective: int = 1,
) -> dict:
    """
    综合决策：这个查询应该回答还是拒答？

    决策流程：
    1. 检索分数评估 → 如果 empty → 拒答
    2. Cross-Encoder 相关性验证 → 如果相关性比例太低 → 部分拒答
    3. 综合判断 → 返回决策
    """
    # 第一步：检索分数评估
    confidence = assess_retrieval_confidence(
        query, results,
        min_score=score_threshold,
        min_results=min_effective,
    )

    if confidence["confidence_level"] == "empty":
        return {
            "decision": "refuse",
            "refusal_type": "no_knowledge",
            "confidence": confidence,
            "relevance_ratio": 0.0,
            "reason": "知识库中未检索到任何相关内容",
        }

    # 第二步：Cross-Encoder 验证
    relevance_ratio = cross_encoder_verify(
        query, results, threshold=cross_encoder_threshold
    )

    # 第三步：综合判断
    if confidence["confidence_level"] == "high" and relevance_ratio >= 0.5:
        return {
            "decision": "answer",
            "refusal_type": None,
            "confidence": confidence,
            "relevance_ratio": relevance_ratio,
            "reason": f"置信度高（{confidence['confidence_level']}），相关性充分（{relevance_ratio:.0%}）",
        }
    elif confidence["confidence_level"] == "low" or relevance_ratio < 0.3:
        return {
            "decision": "refuse",
            "refusal_type": "low_confidence",
            "confidence": confidence,
            "relevance_ratio": relevance_ratio,
            "reason": f"置信度不足（{confidence['confidence_level']}）或相关性偏低（{relevance_ratio:.0%}）",
        }
    else:
        return {
            "decision": "partial",
            "refusal_type": "partial_knowledge",
            "confidence": confidence,
            "relevance_ratio": relevance_ratio,
            "reason": f"部分相关（相关性 {relevance_ratio:.0%}），可提供部分信息",
        }


# ========== 使用示例 ==========
if __name__ == "__main__":
    # 场景 1：检索结果为空
    print("=== 场景 1：空结果 ===")
    decision = decide_if_should_answer("手机防水等级是多少？", [])
    print(f"决策：{decision['decision']}")
    print(f"原因：{decision['reason']}")

    # 场景 2：检索结果分数高
    print("\n=== 场景 2：高置信度 ===")
    decision = decide_if_should_answer(
        "退货的运费谁承担？",
        [
            {"text": "7天无理由退货由客户承担运费，质量问题由商家承担。", "score": 0.85},
            {"text": "运费政策根据具体情况而定。", "score": 0.72},
        ],
    )
    print(f"决策：{decision['decision']}")
    print(f"原因：{decision['reason']}")

    # 场景 3：检索结果分数低
    print("\n=== 场景 3：低置信度 ===")
    decision = decide_if_should_answer(
        "你们的手机能放水里多久？",
        [
            {"text": "本产品享受一年质保服务。", "score": 0.18},
            {"text": "售后服务热线：400-800-8888。", "score": 0.12},
        ],
    )
    print(f"决策：{decision['decision']}")
    print(f"原因：{decision['reason']}")

⚠️ 重要：检索分数阈值的校准

min_score=0.4 是一个经验值，实际使用时必须针对你的 embedding 模型校准：

  • OpenAI text-embedding-3-small：分数通常在 0.6-0.9 之间，阈值建议 0.65
  • BGE-zh 系列：分数分布更广（0.2-0.9），阈值建议 0.5
  • 本地小模型（Ollama）：分数可能整体偏低，阈值建议 0.3

校准方法：在你的知识库上随机采样 200 条查询，人工标注"检索结果是否可用"，然后找最大化 F1 的阈值。详见本课练习题。

什么是 Cross-Encoder？它与 bi-encoder 有何不同？

你在 L4 学过 bi-encoder（如 BGE-zh）和 Cross-Encoder：

  • Bi-Encoder：将查询和文档分别编码为向量 → 余弦相似度（快，适合检索）
  • Cross-Encoder：将查询和文档拼接后输入同一个 transformer → 输出相关性分数（慢但准，适合重排序和验证）

在拒答场景中，Cross-Encoder 的作用是二次验证：检索阶段可能拿到了高分噪音（比如关键词匹配但语义无关），Cross-Encoder 能更精确地识别这种"假阳性"。

上面代码中的 cross_encoder_verify() 用 LLM 替代了真实的 Cross-Encoder 模型——这是为了降低依赖。生产环境中，建议替换为 sentence-transformers 的 Cross-Encoder 或 BGE Re-Ranker，后者比 LLM 调用快 10-100 倍、成本低得多。

第二部分："我不知道"回答模板设计

当系统决定"不回答"时，如何优雅地拒答，直接影响用户体验。硬邦邦的"我无法回答这个问题"会让用户立即感到挫败。

好的拒答回答应该做到三件事：

  1. 透明告知："为什么我无法回答这个问题"——让用户理解系统的局限，而非觉得系统在敷衍
  2. 提供部分信息：即使不能完全回答，也能提供有帮助的相关信息（如果有的话）
  3. 引导用户：给用户下一步行动建议——改天再问、联系人工、或提供更多上下文

2.1 三种拒答模板

根据上一节的 refusal_type，系统应使用不同的回答模板：

🟡 模板一：完全透明拒答

适用场景：检索为空或完全不相关

抱歉，我目前的知识库中没有关于"{{ query }}"的
信息。我还在不断学习和更新中。

建议您可以：
1. 换一种表述方式再次提问
2. 联系人工客服获取准确信息
3. 访问我们的帮助中心自助查询

🟢 模板二：部分信息 + 明确边界

适用场景：部分相关但不够完整

根据我目前掌握的信息，关于"{{ query }}"我
可以告诉您：

{{ partial_info }}

但我不确定这是否完全回答了您的问题，
建议您联系人工客服获取更准确的答案。

🔵 模板三：引导式拒答

适用场景：问题过于模糊或需要人工处理

我理解您在询问"{{ query }}"。

为了更准确地帮助您，是否可以：
1. 提供更多关于{{ suggestion }}的信息？
2. 或直接联系人工客服为您处理？

人工客服可以处理更复杂的个性化问题。

2.2 模板管理模块

"""
refusal_templates.py — 拒答回答模板管理器

根据拒答类型选择适当的回答模板，支持自定义变量注入。
"""

from typing import Any


TEMPLATES: dict[str, dict] = {
    "no_knowledge": {
        "template": """抱歉，我目前的知识库中没有关于"{{ query }}"的信息。

相关知识库还在不断更新中，您可以：

1. **换个说法再问一次**——换个关键词可能会找到相关内容
2. **联系人工客服**——获取最准确的解答
3. 或访问我们的帮助中心自助查询

请问还有什么其他问题我可以帮您解答吗？""",
        "description": "完全不知道——检索结果为空",
    },
    "low_confidence": {
        "template": """关于"{{ query }}"，我没有找到足够可靠的信息来回答您。

我检索到的内容与您的问题相关度不高，为了避免给您错误信息，建议您：

1. **联系人工客服**——为您准确解答
2. **重新表述问题**——可能换个说法我能找到更多信息

很抱歉没能直接帮到您。""",
        "description": "检索结果不可靠——分数低或相关性不足",
    },
    "partial_knowledge": {
        "template": """关于"{{ query }}"，我找到了部分相关信息：

{{ partial_info }}

{{#if suggestions}}
**另外，您可能还想了解：**
{{ suggestions }}
{{/if}}

不过我不确定这是否完整回答了您的问题，如需更准确的解答，建议联系人工客服。""",
        "description": "部分相关知识——可提供但不完整",
    },
    "vague_query": {
        "template": """我理解您在询问"{{ query }}"，但这个问题可能涉及多个方面。

为了更准确地帮助您，可以告诉我更多细节吗？例如：
{{#each clarification}}
- {{this}}
{{/each}}

或者您可以直接联系人工客服获取一对一帮助。""",
        "description": "问题太模糊——需要用户澄清",
    },
}


def render_refusal(
    refusal_type: str,
    variables: dict[str, Any],
) -> str:
    """
    根据拒答类型和变量渲染回答文本。

    参数：
        refusal_type: no_knowledge | low_confidence | partial_knowledge | vague_query
        variables: 模板变量，如 {"query": "...", "partial_info": "..."}

    返回：
        渲染后的回答文本（纯文本，非 HTML）
    """
    template_info = TEMPLATES.get(refusal_type)
    if not template_info:
        template_info = TEMPLATES["no_knowledge"]

    template = template_info["template"]
    query = variables.get("query", "")

    # 简易模板引擎（生产环境建议用 Jinja2 或 string.Template）
    result = template.replace("{{ query }}", query)

    partial_info = variables.get("partial_info", "")
    result = result.replace("{{ partial_info }}", partial_info)

    suggestions = variables.get("suggestions", "")
    if "{{/if}}" in result:
        # 处理简单条件块
        if suggestions:
            result = result.replace("{{#if suggestions}}\n", "").replace("\n{{/if}}", "")
        else:
            # 移除整个条件块
            start = result.find("{{#if suggestions}}")
            if start >= 0:
                end = result.find("{{/if}}", start) + len("{{/if}}")
                # 清除条件块内的内容
                inner_start = result.find("\n", start) + 1
                block_content = result[start:end]
                result = result.replace(block_content, "")
        result = result.replace("{{ suggestions }}", suggestions)

    # 处理 each 块
    clarification = variables.get("clarification", [])
    if "{{#each clarification}}" in result:
        if clarification:
            items = "\n".join(f"- {item}" for item in clarification)
            start = result.find("{{#each clarification}}")
            end = result.find("{{/each}}", start) + len("{{/each}}")
            result = result[:start] + items + result[end:]
        else:
            start = result.find("{{#each clarification}}")
            end = result.find("{{/each}}", start) + len("{{/each}}")
            result = result[:start] + "请提供更多信息" + result[end:]

    return result.strip()


def build_refusal_response(decision: dict, query: str, **extra) -> str:
    """构建完整的拒答回复（决策 → 模板渲染）"""
    refusal_type = decision.get("refusal_type", "no_knowledge")
    return render_refusal(refusal_type, {"query": query, **extra})


# ========== 使用示例 ==========
if __name__ == "__main__":
    # 场景 1：完全不知道
    print("=== 场景 1：无知识 ===")
    decision = {"refusal_type": "no_knowledge", "confidence_level": "empty"}
    reply = build_refusal_response(decision, "5G 频段支持情况")
    print(reply)

    print("\n=== 场景 2：部分信息 ===")
    decision = {"refusal_type": "partial_knowledge", "confidence_level": "medium"}
    reply = build_refusal_response(
        decision,
        "手机进水了怎么办？",
        partial_info="IP68 防水等级可在 1.5 米水深下浸泡 30 分钟。",
        suggestions="进水后的正确处理步骤、保修政策",
    )
    print(reply)

模板设计的心理学

好的拒答模板背后有一些心理学原理：

  • 归因透明：说"我的知识库没有收录"而非"无法回答"——将原因归咎于系统的局限而非用户的问题
  • 提供替代路径：每次拒答都给出至少两个后续选项——减少用户的"死胡同感"
  • 语气同理："很抱歉"、"建议您可以"——保持礼貌谦逊
  • 避免过度承诺：不说"稍后就有答案"或"我会转达给相关团队"——除非你真的有这套流程

参考：Nielsen Norman Group 的 error message 设计原则——错误信息应该有四个要素：发生了什么、为什么、怎么解决、不要责怪用户。这些原则同样适用于 AI 拒答。

第三部分：多层兜底链——从本地到人工

当本地知识库检索失败时，系统不应该直接"投降"。一个工业级的 RAG 系统应该有一整套 兜底链路（Graceful Degradation Chain）：

用户提问
→
本地知识库
向量检索
→
置信度评估
→
高置信度？
→
正常回答

↓ 低置信度时逐级降级 ↓
第一级：本地知识库
改写查询再试一次
→
第二级：联网搜索
Web Search API
→
第三级：人工转接
生成工单

    system_prompt = """你是一个搜索查询优化专家。

# ========== 完整兜底链 ==========

def execute_fallback_chain(
    query: str,
    retrieve_fn: callable,
    *,
    enable_web_search: bool = False,
    enable_human_fallback: bool = True,
    max_retries: int = 1,
) -> dict:
    """
    执行完整的多层兜底链。

    流程：
    1. 本地检索 → 置信度评估
    2. 如果低置信度 → 改写查询再检索一次（最多 max_retries 次）
    3. 如果仍然低置信度 → 联网搜索（如果启用）
    4. 如果仍然没有 → 人工转接（如果启用）
    5. 如果什么都不启用 → 透明拒答

    返回最后的决策和回答。
    """
    from confidence import decide_if_should_answer

    # 第一层：本地检索
    results = retrieve_fn(query)
    decision = decide_if_should_answer(query, results)

    if decision["decision"] == "answer":
        return {
            "stage": "local_kb",
            "decision": "answer",
            "results": results,
            "refusal_response": None,
        }

    # 第二层：改写查询重试
    rewritten = query_rewrite(query)
    for i, rq in enumerate(rewritten[:max_retries]):
        retry_results = retrieve_fn(rq)
        retry_decision = decide_if_should_answer(query, retry_results)
        if retry_decision["decision"] == "answer":
            return {
                "stage": "retry",
                "decision": "answer",
                "results": retry_results,
                "rewritten_query": rq,
                "refusal_response": None,
            }

    decision.update(retry_decision if "retry_decision" in dir() else {})

    # 第三层：联网搜索
    if enable_web_search:
        web_result = web_search_fallback(query)
        if web_result:
            return {
                "stage": "web",
                "decision": "answer_web",
                "web_result": web_result,
                "refusal_response": None,
            }

    # 第四层：人工转接
    if enable_human_fallback:
        fallback = human_fallback(query)
        return {
            "stage": "human",
            "decision": "human_transfer",
            "ticket": fallback["ticket"],
            "refusal_response": fallback["message"],
        }

    # 最终：透明拒答
    from refusal_templates import build_refusal_response
    reply = build_refusal_response(decision, query)
    return {
        "stage": "refusal",
        "decision": "refuse",
        "refusal_response": reply,
    }

兜底链设计的核心原则：按成本降级

每一层兜底的成本（时延 + 金钱 + 人力）依次升高：

层级时延金钱成本人力成本推荐场景
1. 本地检索~200ms几乎为零无所有查询
2. 改写重试+~500ms少量 LLM token无首次检索低置信度时
3. 联网搜索+~1-3sAPI 调用费无本地检索失败但用户需要实时信息
4. 人工转接+~5s（应答）客服工资高复杂/敏感/高价值问题

关键决策：不是每个问题都需要走到最后一层。对于简单 FAQ，到第一层就够；对于需要实时数据（如"今天能发货吗"）可以开到第三层；对于投诉/退款，直接跳到第四层。

第四部分：知识库覆盖度检测——发现你的盲区

拒答策略是"被动防御"，而覆盖度检测是"主动改善"。你需要在系统运行中持续回答一个问题：我们知识库的盲区在哪里？

4.1 覆盖度检测的基本方法

覆盖度检测的核心思路：统计真实用户的查询，分析有多少被回答了、多少被拒答了、拒答的原因是什么。

"""
coverage_monitor.py — 知识库覆盖度检测

记录查询日志，分析拒答模式，计算覆盖度指标。
"""

import json
import statistics
from collections import Counter
from datetime import datetime, timedelta
from typing import Any


class CoverageMonitor:
    """
    知识库覆盖度监控器。

    记录每次查询的结果，生成覆盖度报告。
    """

    def __init__(self, log_path: str = "query_log.jsonl"):
        self.log_path = log_path

    def log_query(
        self,
        query: str,
        decision: str,
        refusal_type: str | None = None,
        confidence_level: str = "",
        relevance_ratio: float = 0.0,
        knowledge_base_category: str = "general",
        **extra,
    ):
        """记录一条查询日志"""
        entry = {
            "timestamp": datetime.utcnow().isoformat(),
            "query": query,
            "decision": decision,
            "refusal_type": refusal_type,
            "confidence_level": confidence_level,
            "relevance_ratio": relevance_ratio,
            "knowledge_base_category": knowledge_base_category,
            **extra,
        }
        with open(self.log_path, "a", encoding="utf-8") as f:
            f.write(json.dumps(entry, ensure_ascii=False) + "\n")

    def generate_report(self, hours: int = 24) -> dict:
        """
        生成过去 N 小时的覆盖度报告。

        返回：
            {
                "total_queries": int,
                "answer_rate": float,      # 被回答的占比
                "refusal_rate": float,     # 被拒答的占比
                "human_transfer_rate": float,  # 转人工的占比
                "refusal_breakdown": {      # 拒答类型分布
                    "no_knowledge": int,
                    "low_confidence": int,
                    "partial_knowledge": int,
                    ...
                },
                "top_refused_queries": [str],  # 最常见的拒答查询
                "category_coverage": {      # 每个分类的覆盖度
                    "退货政策": {"queries": 100, "answer_rate": 0.85, ...},
                    ...
                },
                "uncovered_queries": [str],  # 需要优先补充知识的查询
            }
        """
        # 读取日志
        cutoff = datetime.utcnow() - timedelta(hours=hours)
        entries = []
        try:
            with open(self.log_path, "r", encoding="utf-8") as f:
                for line in f:
                    entry = json.loads(line)
                    ts = datetime.fromisoformat(entry["timestamp"])
                    if ts >= cutoff:
                        entries.append(entry)
        except FileNotFoundError:
            return {"total_queries": 0, "error": "日志文件不存在"}

        if not entries:
            return {"total_queries": 0, "message": "指定时间内无查询记录"}

        total = len(entries)
        decisions = [e["decision"] for e in entries]

        # 回答率
        answered = sum(1 for d in decisions if d == "answer")
        refused = sum(1 for d in decisions if d == "refuse")
        transferred = sum(1 for d in decisions if d == "human_transfer")

        # 拒答类型分布
        refusal_types = Counter(
            e.get("refusal_type", "unknown")
            for e in entries
            if e["decision"] == "refuse"
        )

        # 按分类统计
        categories = {}
        for e in entries:
            cat = e.get("knowledge_base_category", "general")
            if cat not in categories:
                categories[cat] = {"total": 0, "answered": 0}
            categories[cat]["total"] += 1
            if e["decision"] == "answer":
                categories[cat]["answered"] += 1
        for cat in categories:
            t = categories[cat]["total"]
            a = categories[cat]["answered"]
            categories[cat]["answer_rate"] = round(a / t, 3) if t else 0.0

        # 最常见的拒答查询（按拒绝类型分组）
        refused_queries = [
            e["query"] for e in entries if e["decision"] in ("refuse", "human_transfer")
        ]
        top_refused = [
            q for q, _ in Counter(refused_queries).most_common(10)
        ]

        # 需要优先补充知识的查询：
        # 规则：被拒答 > 1 次，或转人工 > 1 次
        query_counter = Counter(e["query"] for e in entries if e["decision"] in ("refuse", "human_transfer"))
        uncovered = [q for q, c in query_counter.most_common(20) if c >= 2]

        return {
            "report_period_hours": hours,
            "total_queries": total,
            "answer_rate": round(answered / total, 3) if total else 0.0,
            "refusal_rate": round(refused / total, 3) if total else 0.0,
            "human_transfer_rate": round(transferred / total, 3) if total else 0.0,
            "refusal_breakdown": dict(refusal_types),
            "category_coverage": categories,
            "top_refused_queries": top_refused[:10],
            "uncovered_queries": uncovered[:10],
        }


# ========== 使用示例 ==========
if __name__ == "__main__":
    monitor = CoverageMonitor()

    # 模拟记录一些查询
    monitor.log_query("退货运费谁承担", "answer", confidence_level="high", relevance_ratio=0.9)
    monitor.log_query("手机防水等级", "refuse", "no_knowledge", confidence_level="empty")
    monitor.log_query("5G 频段", "refuse", "no_knowledge")
    monitor.log_query("如何开通国际漫游", "human_transfer", "low_confidence")

    report = monitor.generate_report(hours=72)
    print(f"总查询数：{report['total_queries']}")
    print(f"回答率：{report['answer_rate']:.1%}")
    print(f"拒答率：{report['refusal_rate']:.1%}")
    print(f"转人工率：{report['human_transfer_rate']:.1%}")
    print(f"拒答类型分布：{report['refusal_breakdown']}")
    print(f"需要补充的盲区：{report['uncovered_queries']}")

4.2 驱动知识库迭代的闭环

覆盖度检测的价值不在于报告本身，而在于它能驱动一个 数据驱动型知识库迭代循环：

用户查询
→
RAG 回答/拒答
→
记录日志
→
覆盖度报告
→
识别盲区
→
新增/更新知识
→
重新入库
→
回答率↑

覆盖度目标参考值

不同的成熟度阶段有不同的目标：

阶段回答率目标转人工率策略
初期上线≥ 50%≤ 30%优先收集盲区数据，快速补充高频问题
稳定运行≥ 70%≤ 15%优化检索质量，完善低频问题
成熟阶段≥ 85%≤ 5%持续增量更新，监控回答质量退化

这些值是经验参考——不同行业的"好"标准不同。比如电商客服的回答率天花板比医疗咨询高（因为医疗有更多场景必须转人工）。

第五部分：整合——构建拒绝策略感知的 RAG Pipeline

现在我们将前面四个模块整合在一起，构建一个 拒绝策略感知的完整 RAG Pipeline：

"""
rag_with_refusal.py — 带拒绝策略的 RAG Pipeline

整合：检索 → 置信度评估 → 分层兜底 → 拒答/回答
"""
import os
from openai import OpenAI

from confidence import decide_if_should_answer
from fallback_chain import (
    execute_fallback_chain,
    query_rewrite,
    web_search_fallback,
    human_fallback,
)
from refusal_templates import build_refusal_response


client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))


class KnowledgeBase:
    """
    知识库接口——你的向量检索封装。

    这是占位实现，实际应使用 Chroma / Qdrant / Milvus 等真实向量库。
    """

    def __init__(self):
        self.documents = []

    def add_documents(self, docs: list[dict]):
        """添加文档，每项含 {"text": str, "id": str, "metadata": dict}"""
        self.documents = docs  # 简化实现——实际应向量化后存入向量库

    def search(self, query: str, top_k: int = 5) -> list[dict]:
        """检索——占位实现，模拟返回结果"""
        # 实际应调用向量库的相似度搜索
        results = []
        for i, doc in enumerate(self.documents[:top_k]):
            # 模拟相似度分数
            score = max(0.0, 1.0 - i * 0.2)
            results.append({
                "id": doc.get("id", str(i)),
                "text": doc["text"],
                "score": score,
                "metadata": doc.get("metadata", {}),
            })
        return results


def build_rag_prompt(query: str, contexts: list[str]) -> list[dict]:
    """构建 RAG 生成 prompt"""
    context_text = "\n\n".join(
        f"[{i+1}] {c}" for i, c in enumerate(contexts)
    )
    system = """你是一个客服助手。请根据提供的上下文信息回答用户问题。

规则：
1. 只能使用上下文中明确提到的信息
2. 如果上下文不足以回答问题，请如实说明
3. 引用信息来源（使用 [1]、[2] 标注）
4. 用中文回答"""

    user = f"""【上下文信息】
{context_text}

【用户问题】
{query}

【回答】"""
    return [
        {"role": "system", "content": system},
        {"role": "user", "content": user},
    ]


class RAGWithRefusal:
    """
    带拒绝策略的 RAG 系统。

    使用方式：
        kb = KnowledgeBase()
        kb.add_documents([...])
        rag = RAGWithRefusal(kb)
        result = rag.answer("手机防水等级是多少？")
        print(result["reply"])
    """

    def __init__(
        self,
        knowledge_base: KnowledgeBase,
        *,
        enable_web_search: bool = False,
        enable_human_fallback: bool = True,
        top_k: int = 5,
        score_threshold: float = 0.4,
    ):
        self.kb = knowledge_base
        self.enable_web_search = enable_web_search
        self.enable_human_fallback = enable_human_fallback
        self.top_k = top_k
        self.score_threshold = score_threshold

    def answer(self, query: str) -> dict:
        """
        对用户查询执行完整的 RAG + 拒答流程。

        返回：
            {
                "reply": str,           # 最终回复文本
                "decision": str,        # answer | refuse | human_transfer
                "stage": str,           # local_kb | retry | web | human | refusal
                "confidence": {...},    # 置信度评估详情
                "sources": [...],       # 引用的来源（如果有）
            }
        """
        # 第一步：本地检索
        results = self.kb.search(query, top_k=self.top_k)
        decision = decide_if_should_answer(
            query, results,
            score_threshold=self.score_threshold,
        )

        # 第二步：根据决策决定下一步
        if decision["decision"] == "answer":
            # 正常回答
            contexts = [r["text"] for r in results]
            messages = build_rag_prompt(query, contexts)
            resp = client.chat.completions.create(
                model="gpt-4o-mini",
                temperature=0.3,
                messages=messages,
            )
            reply = resp.choices[0].message.content
            return {
                "reply": reply,
                "decision": "answer",
                "stage": "local_kb",
                "confidence": decision,
                "sources": results,
            }

        # 第三步：改写重试
        rewritten = query_rewrite(query)
        for rq in rewritten[:2]:
            retry_results = self.kb.search(rq, top_k=self.top_k)
            retry_decision = decide_if_should_answer(
                query, retry_results,
                score_threshold=self.score_threshold,
            )
            if retry_decision["decision"] == "answer":
                contexts = [r["text"] for r in retry_results]
                messages = build_rag_prompt(query, contexts)
                resp = client.chat.completions.create(
                    model="gpt-4o-mini",
                    temperature=0.3,
                    messages=messages,
                )
                return {
                    "reply": resp.choices[0].message.content,
                    "decision": "answer",
                    "stage": "retry",
                    "confidence": retry_decision,
                    "rewritten_query": rq,
                    "sources": retry_results,
                }

        # 第四步：联网搜索兜底
        if self.enable_web_search:
            web_result = web_search_fallback(query)
            if web_result:
                # 用联网结果作为上下文生成回答
                contexts = [web_result["content"]]
                messages = build_rag_prompt(
                    f"{query}\n（注意：以下信息来自在线搜索，可靠性未经审核，请告知用户）",
                    contexts,
                )
                resp = client.chat.completions.create(
                    model="gpt-4o-mini",
                    temperature=0.3,
                    messages=messages,
                )
                reply = resp.choices[0].message.content
                return {
                    "reply": reply + "\n\n⚠️ 以上信息来自在线搜索，建议核实后使用。",
                    "decision": "answer_web",
                    "stage": "web",
                    "sources": [web_result],
                }

        # 第五步：人工转接
        if self.enable_human_fallback:
            fallback = human_fallback(query)
            return {
                "reply": fallback["message"],
                "decision": "human_transfer",
                "stage": "human",
                "ticket": fallback["ticket"],
            }

        # 最终：透明拒答
        reply = build_refusal_response(decision, query)
        return {
            "reply": reply,
            "decision": "refuse",
            "stage": "refusal",
            "confidence": decision,
        }
