1.2 Pydantic 模型 —— API 的数据契约

任何 API 设计的第一步是定义请求和响应的数据结构。FastAPI 使用 Pydantic 做校验和文档生成：

"""
app/models.py — RAG API 的请求/响应模型
"""

from pydantic import BaseModel, Field
from typing import Optional, Any


class RetrieveRequest(BaseModel):
    """检索请求：只检索不生成"""
    query: str = Field(..., description="用户查询")
    top_k: int = Field(default=5, ge=1, le=50, description="返回结果数")
    score_threshold: float = Field(default=0.0, ge=0.0, le=1.0, description="最低分数阈值")
    history: Optional[list[dict]] = Field(default=None, description="对话历史（可选）")


class RetrievedChunk(BaseModel):
    """单条检索结果"""
    text: str
    score: float
    source: Optional[str] = None
    metadata: dict = Field(default_factory=dict)


class RetrieveResponse(BaseModel):
    """检索响应"""
    query: str
    augmented_query: Optional[str] = None
    results: list[RetrievedChunk]
    total_found: int
    latency_ms: int


class AskRequest(BaseModel):
    """问答请求"""
    query: str = Field(..., description="用户问题")
    session_id: Optional[str] = Field(default=None, description="会话 ID（无则新建）")
    user_id: Optional[str] = Field(default="anonymous", description="用户标识")
    stream: bool = Field(default=False, description="是否流式输出")
    temperature: Optional[float] = Field(default=None, ge=0.0, le=2.0)
    max_tokens: Optional[int] = Field(default=None, ge=1, le=16384)


class AskResponse(BaseModel):
    """问答响应"""
    answer: str
    session_id: str
    augmented_query: Optional[str] = None
    model: str
    history_count: int
    retrieval_results: list[RetrievedChunk] | None = None
    refusal: bool = False
    latency_ms: int


class EvaluateRequest(BaseModel):
    """质量评估请求"""
    query: str
    answer: str
    contexts: list[str]
    metrics: list[str] = Field(default=["faithfulness", "answer_relevancy"])


class EvaluateResponse(BaseModel):
    """质量评估响应"""
    query: str
    scores: dict[str, float]
    details: Optional[dict] = None


class HealthResponse(BaseModel):
    """健康检查响应"""
    status: str  # "healthy" | "degraded" | "unhealthy"
    version: str
    services: dict[str, str]  # 各依赖服务的健康状态
    rag_ready: bool  # 知识库是否已加载

1.3 配置管理 —— 不再把 Key 写死在代码里

关键原则 所有环境差异都通过配置管理，绝不硬编码。

"""
app/config.py — 集中配置管理
"""

from pydantic_settings import BaseSettings
from typing import Optional


class Settings(BaseSettings):
    """
    应用配置 —— 优先从环境变量读取，支持 .env 文件。

    这是 Pydantic 的 BaseSettings，它会自动从环境变量加载值。
    环境变量名 = 属性名的大写形式（如 app_name → APP_NAME）
    """

    # ---- 基础 ----
    app_name: str = "RAG Service"
    version: str = "1.0.0"
    debug: bool = False

    # ---- LLM API Keys ----
    openai_api_key: Optional[str] = None
    dashscope_api_key: Optional[str] = None  # 通义千问
    deepseek_api_key: Optional[str] = None

    # ---- 生成模型配置 ----
    llm_provider: str = "openai"  # openai | tongyi | deepseek
    llm_model: str = "gpt-4o-mini"

    # ---- Embedding 配置 ----
    embedding_provider: str = "openai"  # openai | bge | local
    embedding_model: str = "text-embedding-3-small"
    embedding_dimension: int = 1536

    # ---- 向量数据库 ----
    vector_store_type: str = "chroma"  # chroma | qdrant
    chroma_host: str = "chroma"        # Docker Compose 服务名
    chroma_port: int = 8000
    chroma_collection: str = "rag_knowledge"

    # ---- Redis（缓存 + 会话存储） ----
    redis_host: str = "redis"
    redis_port: int = 6379
    redis_password: Optional[str] = None

    # ---- 检索参数 ----
    retrieval_top_k: int = 5
    retrieval_score_threshold: float = 0.4

    # ---- 对话管理 ----
    history_window_size: int = 6
    session_timeout_minutes: int = 30

    # ---- 日志 ----
    log_level: str = "INFO"
    log_format: str = "json"  # json | text

    class Config:
        env_file = ".env"
        env_file_encoding = "utf-8"


# 全局单例（FastAPI 依赖注入用）
settings = Settings()

为什么用 Pydantic Settings？

pydantic-settings 是 Pydantic 的官方配置管理库。相比 os.environ.get() 散落在各处，它有这些优势：

  • 类型校验：debug: bool 自动解析 "true"/"false"/"1"/"0"
  • 自动加载 .env：配置 env_file 后自动读取
  • IDE 补全：所有配置项像普通 Python 属性一样有补全和类型提示
  • 单一事实源：所有配置项集中在一个类中，一眼看清全部配置

1.4 RAG 管线封装 —— 服务核心

把 L10 的 ConversationalRAG 适配成可以在 FastAPI 中使用的形态。核心变化是：依赖外部化——向量库、模型、缓存都从配置管理，而非硬编码。

"""
app/rag_pipeline.py — RAG 管线（FastAPI 适配版）
"""

import time
import logging
from typing import Optional
from openai import OpenAI

from .config import settings
from .chat_history import SlidingWindowHistory
from .context_injection import resolve_references, resolve_references_rule_based
from .model_adapter import create_chat_adapter, BaseChatAdapter
from .logger import get_logger

logger = get_logger(__name__)


class RAGPipeline:
    """
    RAG 管线封装 —— 注入配置后供 FastAPI 路由调用。

    与 L10 的 ConversationalRAG 不同之处：
    1. 不在内部创建 SessionManager（让 FastAPI 管理会话生命周期）
    2. 依赖配置对象而非硬编码参数
    3. 加了详细的结构化日志
    """

    def __init__(self):
        """初始化 RAG 管线（所有依赖通过 settings 配置）"""
        # 生成模型适配器
        self.adapter = create_chat_adapter(
            settings.llm_provider,
            settings.llm_model,
        )
        # 会话存储（生产环境会迁移到 Redis，这里先用内存）
        self._sessions: dict[str, SlidingWindowHistory] = {}
        # 知识库文档描述（用于健康检查和信息展示）
        self.doc_count = 0
        self.is_ready = False
        logger.info("RAGPipeline 初始化完成",
                     provider=settings.llm_provider,
                     model=settings.llm_model)

    def retrieve(self, query: str, top_k: int = 5,
                 score_threshold: float = 0.0) -> list[dict]:
        """
        执行检索，返回结果列表。

        注意：当前使用 OpenAI Embedding API + Chroma HTTP。
        在 L5/L12 的评估框架下，这个方法的实现可替换为
        Qdrant/Milvus/Elasticsearch 等后端。
        """
        start = time.time()
        results = self._search_vector_db(query, top_k, score_threshold)
        latency = int((time.time() - start) * 1000)
        logger.info("检索完成", query=query, results=len(results),
                     latency_ms=latency)
        return results

    def _search_vector_db(self, query: str, top_k: int,
                          score_threshold: float) -> list[dict]:
        """
        向量数据库检索。

        这里演示用 Chroma HTTP Client 连接容器化 Chroma。
        实际实现取决于你的向量库选型。
        """
        # 使用 Chroma HTTP Client 连接到容器中的 Chroma
        try:
            import chromadb
            client = chromadb.HttpClient(
                host=settings.chroma_host,
                port=settings.chroma_port,
            )
            collection = client.get_or_create_collection(
                name=settings.chroma_collection
            )

            # 先获取 embedding
            embedding = self._get_embedding(query)

            # 检索
            result = collection.query(
                query_embeddings=[embedding],
                n_results=top_k,
            )

            items = []
            for i in range(len(result["ids"][0])):
                score = result["distances"][0][i] if result.get("distances") else 0
                if score_threshold > 0 and score < score_threshold:
                    continue
                items.append({
                    "text": result["documents"][0][i],
                    "score": score,
                    "source": result["metadatas"][0][i].get("source", ""),
                    "metadata": result["metadatas"][0][i],
                })
            return items

        except Exception as e:
            logger.error("向量库检索失败", error=str(e))
            return []

    def _get_embedding(self, text: str) -> list[float]:
        """获取文本的 embedding 向量"""
        if settings.embedding_provider == "openai":
            client = OpenAI(api_key=settings.openai_api_key)
            resp = client.embeddings.create(
                model=settings.embedding_model,
                input=text,
            )
            return resp.data[0].embedding
        else:
            raise ValueError(f"不支持的 embedding provider: {settings.embedding_provider}")

    def ask(self, query: str, session_id: Optional[str] = None,
            user_id: str = "anonymous",
            temperature: Optional[float] = None,
            max_tokens: Optional[float] = None) -> dict:
        """
        完整 RAG 问答。
        """
        start = time.time()

        # 1. 获取/创建会话
        history = self._get_or_create_session(session_id, user_id)

        # 2. 指代消解
        context_text = history.get_recent_context(n=2)
        pronouns = {"它", "该", "这个", "那个", "上述", "该产品"}
        if any(p in query for p in pronouns):
            augmented_query = resolve_references(query, context_text)
        else:
            augmented_query = query

        # 3. 检索
        results = self.retrieve(augmented_query,
                                top_k=settings.retrieval_top_k)

        # 4. 构建生成 prompt
        context_str = ""
        if results:
            context_parts = [
                f"[来源{i+1}] {r['text']}"
                for i, r in enumerate(results)
            ]
            context_str = "\n\n".join(context_parts)

        # 5. 生成回答
        messages = history.get_messages()
        messages.append({"role": "user", "content": query})

        system_content = (
            "你是一个企业知识库助手。请基于以下检索结果回答问题。\n"
            "如果检索结果不包含答案，请说"知识库中暂无相关信息"。\n"
            f"检索结果：\n{context_str}"
        )
        if messages and messages[0]["role"] == "system":
            messages[0]["content"] = system_content
        else:
            messages.insert(0, {"role": "system", "content": system_content})

        try:
            answer = self.adapter.chat(
                messages,
                temperature=temperature or settings.llm_params.get("temperature", 0.7),
                max_tokens=max_tokens or settings.llm_params.get("max_tokens", 1024),
            )
        except Exception as e:
            logger.error("LLM 生成失败", error=str(e))
            answer = f"抱歉，模型调用出错了：{str(e)}。请稍后重试。"

        # 6. 更新历史
        history.add_message("user", query)
        history.add_message("assistant", answer)

        latency_ms = int((time.time() - start) * 1000)
        logger.info("问答完成", query=query, session=session_id,
                     latency_ms=latency_ms, refusal=(not bool(results)))

        return {
            "answer": answer,
            "session_id": session_id or id(history),
            "augmented_query": augmented_query,
            "model": self.adapter.get_model_info()["model"],
            "history_count": len(history.messages) // 2,
            "retrieval_results": [
                {"text": r["text"], "score": r["score"]}
                for r in results[:3]  # 返回前 3 条详情
            ],
            "refusal": not bool(results),
            "latency_ms": latency_ms,
        }

    def _get_or_create_session(self, session_id: Optional[str],
                               user_id: str) -> SlidingWindowHistory:
        """获取或创建对话历史会话"""
        if session_id and session_id in self._sessions:
            return self._sessions[session_id]
        history = SlidingWindowHistory(
            window_size=settings.history_window_size
        )
        history.add_message(
            "system",
            "你是一个企业知识库智能助手。请基于知识库信息回答问题。"
        )
        if session_id:
            self._sessions[session_id] = history
        return history

    def get_status(self) -> dict:
        """获取管线状态"""
        return {
            "ready": self.is_ready,
            "doc_count": self.doc_count,
            "active_sessions": len(self._sessions),
            "llm_provider": settings.llm_provider,
            "llm_model": settings.llm_model,
        }


# 全局 RAG 管线实例（FastAPI 应用启动时初始化）
rag_pipeline = RAGPipeline()

1.5 FastAPI 路由 —— 把 Python 方法变为 HTTP API

现在把 RAGPipeline 的各个方法暴露为 HTTP 端点：

"""
app/main.py — FastAPI 应用入口
"""

import time
from fastapi import FastAPI, HTTPException
from fastapi.middleware.cors import CORSMiddleware
from contextlib import asynccontextmanager

from .config import settings
from .models import (
    RetrieveRequest, RetrieveResponse, RetrievedChunk,
    AskRequest, AskResponse,
    EvaluateRequest, EvaluateResponse,
    HealthResponse,
)
from .rag_pipeline import rag_pipeline
from .logger import get_logger
from .health import check_service_health

logger = get_logger(__name__)


@asynccontextmanager
async def lifespan(app: FastAPI):
    """应用生命周期管理"""
    logger.info("RAG 服务启动", version=settings.version,
                 provider=settings.llm_provider)
    # 启动时尝试连接依赖服务
    await check_service_health()
    rag_pipeline.is_ready = True
    yield
    # 关闭时的清理工作
    rag_pipeline.is_ready = False
    logger.info("RAG 服务关闭")


app = FastAPI(
    title=settings.app_name,
    version=settings.version,
    lifespan=lifespan,
    docs_url="/docs",
    redoc_url="/redoc",
)

# CORS —— 允许前端跨域调用
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)


# ===================== 路由 =====================

@app.get("/health", response_model=HealthResponse,
         summary="健康检查", tags=["系统"])
async def health():
    """服务健康检查 —— 用于 Docker 健康检查 + K8s Probe"""
    services = check_service_health()
    all_healthy = all(s == "healthy" for s in services.values())
    return HealthResponse(
        status="healthy" if all_healthy else "degraded",
        version=settings.version,
        services=services,
        rag_ready=rag_pipeline.is_ready,
    )


@app.post("/retrieve", response_model=RetrieveResponse,
          summary="语义检索", tags=["RAG"])
async def retrieve(req: RetrieveRequest):
    """
    语义检索 —— 只检索不生成。

    接收用户查询，返回知识库中最相关的文档片段。
    支持可选的分数过滤和对话历史增强。
    """
    query = req.query
    # 如果有历史，做指代消解
    augmented_query = query
    if req.history:
        recent = "\n".join(
            f"{m['role']}: {m['content']}"
            for m in req.history[-4:]
        )
        from .context_injection import resolve_references
        augmented_query = resolve_references(query, recent)

    results = rag_pipeline.retrieve(
        augmented_query,
        top_k=req.top_k,
        score_threshold=req.score_threshold,
    )

    chunks = [
        RetrievedChunk(
            text=r["text"],
            score=r["score"],
            source=r.get("source"),
            metadata=r.get("metadata", {}),
        )
        for r in results
    ]

    return RetrieveResponse(
        query=query,
        augmented_query=augmented_query if augmented_query != query else None,
        results=chunks,
        total_found=len(chunks),
        latency_ms=sum(r["score"] for r in results) if results else 0,
    )


@app.post("/ask", response_model=AskResponse,
          summary="RAG 问答", tags=["RAG"])
async def ask(req: AskRequest):
    """
    完整 RAG 问答 —— 检索 + 生成。

    支持多轮对话（传递 session_id），自动进行指代消解。
    可选流式输出（stream=True）。
    """
    result = rag_pipeline.ask(
        query=req.query,
        session_id=req.session_id,
        user_id=req.user_id,
        temperature=req.temperature,
        max_tokens=req.max_tokens,
    )
    return AskResponse(**result)


@app.post("/evaluate", response_model=EvaluateResponse,
          summary="回答质量评估", tags=["RAG"])
async def evaluate(req: EvaluateRequest):
    """
    回答质量评估 —— 使用 LLM-as-a-Judge 评估回答质量。

    基于 L7 的 RAGAS 指标，直接复用之前的评估代码。
    """
    from .ragas_eval import RAGEvaluator
    evaluator = RAGEvaluator()
    scores = evaluator.evaluate(
        query=req.query,
        answer=req.answer,
        contexts=req.contexts,
        metrics=req.metrics,
    )
    return EvaluateResponse(
        query=req.query,
        scores=scores,
    )


@app.get("/", summary="服务信息", tags=["系统"])
async def root():
    """服务根路径 —— 返回基本信息"""
    return {
        "service": settings.app_name,
        "version": settings.version,
        "docs": "/docs",
        "health": "/health",
    }

FastAPI 的自动 API 文档

FastAPI 会根据你的 Pydantic 模型和类型注解，自动生成 OpenAPI 文档。启动服务后打开 http://localhost:8000/docs，你就得到一个可直接调试的交互式 API 文档页面——不需要 Swagger 或其他工具。

这也是选择 FastAPI 的重要原因：它让 API 文档"写一次代码就自动生成"，永远不会跟代码不同步。

第二部分：Dockerfile 与多阶段构建

Service 代码写好了，怎么打包成可移植的镜像？Dockerfile 就是镜像的"配方"。

2.1 从"能用"到"生产级"的 Dockerfile

先看一个反面教材，大多数人第一次写的 Dockerfile：

# ❌ 反面教材 —— 别这么写
FROM python:3.12
WORKDIR /app
COPY . .
RUN pip install -r requirements.txt
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

问题在哪？

  1. 没利用缓存：改一行代码，所有依赖重新安装（pip install 很慢）
  2. 镜像太大：包含了编译工具链、文档、包缓存——一个项目可能 1.5GB+
  3. root 运行：容器内以 root 运行——安全隐患
  4. 没版本锁定：基础镜像 tag 是 python:3.12，明天拉可能变成 3.12.x

2.2 生产级 Dockerfile（多阶段构建）

多阶段构建是 Docker 的核心最佳实践。思路：第一个阶段安装编译依赖并安装 Python 包，第二个阶段只复制运行所需的产物，抛弃编译工具和中间文件。

# ===============================================================
# Dockerfile — 多阶段构建
# ===============================================================

# ---------- 阶段一：安装依赖 ----------
# 选择具体的小版本而非大版本，避免意外的 breaking change
FROM python:3.12.4-slim AS builder

WORKDIR /build

# 1. 先只复制依赖清单（利用 Docker 缓存：不改 requirements 就不重装）
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# ---------- 阶段二：最终运行镜像 ----------
# 使用更小的 slim 基础镜像
FROM python:3.12.4-slim

# 安全：以非 root 用户运行
RUN groupadd -r rag && useradd -r -g rag rag

WORKDIR /app

# 只从 builder 阶段复制已安装的包（而不是重复 pip install）
COPY --from=builder /install /usr/local

# 复制应用代码
COPY app/ ./app/

# 创建数据目录并授权
RUN mkdir -p /app/data && chown -R rag:rag /app/data

# 切换到非 root 用户
USER rag

# 端口暴露
EXPOSE 8000

# 健康检查
HEALTHCHECK --interval=30s --timeout=10s --start-period=60s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" \
    || exit 1

# 启动命令
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

生产级 Dockerfile 的 6 条黄金法则

  1. 用 slim 镜像：python:3.12.4-slim 约 120MB vs python:3.12.4 约 900MB——小了 7 倍
  2. 多阶段构建：编译依赖在 builder 阶段完成，最终镜像只保留运行时需要的文件
  3. 充分利用缓存：把变动频率低的层（requirements.txt）放在前面，代码层放后面——这样不改依赖时永远命中缓存
  4. 非 root 运行：创建专用用户，降低容器逃逸风险
  5. HEALTHCHECK：Docker 根据健康检查判断容器状态
  6. EXPOSE 声明：文档化容器监听的端口（虽然不是安全限制）

2.3 构建镜像

# 构建镜像
docker build -t rag-service:latest .

# 查看镜像大小
docker images rag-service
# 预期：~200-250MB（如果只用 slim + 多阶段，而非 ~1.5GB）

减少镜像大小的更多技巧

  • pip install --no-cache-dir：防止 pip 缓存留在镜像中
  • 只用 slim 或 alpine 基础镜像——但注意 alpine 用 musl libc，某些 Python 包（特别是向量库）可能不兼容
  • Python 包隔离：如果用 pip install --user 安装到 /root/.local，只复制 /root/.local 到最终镜像
  • 删除 .pyc 文件和 __pycache__：可以用 find . -name '*.pyc' -delete

3.2 docker-compose.yml 完整编排

这是本课的核心文件——把全部服务用代码定义清楚：

# docker-compose.yml — RAG 服务栈完整编排
#
# 使用方式：
#   启动： docker compose up -d
#   停止： docker compose down
#   查看日志： docker compose logs -f
#   重建服务： docker compose up -d --build rag-api

version: "3.8"

services:

  # ==================== RAG API ====================
  rag-api:
    build:
      context: .
      dockerfile: Dockerfile
    image: rag-service:latest
    container_name: rag-api
    restart: unless-stopped
    ports:
      - "8000:8000"
    environment:
      # 应用配置
      - APP_NAME=RAG Service
      - LOG_LEVEL=INFO
      - LOG_FORMAT=json
      # LLM API Keys（来自 .env 文件或宿主机环境变量）
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      - DASHSCOPE_API_KEY=${DASHSCOPE_API_KEY}
      - DEEPSEEK_API_KEY=${DEEPSEEK_API_KEY}
      # 模型选择
      - LLM_PROVIDER=${LLM_PROVIDER:-openai}
      - LLM_MODEL=${LLM_MODEL:-gpt-4o-mini}
      - EMBEDDING_PROVIDER=${EMBEDDING_PROVIDER:-openai}
      - EMBEDDING_MODEL=${EMBEDDING_MODEL:-text-embedding-3-small}
      # 依赖服务地址（Compose 服务名作为 hostname）
      - CHROMA_HOST=chroma
      - CHROMA_PORT=8000
      - REDIS_HOST=redis
      - REDIS_PORT=6379
    volumes:
      # 持久化数据（如上传的文件、缓存等）
      - ./data:/app/data
    depends_on:
      chroma:
        condition: service_healthy
      redis:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "python", "-c",
             "import urllib.request; r=urllib.request.urlopen('http://localhost:8000/health'); assert r.status==200"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 60s
    networks:
      - rag-network

  # ==================== Chroma 向量库 ====================
  chroma:
    image: chromadb/chroma:0.5.0
    container_name: rag-chroma
    restart: unless-stopped
    volumes:
      # 持久化向量数据
      - ./data/chroma:/chroma/chroma
    environment:
      - IS_PERSISTENT=TRUE
      - ANONYMIZED_TELEMETRY=FALSE
      - PERSIST_DIRECTORY=/chroma/chroma
    healthcheck:
      test: ["CMD", "python", "-c",
             "import urllib.request; urllib.request.urlopen('http://localhost:8000/api/v1/heartbeat')"]
      interval: 15s
      timeout: 5s
      retries: 3
      start_period: 30s
    ports:
      - "8001:8000"  # 映射到主机 8001 避免与 RAG API 冲突
    networks:
      - rag-network

  # ==================== Redis 缓存 ====================
  redis:
    image: redis:7-alpine
    container_name: rag-redis
    restart: unless-stopped
    command: redis-server --appendonly yes --requirepass ${REDIS_PASSWORD:-}
    volumes:
      - ./data/redis:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 3
    ports:
      - "6379:6379"
    networks:
      - rag-network

  # ==================== Nginx 网关 ====================
  nginx:
    image: nginx:alpine
    container_name: rag-nginx
    restart: unless-stopped
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - rag-api
    networks:
      - rag-network

# ==================== 网络 ====================
networks:
  rag-network:
    driver: bridge

编排设计要点

  • depends_on + condition：RAG API 在 Chroma 和 Redis 健康后才启动——避免启动期连不上依赖报错
  • restart: unless-stopped：非手动停止的容器崩溃后自动重启——这是生产部署的基本保障
  • volumes 挂载持久化：Chroma 的向量数据、Redis 的 AOF 文件都挂载到宿主机 ./data/——容器删了数据不丢
  • 专用网络：所有服务在 rag-network 内，服务名即 hostname（RAG API 通过 http://chroma:8000 访问 Chroma）
  • 端口映射：仅暴露 Nginx（:80）和 Redis（:6379）到宿主机。Chroma 只在内部网络可访问，避免直接暴露

3.3 Nginx 反向代理配置

Nginx 作为 API 网关，统一入口、做请求转发、静态资源服务：

# nginx.conf — Nginx 反向代理配置
upstream rag_api {
    server rag-api:8000;
}

server {
    listen 80;
    server_name _;

    # 请求日志（JSON 格式）
    log_format json escape=json
        '{'
          '"time":"$time_iso8601",'
          '"remote_addr":"$remote_addr",'
          '"method":"$request_method",'
          '"path":"$uri",'
          '"status":$status,'
          '"body_bytes":$body_bytes_sent,'
          '"request_time":$request_time,'
          '"referer":"$http_referer",'
          '"user_agent":"$http_user_agent"'
        '}';

    access_log /var/log/nginx/access.log json;
    error_log /var/log/nginx/error.log;

    # API 路由
    location /api/ {
        # 去掉 /api 前缀再转发
        rewrite ^/api(/.*)$ $1 break;
        proxy_pass http://rag_api;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时配置（LLM 生成可能较长）
        proxy_read_timeout 120s;
        proxy_connect_timeout 10s;
    }

    # 健康检查路径直接透传
    location /health {
        proxy_pass http://rag_api/health;
        proxy_set_header Host $host;
    }

    # 静态文件服务（前端页面等）
    location / {
        root /usr/share/nginx/html;
        index index.html;
        # 如果未来有前端 SPA，添加 try_files
        # try_files $uri $uri/ /index.html;
    }
}

Nginx 在这里的作用：

  • 统一入口 :80/api/* → RAG API :8000/*
  • 超时配置（proxy_read_timeout 120s）——LLM 生成可能耗时 30s+，默认 60s 可能不够
  • JSON 格式的访问日志
  • 未来静态前端页面的托管入口

env_file vs environment 字段

Docker Compose 支持两种环境变量注入方式：

第五部分：健康检查与重启策略

生产系统中，服务的"自愈能力"是标配而非选配。你有三个层次可以保障：

5.1 Docker 重启策略（最基础的自愈）

策略行为推荐场景
"no"从不重启一次性任务容器
"always"总是重启（包括手动 stop 后）一般不用——stop 了还重启很烦
"unless-stopped"退出码非 0 则重启；手动 stop 不重启最推荐：RAG API / 数据库
"on-failure"仅退出码非 0 时重启可能因外部原因崩溃的任务

在 docker-compose.yml 中设置：

services:
  rag-api:
    restart: unless-stopped

5.2 Health Check（基于应用真实状态的健康判断）

重启策略只管"进程是否活着"——但进程活着不代表应用正常。比如：

  • Python 进程没挂，但 LLM API 连不上了
  • Chroma 进程在，但磁盘满了无法写入
  • Redis 还活着，但连接数打满了

健康检查从应用层探测真实状态。我们的 RAG API 有专门的 /health 端点：

# app/health.py — 健康检查逻辑

import logging
from .config import settings

logger = logging.getLogger(__name__)


def check_service_health() -> dict[str, str]:
    """
    检查所有依赖服务的健康状态。

    返回：{服务名: "healthy" | "degraded" | "unhealthy"}
    """
    services = {}

    # 检查 Chroma
    try:
        import chromadb
        client = chromadb.HttpClient(
            host=settings.chroma_host,
            port=settings.chroma_port,
        )
        client.heartbeat()
        services["chroma"] = "healthy"
    except Exception as e:
        logger.warning("Chroma 健康检查失败", extra={"error": str(e)})
        services["chroma"] = "unhealthy"

    # 检查 Redis
    try:
        import redis
        r = redis.Redis(
            host=settings.redis_host,
            port=settings.redis_port,
            password=settings.redis_password,
            socket_connect_timeout=3,
        )
        r.ping()
        services["redis"] = "healthy"
    except Exception as e:
        logger.warning("Redis 健康检查失败", extra={"error": str(e)})
        services["redis"] = "unhealthy"

    # RAG API 自身
    try:
        # 尝试一次简单 embedding（验证 API Key 有效）
        if settings.openai_api_key:
            from openai import OpenAI
            client = OpenAI(api_key=settings.openai_api_key)
            resp = client.embeddings.create(
                model=settings.embedding_model,
                input="healthcheck",
            )
            services["rag-api"] = "healthy"
        else:
            services["rag-api"] = "degraded"
    except Exception as e:
        logger.error("RAG API 健康检查失败", extra={"error": str(e)})
        services["rag-api"] = "unhealthy"

    return services

5.3 depends_on + condition: service_healthy（启动顺序控制）

一个经典的启动时序问题：RAG API 启动时 Chroma 还没就绪，连接被拒绝导致启动失败。

解决方案——在 docker-compose.yml 中声明依赖条件：

services:
  rag-api:
    depends_on:
      chroma:
        condition: service_healthy
      redis:
        condition: service_healthy

这样 Docker 会等待 Chroma 和 Redis 的健康检查通过后，才启动 RAG API。注意：

  • condition: service_healthy 要求被依赖的服务有 HEALTHCHECK 指令
  • 但 depends_on 只控制启动顺序，不保证 RAG API 启动时依赖仍健康——如果 Chroma 在 RAG API 启动过程中挂了呢？所以 RAG API 自身也要有重试逻辑

启动重试：最后的防线

即使有 health check 和 depends_on，启动时仍可能因瞬时问题失败。建议在 RAG API 的初始化逻辑中添加重试（在 rag_pipeline.py 中加入）：

import time
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(5), wait=wait_exponential(multiplier=1, min=2, max=10))
def _init_chroma_client(self):
    import chromadb
    client = chromadb.HttpClient(
        host=settings.chroma_host,
        port=settings.chroma_port,
    )
    client.heartbeat()
    return client

这样即使 Chroma 短暂不可用，RAG API 会在 2s → 4s → 8s → 16s → 32s 后重试，而不是直接崩溃。

第六部分：JSON 结构化日志

在本地开发中，print() 或 Python 默认的文本日志就够了。但在生产环境——特别是容器化和分布式场景中——JSON 格式的结构化日志是必需品。

❌ 文本日志

INFO 2026-06-20 10:30:45 - RAGPipeline - 检索完成
INFO 2026-06-20 10:30:46 - RAGPipeline - 问答完成
ERROR 2026-06-20 10:30:47 - RAGPipeline - LLM调用失败

问题：没法用 grep 精确过滤、没法被日志系统（ELK/Loki）解析、没法自动关联 trace ID

✅ JSON 结构化日志

{"time":"2026-06-20T10:30:45","level":"INFO","logger":"RAGPipeline","message":"检索完成","query":"RAG是什么","results":5,"latency_ms":320}
{"time":"2026-06-20T10:30:46","level":"INFO","logger":"RAGPipeline","message":"问答完成","session":"abc123","latency_ms":2150}
{"time":"2026-06-20T10:30:47","level":"ERROR","logger":"RAGPipeline","message":"LLM调用失败","error":"Connection timeout","provider":"openai"}

优势：可以被 Logstash/Loki 直接解析；可以根据字段精确过滤；可以建立仪表盘监控延迟、错误率

6.1 自定义 JSON Logger

"""
app/logger.py — JSON 结构化日志配置
"""

import sys
import json
import logging
from datetime import datetime, timezone
from typing import Optional


class JSONFormatter(logging.Formatter):
    """
    将日志记录格式化为 JSON 行。
    每行日志输出一个 JSON 对象，便于 ELK/Loki/Grafana 等系统消费。
    """

    def format(self, record: logging.LogRecord) -> str:
        log_entry = {
            "time": datetime.fromtimestamp(record.created, tz=timezone.utc).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
            "module": record.module,
            "line": record.lineno,
        }

        # 添加额外字段（通过 extra= 传入）
        if hasattr(record, "extra"):
            log_entry.update(record.extra)

        # 如果有异常信息，加入堆栈
        if record.exc_info and record.exc_info[0]:
            log_entry["exception"] = {
                "type": record.exc_info[0].__name__,
                "message": str(record.exc_info[1]),
            }

        # 确保中文不乱码
        return json.dumps(log_entry, ensure_ascii=False)


def setup_logging(
    level: str = "INFO",
    log_format: str = "json",
) -> None:
    """
    配置全局日志。

    参数：
        level: 日志级别（DEBUG/INFO/WARNING/ERROR）
        log_format: "json" 或 "text"
    """
    handler = logging.StreamHandler(sys.stdout)

    if log_format == "json":
        handler.setFormatter(JSONFormatter())
    else:
        handler.setFormatter(
            logging.Formatter(
                "%(asctime)s - %(name)s - %(levelname)s - %(message)s"
            )
        )

    root = logging.getLogger()
    root.setLevel(level.upper())
    root.handlers.clear()
    root.addHandler(handler)


def get_logger(name: str) -> logging.Logger:
    """获取带 JSON 支持的 logger"""
    return logging.getLogger(name)

6.2 使用方式

from .logger import get_logger

logger = get_logger(__name__)

# 普通日志
logger.info("服务启动", extra={"version": "1.0.0"})

# 带结构化字段的日志
logger.info("检索完成",
    extra={"query": query, "results": len(results), "latency_ms": latency})

# 错误日志
try:
    # ...
except Exception as e:
    logger.error("LLM 调用失败",
        extra={"provider": "openai", "model": "gpt-4o-mini"},
        exc_info=True)

生产日志的 3 条铁律

  1. 不写敏感信息到日志：永远不会把 API Key、密码、个人身份信息（手机号、身份证号）写到日志里——你可能会发给第三方日志平台，那就等于泄露了
  2. 每条日志都要能追溯：加上 request_id 或 trace_id，这样整个请求链路的日志可以被关联起来——L15 监控课会深入 trace ID 的最佳实践
  3. 错误日志必须有可操作性：光写"出错了"不够——要写什么出错了、什么输入、什么上下文，让值班的人能直接定位问题

故障诊断：常见问题与排查

❌ 问题一：docker compose up 后 RAG API 一直在重启

$ docker compose ps
NAME                STATUS                     PORTS
rag-api             Restarting (1) 10 seconds ago
rag-chroma          Up                        ...
rag-redis           Up                        ...

排查步骤：

  1. 查看 RAG API 的日志：docker compose logs rag-api
    • 如果是 OpenAI API Key 未设置——.env 文件没被加载，检查 OPENAI_API_KEY 是否正确设置且拼写无误
    • 如果是 Cannot connect to chroma:8000——Chroma 还没就绪。检查 depends_on.condition 是否配置正确
    • 如果是 ModuleNotFoundError: openai——pip install 失败了。手动构建镜像：docker compose build rag-api 观察报错
  2. 查看容器的退出码：docker inspect rag-api --format '{{.State.ExitCode}}'——退出码 1 表示应用崩溃，137 表示被 SIGKILL（通常是 OOM）
  3. 进入容器调试：docker compose run rag-api sh 然后手动运行 python -c "from app.main import app; print('ok')"

❌ 问题二：/health 返回 degraded，服务"半死不活"

{
  "status": "degraded",
  "services": {
    "rag-api": "degraded",
    "chroma": "healthy",
    "redis": "healthy"
  }
}

原因：RAG API 健康检查显示 degraded——通常是 API Key 未设置或 embedding 超时。

排查步骤：

  1. 检查 .env 文件是否在 docker-compose.yml 同目录下
  2. 检查 OPENAI_API_KEY 是否正确——试试在 RAG API 容器内手动调用：docker compose exec rag-api python -c "from openai import OpenAI; c=OpenAI(); resp=c.embeddings.create(model='text-embedding-3-small', input='test'); print('OK')"
  3. 如果不在 .env 而在宿主机环境变量中——Docker Compose 默认读取 .env 文件中和宿主机环境变量。检查宿主机：echo $OPENAI_API_KEY

解决方案：要么把 Key 写入 .env 文件，要么在 docker-compose.yml 的 environment: 中直接引用宿主机变量：OPENAI_API_KEY=${OPENAI_API_KEY}。

❌ 问题三：curl localhost/api/ask 返回 502 Bad Gateway

原因：Nginx 转发了请求到 RAG API，但 RAG API 没有及时响应。

排查步骤：

  1. 直接调 RAG API 排除 Nginx 问题：curl -X POST http://localhost:8000/ask -H "Content-Type: application/json" -d '{"query":"hi"}'
    • 如果正常返回 → Nginx 配置有问题
    • 如果也不返回 → RAG API 本身有问题
  2. 检查 Nginx 错误日志：docker compose logs nginx
    • upstream timed out → proxy_read_timeout 太短了（LLM 生成慢的时候可能超过默认 60s）
    • no live upstreams → RAG API 挂了或健康检查未通过
  3. 检查 nginx.conf 中的 proxy_pass http://rag_api;——rag_api 是 upstream 名称，要匹配 docker-compose.yml 中的服务名

❌ 问题四：docker compose up 说"端口已被占用"

Error: response from daemon: driver failed programming external connectivity...
Bind for 0.0.0.0:6379: port is already allocated

原因：宿主机上已经有别的进程占用了 6379 端口（你可能本地装了 Redis）。

解决方案：

  • 方案一：停掉本地服务——sudo systemctl stop redis
  • 方案二：修改映射端口——在 docker-compose.yml 中把 "6379:6379" 改为 "16379:6379"
