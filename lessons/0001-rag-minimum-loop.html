Lesson 0001

RAG 最小闭环：从客服文档到可追溯答案

本课只解决一件事：你要能说清楚并手工演练一个客服问答 RAG Demo 的核心数据流。下一课再开始写 Python Demo。

你今天要获得的能力

学完后，你应该能把“客户提问”变成一个可解释流程：先从知识库找证据，再让 LLM 基于证据回答，而不是让模型凭记忆编答案。RAG 的原始论文把这种思路描述为把生成模型和外部非参数化记忆结合起来，用检索结果增强生成能力；这正适合企业知识库，因为文档会更新，答案还要能追溯来源。

主推荐阅读：Lewis et al., Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks。API 与工具参考见 OpenAI embeddings、OpenAI text generation、Chroma getting started。

核心模型

Chunk客服资料切成小片段
Embed片段转成向量
Store向量和原文进向量库
Retrieve问题取回相似片段
GenerateLLM 基于片段回答
Cite返回来源和置信边界

OpenAI 文档把 embeddings 用于搜索、聚类、推荐等相似度任务；Chroma 的集合用于存储 embeddings、documents 和 metadata，并支持按相似度查询。对客服问答来说，这意味着“客户问题”不必和 FAQ 原文完全同字，只要语义接近，就能检索到候选答案。

客服问答里的一个最小例子

知识片段

doc-001：7 天无理由退货仅适用于未拆封商品。已拆封但存在质量问题的商品，可在签收后 15 天内申请售后检测。

doc-002：普通发票会在订单完成后 24 小时内开具。增值税专用发票需要提交公司抬头、税号、地址、电话和开户行信息。

doc-003：会员积分不能直接提现吗？不能。积分只能用于兑换优惠券、配件或指定服务。

用户问：“拆开用了发现质量问题，还能退吗？” 关键词不完全一样，但语义上接近 doc-001。RAG 系统应该先检索到它，再回答：如果是质量问题，可在签收后 15 天内申请售后检测；不要泛化成“任何拆封都能退”。

代码形状先看懂，不急着跑

下一课我们会把它变成可运行 Demo。现在你只需要理解每行属于闭环里的哪一步。

import chromadb

client = chromadb.Client()
collection = client.get_or_create_collection("support_qa")

collection.upsert(
    ids=["doc-001", "doc-002"],
    documents=[
        "7 天无理由退货仅适用于未拆封商品。已拆封但存在质量问题的商品，可在签收后 15 天内申请售后检测。",
        "普通发票会在订单完成后 24 小时内开具。增值税专用发票需要提交公司抬头、税号、地址、电话和开户行信息。"
    ],
    metadatas=[
        {"source": "return-policy.md", "topic": "refund"},
        {"source": "invoice-policy.md", "topic": "invoice"}
    ],
)

results = collection.query(
    query_texts=["拆开用了发现质量问题，还能退吗？"],
    n_results=2,
)

print(results["documents"])
print(results["metadatas"])

Chroma 的入门文档说明：集合可以存储 documents、embeddings 和 metadata，并返回最相似的结果。Demo 初期可以让 Chroma 自动处理 embedding；后面再替换为你指定的 embedding 模型。

你现在应该记住的三句话

  • Embedding 是为了把文本变成可比较的向量；不是为了直接生成答案。
  • 向量数据库负责“找相似证据”；LLM 负责“把证据组织成回答”。
  • 企业级 RAG 的底线是可追溯：没有证据就拒答，有证据就引用来源。
