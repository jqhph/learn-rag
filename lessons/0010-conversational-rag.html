场景导入：单轮问答的"失忆症"

到 L9 为止，你的 RAG 系统已经能回答问题、能判断何时拒答、能走多层兜底。但有一个致命的缺陷——它没有记忆。

用户
华为 Mate60 Pro 的屏幕尺寸是多少？
RAG 系统
华为 Mate60 Pro 配备了一块 6.82 英寸的 OLED 曲面屏，分辨率 2720 × 1260，支持 120Hz 自适应刷新率。

这就是单轮问答的典型问题：用户自然地说"它"，系统却不知道"它"是谁。在客服场景中，用户天然期待系统能像真人客服一样记住上下文——这是对话式 RAG 要解决的核心问题。

本课将基于 L9 的 RAGWithRefusal 管线，逐步加入：

  • Chat History 管理（不丢记忆）
  • 指代消解（听懂"它""那个""这个"）
  • 会话状态管理（不同用户不同记忆）
  • 多模型适配（灵活切换回答模型）

对话式 RAG vs 单轮 RAG：本质区别

很多人以为"对话式 RAG = 单轮 RAG + 聊天历史塞进 prompt"。但这只是表象。真正的区别在于三点：

  1. 检索上下文：用户的第二句话（如"它多少钱？"）单独检索几乎没有意义——必须结合历史理解"它"指什么，否则检索注定失败
  2. 历史长度管理：对话越长，prompt 越臃肿。不加管理会让 token 消耗爆炸，加得太激进又会丢失关键信息
  3. 模型适配差异：不同模型的 chat API 格式不同、上下文窗口不同、对历史格式要求不同——需要统一抽象

本课将逐个解决这三个问题。

第一部分：三种 Chat History 管理策略

对话历史管理是所有对话式 RAG 的基础。你的选择决定了：

  • token 消耗：每天可能多花几十到几百元
  • 回答质量：丢失关键历史信息会导致答非所问
  • 响应速度：prompt 越长，TTFT（首 token 时间）越长

下面介绍三种主流策略，从简单到复杂。

1.1 策略一：滑动窗口（Sliding Window）

原理：只保留最近 N 轮对话，丢弃最早的内容。

"""
chat_history.py — 对话历史管理三种策略
"""

from dataclasses import dataclass, field
from typing import List, Dict, Optional, Callable
import json


@dataclass
class Message:
    """单条聊天消息"""
    role: str        # "user" | "assistant" | "system"
    content: str
    metadata: dict = field(default_factory=dict)

    def to_dict(self) -> dict:
        return {"role": self.role, "content": self.content}


class SlidingWindowHistory:
    """
    滑动窗口策略——只保留最近 N 轮对话。

    优点：实现简单、token 消耗可预测
    缺点：早期关键信息可能被丢弃
    """

    def __init__(self, window_size: int = 6):
        """
        window_size: 保留的对话轮数（一轮 = 一问 + 一答）
        """
        self.window_size = window_size
        self.messages: list[Message] = []

    def add_message(self, role: str, content: str, **metadata):
        self.messages.append(Message(role=role, content=content, metadata=metadata))
        self._trim()

    def _trim(self):
        """超出窗口时丢弃最旧的消息（但保留首条 system message）"""
        # 计算保留的最大消息数（每轮 2 条消息，+1 给 system）
        max_messages = self.window_size * 2 + 1
        if len(self.messages) > max_messages:
            # 保留第一条（可能是 system message）
            keep = [self.messages[0]] if self.messages[0].role == "system" else []
            # 保留最近 N 轮
            keep.extend(self.messages[-(self.window_size * 2):])
            self.messages = keep

    def get_messages(self) -> list[dict]:
        """返回 OpenAI API 格式的消息列表"""
        return [m.to_dict() for m in self.messages]

    def get_recent_context(self, n: int = 2) -> str:
        """获取最近 n 轮对话的文本摘要（用于指代消解）"""
        recent = self.messages[-(n * 2):]
        return "\n".join(
            f"{'用户' if m.role == 'user' else '助手'}: {m.content}"
            for m in recent
        )

    @property
    def token_estimate(self) -> int:
        """估算当前历史占用的 token 数（粗略：每中文字 ~2 token）"""
        total_chars = sum(len(m.content) for m in self.messages)
        return total_chars // 2  # 粗略估算

    def clear(self):
        self.messages = []


# ========== 使用示例 ==========
if __name__ == "__main__":
    history = SlidingWindowHistory(window_size=3)
    history.add_message("system", "你是华为客服助手。")
    history.add_message("user", "Mate60 Pro 的屏幕多大？")
    history.add_message("assistant", "6.82 英寸 OLED 屏。")
    history.add_message("user", "它支持无线充电吗？")  # "它"需要指代消解
    history.add_message("assistant", "支持 50W 无线快充。")
    print(f"当前历史消息数：{len(history.messages)}")
    print(f"预估 token：{history.token_estimate}")
    print(f"最近一轮上下文：{history.get_recent_context(1)}")

滑动窗口 是最常用、最稳妥的策略。推荐的 window_size 取决于场景：

场景推荐窗口理由
简单 FAQ（一问一答即可）2-3 轮只需支持简单的"刚才说的那个"
产品咨询（连续对比）4-6 轮用户可能对比多个型号，需要保留更多上下文
故障排查（多步诊断）6-8 轮每一步都依赖前一步的信息
长文档分析8-12 轮用户可能逐步深入，需要较长记忆

1.2 策略二：摘要压缩（Summary Compression）

原理：当对话达到一定长度后，用 LLM 把早期对话压缩成一条摘要，替代原始多条消息。

class SummaryCompressionHistory:
    """
    摘要压缩策略——将早期对话压缩为一条摘要消息。

    优点：保留早期关键信息、可支持极长对话
    缺点：需要额外 LLM 调用、有信息丢失风险
    """

    def __init__(
        self,
        max_rounds_before_compress: int = 8,
        compress_fn: Callable[[str], str] | None = None,
    ):
        """
        max_rounds_before_compress: 超过多少轮时触发压缩
        compress_fn: 压缩函数，接收对话文本返回摘要
        """
        self.max_rounds = max_rounds_before_compress
        self.compress_fn = compress_fn or self._default_compress
        self.messages: list[Message] = []
        self._compressed_summary: str | None = None

    def _default_compress(self, dialogue_text: str) -> str:
        """默认压缩函数——用 LLM 生成摘要"""
        from openai import OpenAI
        import os
        client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))
        resp = client.chat.completions.create(
            model="gpt-4o-mini",
            messages=[
                {"role": "system", "content": "将以下对话压缩为一段简洁的中文摘要，保留："
                 "1) 用户提到的产品/型号 2) 已解决的问题 3) 待解决的问题。"
                 "限制在 150 字以内。"},
                {"role": "user", "content": dialogue_text},
            ],
            temperature=0.3,
        )
        return resp.choices[0].message.content

    def add_message(self, role: str, content: str, **metadata):
        self.messages.append(Message(role=role, content=content, metadata=metadata))
        self._maybe_compress()

    def _maybe_compress(self):
        """当对话轮数超过阈值时，将前半部分压缩为摘要"""
        # 统计用户消息数作为"轮数"
        user_msgs = [m for m in self.messages if m.role == "user"]
        if len(user_msgs) > self.max_rounds:
            # 选择前一半的用户轮次进行压缩（保留最近 max_rounds 轮）
            compress_until = len(self.messages) - self.max_rounds * 2
            if compress_until <= 0:
                return

            to_compress = self.messages[:compress_until]
            remaining = self.messages[compress_until:]

            # 构建对话文本
            dialogue_text = "\n".join(
                f"{'用户' if m.role == 'user' else '助手'}: {m.content}"
                for m in to_compress if m.role in ("user", "assistant")
            )

            summary = self.compress_fn(dialogue_text)
            self._compressed_summary = summary

            # 替换为一条摘要消息
            summary_msg = Message(
                role="system",
                content=f"[对话历史摘要] {summary}",
                metadata={"type": "compressed_summary"},
            )
            self.messages = [summary_msg] + remaining

    def get_messages(self) -> list[dict]:
        return [m.to_dict() for m in self.messages]

    def get_recent_context(self, n: int = 2) -> str:
        """获取最近 n 轮对话"""
        # 排除摘要消息
        non_summary = [m for m in self.messages
                       if m.metadata.get("type") != "compressed_summary"]
        recent = non_summary[-(n * 2):]
        result = ""
        if self._compressed_summary:
            result += f"[历史摘要] {self._compressed_summary}\n\n"
        result += "\n".join(
            f"{'用户' if m.role == 'user' else '助手'}: {m.content}"
            for m in recent
        )
        return result

    def clear(self):
        self.messages = []
        self._compressed_summary = None

压缩策略的隐形成本

每次压缩是一次额外的 LLM 调用（~500 token 输入 + ~100 token 输出）。在大量并发对话中，这个成本不可忽视。建议每次压缩时也缓存摘要——如果用户的多轮对话话题未变，直接复用上次的摘要。

另外，摘要存在"遗忘偏差"：LLM 倾向于保留最近提到的信息，早期提到的细节（如"用户说他的订单号是 12345"）可能在压缩中被丢弃。对需要精确信息的场景（如客服查询订单），优先用滑动窗口。

1.3 策略三：关键轮次提取（Key Turn Extraction）

原理：不保留全部对话，而是用 LLM 或规则判断哪些轮次"重要"，只保留这些关键轮次。

class KeyTurnExtractionHistory:
    """
    关键轮次提取策略——只保留标记为"重要"的轮次。

    优点：token 效率最高、关键信息保留最好
    缺点：判断逻辑复杂、可能遗漏"现在不重要但之后重要"的信息
    """

    def __init__(
        self,
        max_turns: int = 10,
        importance_fn: Callable[[str, str], float] | None = None,
    ):
        self.max_turns = max_turns
        # 重要性评分函数：输入 (用户消息, 助手消息) → 0~1 分数
        self.importance_fn = importance_fn or self._default_importance
        self.turns: list[dict] = []  # 每轮 {user_msg, assistant_msg, importance, entities}

    def _default_importance(self, user_msg: str, assistant_msg: str) -> float:
        """默认重要性评分——关键词匹配"""
        important_keywords = [
            "订单", "退款", "投诉", "售后", "保修", "价格", "配置",
            "型号", "参数", "问题", "故障", "无法", "不了",
        ]
        score = 0.0
        combined = user_msg + assistant_msg
        for kw in important_keywords:
            if kw in combined:
                score += 0.2
        return min(score, 1.0)

    def add_message(self, role: str, content: str, **metadata):
        # 需要成对添加（user + assistant）
        if role == "user":
            self._pending_user = Message(role, content, metadata)
        elif role == "assistant" and hasattr(self, "_pending_user"):
            user_msg = self._pending_user.content
            assistant_msg = content

            # 计算重要性
            importance = self.importance_fn(user_msg, assistant_msg)

            # 提取实体（简单用关键词）
            entities = self._extract_entities(user_msg + assistant_msg)

            self.turns.append({
                "user": user_msg,
                "assistant": assistant_msg,
                "importance": importance,
                "entities": entities,
                "timestamp": len(self.turns),
            })
            del self._pending_user

            # 裁剪低重要性轮次
            self._trim()
        elif role == "system":
            self._system_msg = Message(role, content, metadata)

    def _extract_entities(self, text: str) -> list[str]:
        """简单实体提取（生产环境建议用 NER 模型）"""
        import re
        # 匹配产品型号（如 Mate60 Pro、iPhone 15）
        patterns = [
            r'[A-Z][a-z]+[\d\s][A-Za-z0-9\s]{1,20}',  # 产品型号
            r'\d{5,}',                                   # 订单号
        ]
        entities = []
        for p in patterns:
            entities.extend(re.findall(p, text))
        return list(set(entities))

    def _trim(self):
        """保留高重要性轮次，确保不超过 max_turns"""
        if len(self.turns) <= self.max_turns:
            return
        # 按重要性排序，保留最重要的
        self.turns.sort(key=lambda t: t["importance"], reverse=True)
        self.turns = self.turns[:self.max_turns]
        # 恢复时间顺序
        self.turns.sort(key=lambda t: t["timestamp"])

    def get_messages(self) -> list[dict]:
        """转换为 API 格式"""
        result = []
        if hasattr(self, "_system_msg"):
            result.append(self._system_msg.to_dict())
        for turn in self.turns:
            result.append({"role": "user", "content": turn["user"]})
            result.append({"role": "assistant", "content": turn["assistant"]})
        return result

    def get_recent_context(self, n: int = 2) -> str:
        recent = self.turns[-n:]
        lines = []
        for t in recent:
            lines.append(f"用户: {t['user']}")
            lines.append(f"助手: {t['assistant']}")
        return "\n".join(lines)

    def get_mentioned_entities(self) -> list[str]:
        """获取本轮对话中提到的所有实体"""
        entities = set()
        for t in self.turns:
            entities.update(t["entities"])
        return list(entities)

    def clear(self):
        self.turns = []
        if hasattr(self, "_system_msg"):
            del self._system_msg
        if hasattr(self, "_pending_user"):
            del self._pending_user

实战建议：先选滑动窗口

对于大部分客服场景，滑动窗口（窗口大小 6-8 轮） 是最稳妥的起点：

  • 实现简单，没有额外 LLM 调用成本
  • 客服对话通常在 10 轮内解决，8 轮窗口覆盖绝大多数场景
  • 如果发现"系统忘了用户之前说的"，说明窗口太小了
  • 到了 L13（生产化部署）阶段，再考虑用 Redis 做更复杂的历史存储

只有当对话长度经常超过 15-20 轮，且 token 成本敏感时，再考虑升级到摘要压缩或关键轮次提取。

第二部分：指代消解与上下文检索注入

有了 Chat History，我们还需要解决一个关键问题：用户说"它多少钱？"，系统怎么知道"它"是华为 Mate60 Pro？

这就叫指代消解（Coreference Resolution / Referent Resolution）。有两种实现路径：

2.1 路径一：prompt 级别的上下文注入（推荐）

核心思路：检索阶段之前，先用 LLM 把用户的问题"翻译"成一个自包含的查询。这样检索和生成都能独立理解用户意图。

"""
context_injection.py — 指代消解与上下文注入

将"它多少钱？"→"关于前面提到的华为Mate60 Pro，它的价格是多少？"
"""

from typing import Optional
from openai import OpenAI
import os


def resolve_references(
    query: str,
    recent_context: str,
    mentioned_entities: list[str] | None = None,
    model: str = "gpt-4o-mini",
) -> str:
    """
    指代消解：将依赖上下文的查询改写为自包含查询。

    参数：
        query: 当前用户问题（可能含"它""这个""那个"等指代词）
        recent_context: 最近几轮对话文本
        mentioned_entities: 对话中已提到的实体列表（可选）

    返回：
        改写后的自包含查询
    """
    client = OpenAI(api_key=os.environ.get("OPENAI_API_KEY"))

    entities_hint = ""
    if mentioned_entities:
        entities_hint = f"\n对话中提到的实体：{', '.join(mentioned_entities)}"

    resp = client.chat.completions.create(
        model=model,
        temperature=0.1,  # 低温度：改写追求准确性而非创造性
        messages=[
            {"role": "system", "content": f"""你是一个查询改写专家。
你的任务是将用户的当前问题改写为自包含的搜索查询
——即使脱离对话历史也能理解。

规则：
1. 将"它""这个""那个""该产品""上述"等指代词替换为具体名称
2. 补全省略的信息（如"多少钱"→"XX产品多少钱"）
3. 保留原文的核心意图和关键信息
4. 如果查询已经是自包含的，直接返回原文
5. 只返回改写后的查询文本，不要解释
6. 保持中文输出{entities_hint}"""},
            {"role": "user", "content": f"""对话历史（最近几轮）：
{recent_context}

用户当前问题：{query}

改写后的自包含查询："""},
        ],
    )
    rewritten = resp.choices[0].message.content.strip()
    return rewritten


def resolve_references_rule_based(
    query: str,
    recent_context: str,
    mentioned_entities: list[str] | None = None,
) -> str:
    """
    基于规则的简单指代消解（不用 LLM，适合低延迟场景）。

    适用场景：指代模式简单、延迟敏感、不想增加 LLM 调用。
    """
    # 检测指代词
    pronouns = {"它", "该", "这个", "那个", "上述", "该产品", "该设备", "该商品"}
    has_pronoun = any(pronoun in query for pronoun in pronouns)

    if not has_pronoun:
        return query  # 没有指代词，直接返回

    # 尝试从最近对话中提取名词实体
    # 策略：取上一轮用户/助手中提到的最可能的"话题主体"
    import re

    # 提取最近一句话中的产品名（常见模式：中文/英文产品名）
    product_patterns = re.findall(
        r'[一-鿿]{2,10}[\s]?[A-Za-z0-9\s]{1,20}',  # 中文+英文型号
        recent_context,
    )

    if not product_patterns:
        # 尝试提取长名词短语
        product_patterns = re.findall(
            r'[一-鿿]{4,20}',  # 4 个中文字以上的短语
            recent_context,
        )

    if product_patterns:
        # 取最后一个匹配的（通常是最近的讨论焦点）
        target = product_patterns[-1].strip()
        # 替换第一个指代词
        for pronoun in pronouns:
            if pronoun in query:
                query = query.replace(pronoun, target, 1)
                break

    return query

2.2 路径二：检索时注入历史上下文（替代方案）

如果你不想在检索前改写查询（比如延迟敏感到极致），可以换一种思路：检索时，把历史上下文拼接到当前查询上。

def inject_history_to_retrieval(
    query: str,
    history_context: str,
    strategy: str = "append",
) -> str:
    """
    将历史上下文注入到检索查询中。

    策略：
    - "append": 将历史追加到查询后面（适合短查询）
    - "prepend": 将历史放在查询前面（适合历史依赖强的查询）
    - "concat_full": 拼接完整上下文
    """
    if strategy == "append":
        return f"{query}\n\n[上下文]{history_context}"
    elif strategy == "prepend":
        return f"[上下文]{history_context}\n\n{query}"
    elif strategy == "concat_full":
        return f"{history_context}\n\n当前问题：{query}"
    return query

两种路径的选择建议

维度路径一：检索前改写路径二：检索时拼上下文
检索质量⭐⭐⭐⭐⭐ 最优（改写后查询更精确）⭐⭐ 较差（噪音多，向量可能被稀释）
延迟+200~400ms（多一次 LLM 调用）几乎无额外延迟
成本每次对话 +~200 token 的 LLM 调用零额外成本
适用场景优先推荐极低延迟场景或规则级指代消解

实战建议：在 L10 阶段优先用路径一（检索前改写），因为现在的 LLM API 调用成本已经非常低（gpt-4o-mini 每百万 token ~$0.15），而检索质量的提升非常显著。

第三部分：会话状态管理

3.1 Session 状态模型

"""
session.py — 会话状态管理

为每个会话维护独立的状态。
"""

from dataclasses import dataclass, field
from typing import Optional
from datetime import datetime
import uuid


@dataclass
class SessionState:
    """完整的会话状态"""
    session_id: str
    user_id: str
    history: SlidingWindowHistory
    created_at: datetime
    last_active: datetime
    context: dict = field(default_factory=dict)
    # context 可存储的话题信息：
    # {
    #   "current_topic": "华为Mate60 Pro",
    #   "mentioned_products": ["Mate60 Pro", "P60"],
    #   "resolved_issues": ["屏幕尺寸", "电池"],
    #   "pending_issues": ["价格"],
    #   "user_intent": "product_inquiry",
    # }


class SessionManager:
    """
    会话管理器——创建、获取、清理会话。

    生产环境建议用 Redis 替代内存存储（L17 会讲）。
    """

    def __init__(self, session_timeout_minutes: int = 30):
        self.sessions: dict[str, SessionState] = {}
        self.timeout_minutes = session_timeout_minutes

    def create_session(self, user_id: str, system_prompt: str | None = None) -> SessionState:
        """创建新会话"""
        session_id = str(uuid.uuid4())[:8]
        history = SlidingWindowHistory(window_size=6)

        if system_prompt:
            history.add_message("system", system_prompt)

        session = SessionState(
            session_id=session_id,
            user_id=user_id,
            history=history,
            created_at=datetime.now(),
            last_active=datetime.now(),
        )
        self.sessions[session_id] = session
        return session

    def get_session(self, session_id: str) -> Optional[SessionState]:
        """获取会话，更新最后活跃时间"""
        session = self.sessions.get(session_id)
        if session:
            session.last_active = datetime.now()
        return session

    def delete_session(self, session_id: str):
        """删除会话"""
        self.sessions.pop(session_id, None)

    def cleanup_expired(self):
        """清理超时会话"""
        now = datetime.now()
        expired = [
            sid for sid, s in self.sessions.items()
            if (now - s.last_active).total_seconds() > self.timeout_minutes * 60
        ]
        for sid in expired:
            del self.sessions[sid]
        return len(expired)

    def update_context(self, session_id: str, **context_updates):
        """更新会话的上下文信息"""
        session = self.get_session(session_id)
        if session:
            session.context.update(context_updates)

    @property
    def active_count(self) -> int:
        return len(self.sessions)

内存存储 vs Redis 持久化

当前 SessionManager 用 dict 在内存中存储所有会话状态——这有两个问题：

  • 进程重启后所有会话丢失：用户不得不重新开始对话

到 L17 成本控制与性能优化 中，你会学到用 Redis 做会话存储，解决这两个问题。现阶段用内存存储足够学习和原型验证。

第四部分：多模型 Chat API 统一适配层

L8 技术栈全景图中，你看到了多种生成模型的选择。但在多轮对话场景中，不同模型的 Chat API 格式、参数、能力差异很大——你需要一个适配层来统一管理它们。

为什么需要适配层？

客服场景中，"多模型切换"不是锦上添花，而是刚需：

  • 成本优化：简单问题用 gpt-4o-mini，复杂问题用 gpt-4o——可节省 80% 的 LLM 成本
  • 合规要求：国内客服场景可能需要用通义千问或 DeepSeek，而非 OpenAI
  • 容灾备用：OpenAI API 挂了时自动切换到 DeepSeek

4.1 适配层接口设计

"""
model_adapter.py — 多模型 Chat API 统一适配层

支持 OpenAI / 通义千问( DashScope ) / DeepSeek 三种模型。
"""

from abc import ABC, abstractmethod
from typing import Optional
import os


class BaseChatAdapter(ABC):
    """模型适配器基类——所有模型适配器实现此接口"""

    @abstractmethod
    def chat(
        self,
        messages: list[dict],
        temperature: float = 0.7,
        max_tokens: int = 1024,
        **kwargs,
    ) -> str:
        """发送对话并返回回复文本"""
        ...

    @abstractmethod
    def get_model_info(self) -> dict:
        """返回模型信息"""
        ...


class OpenAIChatAdapter(BaseChatAdapter):
    """OpenAI API 适配器"""

    def __init__(
        self,
        model: str = "gpt-4o-mini",
        api_key: str | None = None,
        base_url: str | None = None,
    ):
        from openai import OpenAI
        self.model = model
        self.client = OpenAI(
            api_key=api_key or os.environ.get("OPENAI_API_KEY"),
            base_url=base_url,
        )

    def chat(
        self,
        messages: list[dict],
        temperature: float = 0.7,
        max_tokens: int = 1024,
        **kwargs,
    ) -> str:
        resp = self.client.chat.completions.create(
            model=self.model,
            messages=messages,
            temperature=temperature,
            max_tokens=max_tokens,
            **kwargs,
        )
        return resp.choices[0].message.content

    def get_model_info(self) -> dict:
        return {
            "provider": "openai",
            "model": self.model,
            "context_window": 128000,  # GPT-4o-mini
            "supports_streaming": True,
        }


class TongyiChatAdapter(BaseChatAdapter):
    """通义千问 DashScope API 适配器"""

    def __init__(
        self,
        model: str = "qwen-plus",
        api_key: str | None = None,
    ):
        self.model = model
        self.api_key = api_key or os.environ.get("DASHSCOPE_API_KEY")

    def chat(
        self,
        messages: list[dict],
        temperature: float = 0.7,
        max_tokens: int = 1024,
        **kwargs,
    ) -> str:
        from http.client import HTTPSConnection
        import json

        # DashScope API 格式转换
        body = json.dumps({
            "model": self.model,
            "input": {
                "messages": messages,
            },
            "parameters": {
                "temperature": temperature,
                "max_tokens": max_tokens,
                **kwargs,
            },
        })

        conn = HTTPSConnection("dashscope.aliyuncs.com")
        conn.request(
            "POST",
            "/api/v1/services/aigc/text-generation/generation",
            body=body,
            headers={
                "Authorization": f"Bearer {self.api_key}",
                "Content-Type": "application/json",
            },
        )
        resp = conn.getresponse()
        data = json.loads(resp.read())
        conn.close()

        return data["output"]["text"]

    def get_model_info(self) -> dict:
        return {
            "provider": "tongyi",
            "model": self.model,
            "context_window": 32000,  # qwen-plus
            "supports_streaming": True,
        }

def create_chat_adapter(
    provider: str = "openai",
    model: str | None = None,
    **kwargs,
) -> BaseChatAdapter:
    """
    适配器工厂——根据 provider 创建对应的适配器实例。

    参数：
        provider: "openai" | "tongyi" | "deepseek"
        model: 模型名称（不指定则使用默认）
        **kwargs: 传递给适配器的其他参数（api_key, base_url 等）
    """
    providers = {
        "openai": (OpenAIChatAdapter, "gpt-4o-mini"),
        "tongyi": (TongyiChatAdapter, "qwen-plus"),
        "deepseek": (DeepSeekChatAdapter, "deepseek-chat"),
    }

    cls, default_model = providers.get(provider, (OpenAIChatAdapter, "gpt-4o-mini"))
    return cls(model=model or default_model, **kwargs)

4.2 模型切换策略

有了适配层，你可以实现灵活的模型切换：

class RouterConfig:
    """模型路由配置"""

    def __init__(self):
        # 默认适配器
        self.default_adapter = create_chat_adapter("openai", "gpt-4o-mini")
        # 复杂问题用更强模型
        self.complex_adapter = create_chat_adapter("openai", "gpt-4o")
        # 国内场景
        self.domestic_adapter = create_chat_adapter("tongyi", "qwen-max")
        # 备用模型（当主模型 API 不可用时）
        self.fallback_adapter = create_chat_adapter("deepseek", "deepseek-chat")

    def select_adapter(self, query: str, context: dict) -> BaseChatAdapter:
        """根据查询和上下文选择最合适的模型"""
        # 规则：含敏感词用国内模型
        sensitive_keywords = ["退款", "投诉", "法律", "合规"]
        if any(kw in query for kw in sensitive_keywords):
            return self.domestic_adapter

        # 规则：复杂推理用强模型
        complex_patterns = ["为什么", "对比", "区别", "原理", "建议"]
        complexity_score = sum(1 for p in complex_patterns if p in query)
        if complexity_score >= 2:
            return self.complex_adapter

        # 默认用轻量模型
        return self.default_adapter

模型适配层的设计原则

  • 接口统一：所有适配器暴露相同的 chat(messages, temperature, max_tokens) → str 接口
  • 配置化：模型选择通过配置驱动，不改代码即可切换
  • 优雅降级：主模型失败时自动切备用模型（后面故障诊断中会实现）
  • 可观测：每次调用记录用了哪个模型、耗时、token 数——为 L15 的监控做准备

这是对话式 RAG 上线时一定会遇到的工程问题——对话越长，prompt 越长，响应越慢，token 成本越高。你需要通过实验找到你的场景的最优平衡点。

  • TTFT > 3s：用户开始不耐烦，对话体验明显下降

"""
history_benchmark.py — 历史长度与延迟的量化对比

在你的真实场景中跑这个 benchmark，找到最优窗口大小。
"""

import time
from typing import Callable
import matplotlib.pyplot as plt


def benchmark_history_lengths(
    chat_fn: Callable[[list[dict]], str],
    history_messages: list[dict],
    window_sizes: list[int] = [2, 4, 6, 8, 10, 15, 20],
    num_runs: int = 3,
) -> dict:
    """
    对比不同窗口大小下的延迟和 token 消耗。

    参数：
        chat_fn: 接收 messages 列表并返回回复的函数
        history_messages: 完整的历史消息列表
        window_sizes: 要测试的窗口大小（轮数）
        num_runs: 每个窗口测试次数

    返回：
        {window_size: {"avg_latency": ..., "token_count": ...}}
    """
    results = {}

    for ws in window_sizes:
        latencies = []

        for _ in range(num_runs):
            # 模拟滑动窗口
            max_msgs = ws * 2 + 1  # 每轮 2 条 + system
            if len(history_messages) > max_msgs:
                test_messages = (
                    [history_messages[0]] +  # system
                    history_messages[-(ws * 2):]  # 最近 ws 轮
                )
            else:
                test_messages = history_messages

            # 计时
            start = time.time()
            chat_fn(test_messages)
            latencies.append(time.time() - start)

        token_count = sum(len(m.get("content", "")) for m in test_messages) // 2

        results[ws] = {
            "avg_latency": sum(latencies) / len(latencies),
            "min_latency": min(latencies),
            "max_latency": max(latencies),
            "token_count": token_count,
            "message_count": len(test_messages),
        }

        print(f"窗口 {ws} 轮 | 延迟 {results[ws]['avg_latency']:.2f}s | "
              f"token ~{token_count} | 消息 {len(test_messages)} 条")

    return results

延迟优化的几条黄金法则

  1. 滑动窗口 6 轮是安全的起点——Google 的研究和工业实践都表明，客服场景中 90% 的指代关系在最近 4-6 轮内可解
  2. 不要对每一轮对话都做指代消解 LLM 调用——只有在检测到指代词时才调用 LLM 改写，否则用规则版本
  3. system prompt 要精简——每多 100 token 的 system prompt，TTFT 增加约 30-50ms
  4. 考虑用 streaming——首 token 出来后就开始显示，即使后续 token 还在生成，用户感知的延迟会大幅降低

"""

在 L9 的 RAGWithRefusal 基础上，增加对话能力。
"""

from dataclasses import dataclass, field
from typing import Optional
from datetime import datetime
import os
from openai import OpenAI


# ========== 导入 L9 的组件 ==========
# 假设 L9 的代码在以下模块中：
from confidence import decide_if_should_answer
from refusal_templates import build_refusal_response
from fallback_chain import execute_fallback_chain, query_rewrite

# 导入本课组件
from chat_history import SlidingWindowHistory, Message
from context_injection import resolve_references, resolve_references_rule_based
from session import SessionManager, SessionState
from model_adapter import create_chat_adapter, BaseChatAdapter


@dataclass
class ConversationConfig:
    """对话式 RAG 配置"""
    # 历史管理
    history_window_size: int = 6
    history_strategy: str = "sliding_window"  # sliding_window | summary | key_turn

    # 指代消解
    use_llm_reference: bool = True
    # 仅在检测到指代词时使用 LLM（节省成本）
    reference_llm_only_when_needed: bool = True

    # 模型适配
    model_provider: str = "openai"
    model_name: str = "gpt-4o-mini"
    base_url: str | None = None

    # 检索
    retrieval_top_k: int = 5
    score_threshold: float = 0.4

    # 会话
    session_timeout_minutes: int = 30

    # 生成
    temperature: float = 0.7
    max_tokens: int = 1024

    # 兜底
    enable_web_search: bool = False
    enable_human_fallback: bool = True


class ConversationalRAG:
    """
    多轮对话 RAG 管线。

    使用方式：
        rag = ConversationalRAG()
        session = rag.create_session("user_001")

        # 多轮对话
        reply1 = rag.ask("Mate60 Pro的屏幕多大？", session.session_id)
        reply2 = rag.ask("它支持无线充电吗？", session.session_id)
        reply3 = rag.ask("多少钱？", session.session_id)
    """

    def __init__(
        self,
        config: ConversationConfig | None = None,
        retrieve_fn: callable | None = None,
    ):
        self.config = config or ConversationConfig()
        self.session_manager = SessionManager(
            session_timeout_minutes=self.config.session_timeout_minutes
        )
        self.adapter = create_chat_adapter(
            self.config.model_provider,
            self.config.model_name,
            base_url=self.config.base_url,
        )
        # 检索函数（由外部注入，保持与 L1-L9 一致的解耦）
        self._retrieve_fn = retrieve_fn or self._default_retrieve

        # 统计
        self.total_tokens_used = 0
        self.total_queries = 0

    def _default_retrieve(self, query: str, top_k: int = 5) -> list[dict]:
        """默认检索函数（占位，实际由外部注入）"""
        return []  # 练习中替换为真实检索

    def set_retrieve_fn(self, fn: callable):
        """注入检索函数"""
        self._retrieve_fn = fn

    def create_session(
        self,
        user_id: str,
        system_prompt: str | None = None,
        metadata: dict | None = None,
    ) -> SessionState:
        """创建新的对话会话"""
        if not system_prompt:
            system_prompt = """你是企业知识库智能助手。
请基于知识库信息回答用户问题。如果知识库没有相关信息，请诚实告知。
回答时请引用信息来源。保持专业、友好、简洁。"""
        return self.session_manager.create_session(user_id, system_prompt)

    def _needs_reference_resolution(self, query: str) -> bool:
        """检测查询是否需要指代消解"""
        pronouns = {"它", "该", "这个", "那个", "上述", "该产品", "该设备", "该商品",
                     "它们", "这些", "那些", "其", "此"}
        return any(pronoun in query for pronoun in pronouns)

    def ask(self, query: str, session_id: str) -> dict:
        """
        处理一次对话请求。

        参数：
            query: 用户问题
            session_id: 会话 ID

        返回：
            {
                "answer": str,            # 最终回答
                "augmented_query": str,   # 增强后的查询
                "model": str,             # 使用的模型
                "history_count": int,     # 当前历史轮数
                "token_estimate": int,    # token 估算
                "latency_ms": int,        # 总耗时
                "retrieval_results": int, # 检索结果数
                "refusal": bool,          # 是否拒答
            }
        """
        import time
        start = time.time()

        # 1. 获取会话
        session = self.session_manager.get_session(session_id)
        if not session:
            return {"answer": "会话已过期，请重新开始对话。", "error": "session_expired"}

        self.total_queries += 1

        # 2. 指代消解
        context_text = session.history.get_recent_context(n=2)

        if self._needs_reference_resolution(query) or not self.config.reference_llm_only_when_needed:
            if self.config.use_llm_reference:
                augmented_query = resolve_references(query, context_text)
            else:
                augmented_query = resolve_references_rule_based(query, context_text)
        else:
            augmented_query = query

        # 3. 检索
        results = self._retrieve_fn(augmented_query, top_k=self.config.retrieval_top_k)

        # 4. 置信度评估（复用 L9 的 decision 逻辑）
        from confidence import decide_if_should_answer
        decision = decide_if_should_answer(query, results)

        # 5. 构建 prompt
        context_parts = []
        if results:
            for i, r in enumerate(results):
                context_parts.append(f"[来源{i+1}] {r.get('text', '')}")
            context_str = "\n\n".join(context_parts)
        else:
            context_str = ""

        # 6. 处理拒答
        if decision["decision"] != "answer":
            message_history = session.history.get_messages()
            message_history.append({"role": "user", "content": query})

            # 用适配器回答（让 LLM 根据上下文决定是否拒答？不——L9 的逻辑已经决定了）
            # 这里直接生成拒答回复，但保持对话语气
            from refusal_templates import build_refusal_response
            refusal_reply = build_refusal_response(decision, query)

            # 更新历史
            session.history.add_message("user", query)
            session.history.add_message("assistant", refusal_reply)

            latency_ms = int((time.time() - start) * 1000)
            return {
                "answer": refusal_reply,
                "augmented_query": augmented_query,
                "model": self.adapter.get_model_info()["model"],
                "history_count": len(session.history.messages) // 2,
                "token_estimate": session.history.token_estimate,
                "latency_ms": latency_ms,
                "retrieval_results": len(results),
                "refusal": True,
                "decision_detail": decision,
            }

        # 7. 带历史上下文的生成
        message_history = session.history.get_messages()
        message_history.append({"role": "user", "content": query})

        # 如果没有历史，用标准格式；有历史则在 system prompt 后插入搜索结果
        generation_context = f"""基于以下检索结果回答用户问题。如果检索结果中有相关信息，请引用来源。
如果检索结果不充分，请诚实告知。

检索结果：
{context_str}

请注意对话上下文：用户之前问过相关问题。"""

        # 找到 system message 并添加检索上下文
        if message_history and message_history[0]["role"] == "system":
            original_system = message_history[0]["content"]
            message_history[0]["content"] = f"{original_system}\n\n{generation_context}"
        else:
            message_history.insert(0, {
                "role": "system",
                "content": generation_context,
            })

        try:
            answer = self.adapter.chat(
                message_history,
                temperature=self.config.temperature,
                max_tokens=self.config.max_tokens,
            )
        except Exception as e:
            answer = f"抱歉，模型调用出错了：{str(e)}。请稍后重试。"

        # 8. 更新历史
        session.history.add_message("user", query)
        session.history.add_message("assistant", answer)

        # 9. 更新会话上下文
        self.session_manager.update_context(
            session_id,
            last_topic=self._extract_topic(query, answer),
        )

        latency_ms = int((time.time() - start) * 1000)

        return {
            "answer": answer,
            "augmented_query": augmented_query,
            "model": self.adapter.get_model_info()["model"],
            "history_count": len(session.history.messages) // 2,
            "token_estimate": session.history.token_estimate,
            "latency_ms": latency_ms,
            "retrieval_results": len(results),
            "refusal": False,
        }

    def _extract_topic(self, query: str, answer: str) -> str:
        """简单提取话题（仅做演示，生产环境建议用 NER）"""
        import re
        # 尝试提取产品名
        match = re.search(r'[一-鿿]{2,10}[\s]?[A-Za-z0-9\s]{1,20}', query)
        return match.group().strip() if match else "general"

    def switch_model(self, provider: str, model_name: str | None = None):
        """切换生成模型（运行时切换）"""
        self.config.model_provider = provider
        if model_name:
            self.config.model_name = model_name
        self.adapter = create_chat_adapter(
            self.config.model_provider,
            self.config.model_name,
        )

故障诊断：常见问题与排查

❌ 问题一：用户说"它多少钱？"，系统回答"请指定产品"

原因：指代消解失效了——"它"没有被替换为具体产品名。

排查步骤：

  1. 检查 resolve_references() 的输入 recent_context 是否包含上一轮的产品名
  2. 检查 use_llm_reference 是否为 True，以及 API Key 是否正确
  3. 如果用的是规则版本（rule-based），检查 product_patterns 正则是否匹配了你的产品名格式
  4. 如果用的是 LLM 版本，打印 rewritten 看看 LLM 返回了什么——可能是 prompt 不够明确

急救方案：临时启用将整个历史上下文拼接到检索查询中（路径二），等指代消解修好后再切回路径一。

❌ 问题二：对话到第 10 轮时，响应突然变得非常慢

原因：随着对话增长，prompt 越来越长，TTFT 线性增加。

排查步骤：

  1. 打印 token_estimate 看看对话到 10 轮时 prompt 有多少 token
  2. 检查 SlidingWindowHistory._trim() 是否在正常工作——有时只加了消息没触发裁剪
  3. 检查 window_size 设置是否合理（10 轮 × 2 = 20 条消息 + system，可能已经几千 token）

解决方案：减小 window_size，或者在每一轮对话后调用 history._trim() 确保裁剪生效。

❌ 问题三：切换模型后，对话历史格式导致 API 报错

原因：不同模型的 Chat API 对 message 格式要求有细微差异。

常见差异：

  • 有些模型要求 messages 第一条必须是 system（OpenAI、DeepSeek），有些没有限制
  • 通义千问的 DashScope API 要求 messages 中不能有空 content
  • 部分模型不支持 name 字段

解决方案：在 BaseChatAdapter.chat() 方法中添加消息格式校验和清理逻辑：

def _sanitize_messages(self, messages: list[dict]) -> list[dict]:
    """清理消息格式，确保兼容性"""
    cleaned = []
    for m in messages:
        # 确保 content 不为空
        if not m.get("content"):
            m["content"] = "..."
        # 确保第一条是 system（如果模型要求）
        cleaned.append({"role": m["role"], "content": m["content"]})
    return cleaned
